# testcraft

> 面向 AI 的结构化测试知识管理系统

testcraft 是一套运行于 AI 助手之上的测试能力框架，解决 AI 跨会话无持久记忆的根本问题，让测试知识可以在模块之间、会话之间、模型之间持续积累。

任何具备文件读写能力的 AI 模型或智能体均可使用，不绑定特定工具或平台。

---

## 为什么需要 testcraft

原生 AI 在测试任务上的核心局限：

| 问题 | 原生 AI | testcraft |
|------|---------|-----------|
| 跨会话知识 | 每次重新分析，重复劳动 | story 系统持久化，下次从断点继续 |
| 覆盖完整性 | 依赖提示词质量，随机性高 | 7问/五镜头结构保底，不遗漏维度 |
| 测试方向 | 验证性（系统是否工作）| 对抗性（用户在哪里会受伤）|
| PRD→TC 可追溯 | 无，黑盒输出 | 断言链路显式记录 |
| 多模块覆盖管理 | 无，无法全局视图 | index.json 统一管理进度和状态 |
| 团队协作 | 每人各做一套 | story 是共享知识库 |

## 核心思考哲学

testcraft 的核心竞争力不是工具，而是**思考框架**。任何模型按照这套框架执行，都能在方向上保证测试质量——模型越强，细节越好；框架保证大方向不走偏。

### 三驱动认知框架

读任何信息源（需求文档、代码、口述）时，始终用三个驱动保持认知方向：

**假设优先**：读之前先用一句话假设这个模块「做什么、谁触发、影响什么」。读是在验证和修正假设，不是从零积累信息。

**模型驱动**：每读一段，问——这证实还是推翻了假设的哪个部分？

**证据驱动**：每条论断必须有来源（PRD 文字 / 代码实现）。「我认为是这样」不是断言，是待验证假设。

**边界驱动**：每个系统交互都有两端。遇到边界（外部 API 调用、跨模块依赖）时判断另一端是否已知；未知且关键时主动寻找。

认知目标：**以测试为目的——完整理解功能行为（为验证类测试提供基础），同时识别潜在风险点（为异常类测试提供方向）。两者都不能偏废。**

---

### 渐进式阅读策略

大型信息源（长文档、大文件）的通用读法，不限定工具类型，任何 agent 自行选择实现方式：

```
第一阶段：导航（建地图，成本极低）
  扫描信息结构，建立「这里有什么、在哪里」的地图。
  不深度理解，只做位置定位。

第二阶段：切片精读（按问题，成本按需）
  每次精读操作针对一个「语义单元」，回答假设中的一个开放问题。
  没有开放问题时停止，不继续读。
```

**语义单元的定义（由各适配器声明）**：
- 代码 → 函数体（完整控制流的最小自包含单元）
- 需求文档 → 功能点段落（描述一个完整业务场景的最小文字单元）

---

### 用户影响优先（测试设计原则）

设计测试用例前先问一个问题：

> **如果这个模块今天在生产中出问题，用户最先感受到什么损失？**

以这个答案定义 2-3 个优先测试场景，为这些场景设计多角度用例；再用断言集合补全系统性覆盖。

置信度为 `⚠️`（PRD 与代码矛盾）和 `（待代码确认）`（仅有 PRD 无代码验证）的断言，追加高优先级覆盖——这是系统中已知的不确定点，也是最容易出 bug 的地方。

```

└── capability/
    ├── qa-understand/              # 测试意图理解与提炼
    │   ├── SKILL.md                # 三驱动 + 渐进式阅读策略 + 执行计划格式
    │   └── adapters/
    │       ├── text.md             # 文本源：7问追问提炼
    │       ├── code.md             # 代码源：五镜头提炼（支持多仓库）
    │       └── synthesis.md        # 多源证据聚合与置信度判断
    ├── qa-functional-test/         # 功能测试用例设计
    │   └── SKILL.md
    ├── qa-case-review/             # 用例质量评审
```

```
    └── {模块名}/
```

## 快速上手

### 第一步：配置

创建配置文件 `~/.claude/testcraft-config.json`：

```json
{
  "practices_path": "/absolute/path/to/your/practices",
  "story_path": "/absolute/path/to/your/story"
}
```

### 第二步：初始化 practices

参考 `practices-bootstrap.md` 中的目录结构创建规范文件。practices 是你的测试规范库，定义断言格式、用例标准、覆盖规则等。一次建设，所有项目复用。

### 第三步：在 Claude Code 中触发

testcraft 在 Claude Code 中以 skill 形式运行，以下描述会自动触发对应能力：

```bash
# 理解需求，提炼测试断言
"帮我分析用户登录模块的需求文档，沉淀到 story"

# 阅读代码，提炼测试断言
"帮我阅读 src/auth/ 目录下的代码，更新登录模块的 story"

# 同时读需求和代码（多源模式，自动聚合置信度）
"读登录模块的 PRD 和代码，更新 story"

# 生成测试用例
"给登录模块出测试用例"

# 评审测试用例
"评审登录模块的测试用例"

# 生成脑图产物
"给登录模块出全量测试用例并输出脑图"
```

## 能力详解

### qa-understand：测试意图理解与提炼

**输入**：需求文档（PRD）/ 代码文件 / 两者混合

**输出**：
- `业务逻辑.md`：从需求视角提炼的可测试断言
- `代码逻辑.md`：从代码视角提炼的可测试断言
- `index.json`：模块元数据（断言序号、代码路径、模块状态）

**断言格式**：

```
[assert-AUTH-001] 【权限】（✓）登录接口 · 正确用户名密码 → 返回 JWT token | upd:v1.0
[assert-AUTH-002] 【输入约束】（✓）密码字段 · 少于 8 位 → 前端拒绝提交 | upd:v1.0
[assert-AUTH-003] 【异常】（待代码确认）登录 · 账号被锁定时 → 返回具体错误码 | upd:v1.0
[assert-AUTH-004] 【上下游】（✓）登录成功 · token 写入后
  → [用户中心::会话管理] 创建活跃会话记录 | upd:v1.0
```

**置信度说明**：

| 标注 | 含义 | 测试建议 |
|------|------|---------|
| `（✓）` | PRD + 所有代码源确认 | 正常覆盖 |
| `（✓,部分代码确认）` | PRD + 部分代码确认 | 正常覆盖，注明待核实 |
| `（待代码确认）` | 仅 PRD，无代码验证 | 高优先，设计能暴露实现问题的用例 |
| `（代码独立）` | 仅代码发现，PRD 未描述 | 按业务重要程度覆盖 |
| `⚠️` | PRD 与代码结论矛盾 | 暂停，人工确认后继续 |

**多仓库支持**：同时分析多个代码仓库（如前端 + 后端），synthesis 层自动识别来源（`[来源:frontend]`、`[来源:backend]`）、聚合置信度、标注矛盾点。

---

### qa-functional-test：功能测试用例设计

**输入**：qa-understand 产出的断言集合

**核心流程**：

1. 先问：如果这个模块今天出问题，用户最先感受到什么损失？
2. 定义 2-3 个优先测试场景（多角度覆盖）
3. `⚠️` 和「待代码确认」断言追加高优先用例
4. 其余断言按覆盖规则处理

**用例格式**：

```
用例 ID：TC-AUTH-001
标题：正确账号密码登录返回有效 token
功能点：用户登录 > 账号密码登录
版本：v1.0
前置条件：存在状态正常的用户账号，用户未登录
测试步骤：
  1. 输入正确的用户名和密码，点击登录
     → 预期：接口返回 HTTP 200，响应体包含有效的 JWT token
  2. 使用该 token 访问需要认证的接口
     → 预期：接口正常响应，不返回 401
优先级：P0
测试维度：正向
```

**可选产物**：FreeMind `.mm` 格式脑图，可直接导入 XMind / FreeMind / MindManager。

---

### qa-case-review：用例质量评审

从三个层次系统性检查：

| 层次 | 检查内容 |
|------|---------|
| **结构层** | 字段完整性；触发性步骤是否有内联预期 |
| **覆盖层** | 与断言基准对比，每条断言是否有对应 TC |
| **归因层** | 发现缺口时溯源：是用例设计遗漏，还是断言理解遗漏 |

输出评审报告，含阻断（必须修复）/ 需改进 / 建议三级。

## Story 设计原则

### 只维护当前态，变更追踪入 changelog

内容文件按功能点/组件分节，变更时原地更新：

```markdown
## 功能点：账号密码登录
> last-upd:v1.1

[assert-AUTH-001] 【权限】（✓）登录接口 · 正确凭据 → 返回 token | upd:v1.0
[assert-AUTH-005] 【输入约束】（✓）密码 · 超过 128 位 → 截断处理 | upd:v1.1
```

变更原因写入对应的 `.changelog.md`，正常执行不加载，只在溯源时读。

### 三层加载，频繁访问的层最精简

```
Layer 1  index.json      每次任务必读，极小，全局概览
Layer 2  内容文件         按需读，只读当前需要的模块
Layer 3  changelog        正常任务不读，只在溯源时用
```

### 跨模块引用结构化标签

```
[assert-ORDER-012] 【上下游】订单创建 · 支付成功后
  → [库存::扣减] 触发库存扣减事务 | upd:v2.0
```

`index.json` 的 `depends_on` 字段从这些标签自动聚合，不单独维护，防止幽灵数据。

## 扩展指南

### 新增技术栈能力（如性能测试）

1. 在 `practices/tech-stacks/` 下新建技术栈规范目录
2. 新建 `capability/qa-{name}/SKILL.md`，定义该领域的思考哲学
3. 在 `testcraft/SKILL.md` 的能力列表中注册并设置状态为 ✅

### 当前已内置但待建设的能力（Phase 2）

- `qa-auto-test`：自动化测试代码生成
- `qa-performance-test`：性能/压力测试设计
- `qa-security-test`：安全测试设计
- `qa-chaos-test`：混沌测试设计

## 与原生 AI 测试对比

| 维度 | 原生 AI（如 Claude Code）| testcraft |
|------|------------------------|-----------|
| 单次任务质量 | 高（训练内化的测试直觉）| 相当，方向一致 |
| 跨会话积累 | ✗ | ✓ story 系统 |
| 多模块覆盖管理 | ✗ | ✓ index.json |
| PRD→断言→TC 链路 | ✗（黑盒）| ✓（显式可查）|
| 团队/多模型协作 | ✗ | ✓（story 是共享知识库）|
| 版本变更影响分析 | ✗ | ✓（changelog + 置信度）|

**核心定位**：testcraft 不是替代 AI 的测试能力，而是让 AI 的测试工作可积累、可追溯、可跨会话延续的知识基础设施。原生 AI 提供智力，testcraft 提供记忆和结构。

## 设计哲学速查

| 原则 | 核心思想 | 判断标准 |
|------|---------|---------|
| AI 是主体，框架是支撑 | 框架给工具和边界，不规定每个动作 | 违反 = 把 AI 变成流水线 |
| Capability = 思考哲学 | 装备思考方式，不是执行步骤清单 | 改 capability 前问：这改变了思考方式吗？ |
| Practices = 公司规范 | 防失控但不过度规范，保留智力空间 | 改 practices 前问：在防止真实失控风险吗？ |
| Story = 项目上下文 | 知识持久化，支持跨会话/跨模型协作 | 改 story 结构前问：AI 能快速理解上下文吗？ |
| 渐进式披露 | 按上下文依赖切割，不按角色切割 | 改加载时机前问：会造成上下文断流吗？ |

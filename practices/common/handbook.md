# qalore 通用规范

## story 维护约定

### 增量原则

所有 story 文件只增不删（changelog 类文件头部插入新条目，内容文件按变更类型操作），**禁止全量覆盖**：

| 文件 | 写入方式 |
|------|---------|
| `{模块名}-功能-业务逻辑.md` | 新增 / 覆盖更新 / 删除（按变更类型，见「断言变更规则」） |
| `{模块名}-功能-业务逻辑.changelog.md` | **只增不删**（新条目头部插入，旧条目不动） |
| `{模块名}-功能-代码逻辑.md` | 新增 / 覆盖更新 / 删除（按变更类型，见「断言变更规则」） |
| `{模块名}-功能-代码逻辑.changelog.md` | **只增不删**（新条目头部插入，旧条目不动） |
| `{模块名}-功能-测试用例.md` | 新增 / 覆盖更新 / 删除（按变更类型，见「用例变更规则」） |
| `{模块名}-功能-测试用例.changelog.md` | **只增不删**（新条目头部插入，旧条目不动） |
| `{项目名}/index.json` | 只增不删条目 |

### 自解释原则

每个 story 文件必须能被全新实例的 AI 独立读懂，不依赖任何对话上下文：
- 禁止使用「参见上次讨论」「同上」等表述
- 变更原因必须明确记录在 changelog 中
- 所有引用必须包含完整路径，不依赖隐含约定

### 断点恢复规范

每完成一个模块立即写入，不等全部完成后批量写入。

会话中断后恢复流程：

```
读 story/{项目}/index.json → 获取全部模块列表
    ↓
逐模块检查 status 字段，了解当前状态：
  status.pending_count > 0   → 有待确认项未解决，提示用户处理后再继续（不强制阻断）
  status.business_logic = false → 尚无业务逻辑，从 qa-understand 开始
  status.code_logic = false  → 代码逻辑未提炼（可选，视需求决定是否需要）
  status.tc_count = 0        → 尚无测试用例，从 qa-functional-test 开始
    ↓
从第一个未完成状态的模块继续工作，已完成的模块跳过
```

### 待确认项生命周期规范

待确认项有三种状态，状态标记写在条目行首：

| 状态 | 含义 | 迁移条件 |
|------|------|---------|
| `[pending]` | 已提出，等待确认 | 初始状态 |
| `[resolved]` | 已确认，答案已转化为正式断言 | AI 后续分析发现答案，或用户提供确认信息 |
| `[dismissed]` | 已确认，结论是不需要测试或超出范围 | 用户明确说明无需覆盖 |

**状态迁移规则：**
- `pending → resolved`：在条目末尾追加确认日期和来源，同时在断言列表中新增对应的正式断言（分配 assert ID）
- `pending → dismissed`：在条目末尾追加确认日期和原因
- 已 resolved / dismissed 的条目**不得删除**，保留完整历史

**`resolved` 判定标准：**
`pending → resolved` 迁移必须同时满足：
1. 有明确的代码证据或用户确认
2. 回答的是"是什么"的问题（实现细节、参数值、接口名称）
3. **不能**用于回答"应不应该"的问题（UX 设计、架构决策、产品方向、兼容性策略）

以下类型即使代码有明确实现，也不得标记为 resolved，须保持 `[pending]` 并标注团队归属：
- 架构决策（如组件是否拆分、是否懒加载）
- UX 行为（如加载时长、降级策略、防抖时长）
- 产品流程（如创建后是否跳转配置页）
- 兼容性策略（如 API fallback 是否需要统一）

以上类型应保持格式：`- [pending] 【{团队}】{问题描述}`，团队标识：`前端` / `后端` / `PM`。

**`index.json` 中 `pending_count` 字段：**
- 值 = 当前模块待确认项中 `[pending]` 状态的数量
- 每次写入业务逻辑.md 或代码逻辑.md 后同步更新
- `pending_count > 0` 时，断点恢复流程提示先处理待确认项

### 断言变更规则

需求或代码变更时，对业务逻辑.md 和代码逻辑.md 中的功能点按以下三种情况处理，**不得混用**：

| 情况 | 判断标准 | 操作 |
|------|---------|------|
| 3.1 新增 | 变更引入了新功能点，story 中尚无对应章节 | 追加新功能点章节（`## 功能点：{name}` 或 `## 组件：{name}`），含全部断言 |
| 3.2 更新 | 功能点仍然存在，但业务规则或实现逻辑发生变化 | 找到对应章节，原地覆盖变更的断言行；同步刷新该断言的 `\| upd:{版本/日期}` 标记和节头 `> last-upd:{版本/日期}` |
| 3.3 删除 | 功能点被移除，或旧逻辑被新逻辑完全取代 | 从文件中完整删除该功能点章节 |

三种操作均须同步写入对应的 `.changelog.md`，格式见 `tech-stacks/functional/changelog.md`「断言 changelog 格式」章节。

---

### 用例变更规则

需求变更时，对已有用例按以下三种情况处理，**不得混用**：

| 情况 | 判断标准 | 操作 |
|------|---------|------|
| 3.1 新增 | 需求引入了新功能点，story 中尚无对应用例 | 追加新用例条目 |
| 3.2 更新 | 功能点仍然存在，但业务规则、字段约束或交互逻辑发生变化 | 原地覆盖整条用例内容，字段逐一替换 |
| 3.3 删除 | 功能点被需求移除，或旧校验规则被新规则完全取代 | 从文件中完整删除该用例条目 |

**判断 3.2 vs 3.3 的原则：**
- 测试目标（验证什么）没有改变 → 3.2 更新
- 测试目标本身已不存在或被替换 → 3.3 删除

三种操作均须在 changelog 中记录，格式见 `tech-stacks/functional/changelog.md`「changelog 格式」章节。

## story 写入协议

### 写入声明格式

Capability 执行完成后，在产出末尾输出【待沉淀】声明，声明需要写入的 story 文件：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {文件名} | {完整目录路径}/ | {操作} | {一句话描述变更内容} |
```

**操作类型：**
- `新建`：文件不存在时创建
- `只增不删`：changelog 类文件专用：新条目头部插入，旧条目不动，永不删除
- `3.1新增 / 3.2更新 / 3.3删除`：**所有内容文件**的变更操作，适用于业务逻辑.md（见「断言变更规则」）、代码逻辑.md（见「断言变更规则」）、测试用例.md（见「用例变更规则」）

### 多 capability 顺序执行规则

capability 按依赖顺序顺序执行（非并发），每个 capability 执行完后**立即**输出自己的【待沉淀】并等待用户确认，不等待后续 capability 完成。

- qa-understand 执行完 → 输出业务/代码逻辑文件的【待沉淀】→ 用户确认 → 写入
- qa-functional-test 执行完 → 输出 TC 文件的【待沉淀】→ 用户确认 → 写入

每次确认只对当前 capability 的产物负责，不延迟等待合并。

### 确认与执行

用户确认后：
- 所有文件并行写入
- index.json 的多条变更合并后一次写入，不分步
- 遵循本文件「增量原则」和「用例变更规则」

**内容文件写入规程（操作类型为新增/更新/删除时必须遵守）：**
1. 先 Read 目标文件，将现有内容写入 context
2. 在现有内容基础上合并变更（追加新功能点章节 / 覆盖指定章节内断言 / 删除功能点章节）
3. 每条被新增或更新的断言行末须携带 `| upd:{版本/日期}` 标记；节头同步刷新 `> last-upd:{版本/日期}`
4. 将合并后的完整内容 Write 回文件
5. 不得跳过步骤 1 直接 Write——跳过步骤 1 等同于全量覆盖，将清空历史内容
6. 内容文件写入完成后，**必须**同步追加写入对应的 `.changelog.md`，记录本次变更

**写入校验原则：** 内容文件头部声明的计数字段（断言数、功能点数、pending 数）必须在写入前从待写入内容直接统计，禁止手算或预估。写入后仅做轻量传输完整性验证。具体流程由各 capability 的 SKILL.md 定义。

用户拒绝：当次产出不写入任何 story 文件，流程终止。

### index.json 变更声明规则

写入声明中 index.json patch 条目需明确字段路径和值，schema 见 `{practices_path}/schemas/story-index.schema.md`（权威来源）：
- 新模块首建：声明 `modules.{模块名}` 全部初始字段及值
- 字段更新：只声明变更的字段，不涉及的字段不写入声明
- 禁止声明删除已有条目

## 沟通规范

### 必须暂停向用户确认的情况

- 需求描述有歧义，无法确定测试范围
- 无法从任务描述中识别项目名称
- 无法从输入类型和用户描述中确定产物输出范围（全量 / 版本级 / 模块级 / Feature 级）
- 执行计划展示后，等待人类 同意 / 修正 / 拒绝
- story 写入前，等待人类确认待写内容
- 发现 story 中已有数据与当前任务存在冲突
- practices 写入前，等待人类确认待写内容（展示【practices 写入确认】表格）
- 代码阅读中发现无法访问的外部仓库或中间件（外部仓库、第三方服务、未提供的代码仓库等）→ 不单独打断，将所有「外部依赖断点」汇总，在执行计划确认环节集中向用户提问：「发现以下外部依赖，能否提供对应代码或配置？{列出依赖名称和发现位置}」；用户能提供 → 继续探索（最多再追一层，满足即止）；用户不能提供 → 记录为待确认项「链路断点」，不阻断当前任务

### 可自行决策的情况

- 测试用例的具体设计（维度覆盖、步骤描述、预期结果表述）
- 业务逻辑提炼的结构和组织方式
- 在已建设能力范围内选择合适的测试方法

## 执行前确认规范

每个 capability 在执行任何写入或产物生成操作前，必须向用户展示执行计划并等待确认。这是独立于「story 写入协议」的前置检查点。

**通用协议：**
- 完成分析、确定执行范围后，展示计划摘要
- 等待用户响应：
  - 同意 → 执行
  - 修正 → 调整后重新展示
  - 拒绝 → 终止
- 计划摘要的具体字段由各 capability 的 SKILL.md 定义；各 capability 可根据任务复杂度增减字段，简单任务可使用精简格式

**强制包含字段（各 capability 均须包含）：**
- 项目名 / 模块名
- 执行操作范围（会产生或修改哪些产物）
- 当前 practices 规范版本

## 多模块任务规范

多模块任务 = 用户一次请求涉及多个模块（如「给 A、B、C 三个模块出用例」）。

**执行节奏（由 qa-functional-test 自主驱动，网关完成路由后不介入模块间调度）：**
1. 按模块顺序逐个执行全流程（理解 → 设计 → story 写入），每完成一个立即写入，不等全部完成后批量写入
2. 每完成一个模块，告知用户：「{模块名} 已完成，继续下一个（{模块名2}）？」
3. 全部模块的 story 写入完成后，统一执行一次 .mm 产物输出（中间不生成 .mm）

写入节奏遵循「story 维护约定 · 断点恢复规范」章节。

## context 标记规范

practices 文件的按需加载通过 context 标记实现，所有 capability 统一遵守此规范。

**标记格式：** `【practices:{文件简称}:loaded】`

**示例：**
```
【practices:handbook.md:loaded】
【practices:cases.md:loaded】
【practices:assertions.md:loaded】
【practices:output.md:loaded】
【practices:story-formats.md:loaded】
```

**规则：**
- 标记存在 → 该文件内容在 context 中可用，直接使用，跳过 Read（0 token）
- 标记不存在（新会话 / context 被压缩 / 未加载）→ 执行 Read，加载后立即写入标记
- 网关在每次任务开始时验证 `【practices:handbook.md:loaded】`；其余标记由各 capability 在用到时自行检查，规则相同
- practices 版本更新时，网关重新加载并覆写对应标记；capability 无需处理版本判断

### context 标记注册表

以下为 qalore 生态中所有 context 标记的权威注册表。每个标记定义其写入者、读取者、语义、以及 context 压缩后的恢复策略。

**基础设施标记（网关维护）：**

| 标记 | 写入者 | 读取者 | 语义 | 丢失恢复 |
|------|--------|--------|------|---------|
| `【practices:handbook.md:loaded】` | gateway | gateway, 所有 capability | handbook.md 已加载到 context | 重新 Read handbook.md，写入标记 |
| `【practices:{file}:loaded】` | 各 capability | 同一 capability（后续调用） | practices 文件已按需加载 | 重新 Read 对应文件，写入标记 |

**调度层标记（qa-understand 调度层维护）：**

| 标记 | 写入者 | 读取者 | 语义 | 丢失恢复 |
|------|--------|--------|------|---------|
| `【qa-understand-mode: single】` | qa-understand 调度层 | text.md, code.md | 单源模式，适配器独立执行 | 检查当前任务信息源数量重建 |
| `【qa-understand-mode: multi】` | qa-understand 调度层 | text.md, code.md, synthesis.md | 多源模式，适配器顺序执行，synthesis 统一产出 | 检查当前任务信息源数量重建 |
| `【assert_seq_runtime: N】` | text.md → code.md（按序传递） | 下一个适配器 | 跨适配器共享的当前最大 assert ID 序号 | **已废弃**（见问题4修复：改为每个适配器完成后立即更新 index.json 的 assert_seq，消除 context 状态依赖） |
| `【code-source: {仓库名}】` | qa-understand 调度层 | code.md | 当前代码适配器对应的仓库名，用于多仓库来源标注 | 从用户提供的代码路径重新识别 |

**交接标记（capability 间通信）：**

| 标记 | 写入者 | 读取者 | 语义 | 丢失恢复 |
|------|--------|--------|------|---------|
| `【测试意图已提炼】` | synthesis.md（qa-understand 完成时） | qa-functional-test, qa-case-review | qa-understand 已完成，统一交接块可用 | 从 story 文件重建：读业务逻辑.md + 代码逻辑.md |
| `【待沉淀】` | 各 capability（写入声明时） | 用户（确认写入） | 待写入 story 的文件清单 | 不可恢复——需重新执行 capability |

**用户界面标记（各 capability 执行计划展示）：**

| 标记 | 写入者 | 语义 |
|------|--------|------|
| `【测试意图理解执行计划】` | qa-understand | 执行前确认：理解阶段 |
| `【功能测试执行计划】` | qa-functional-test | 执行前确认：用例设计阶段 |
| `【用例评审执行计划】` | qa-case-review | 执行前确认：评审阶段 |
| `【用例评审报告】` | qa-case-review | 评审结果展示 |
| `【冲突提示】` | synthesis.md | 多源置信度冲突展示 |

**约束：**
- 新增标记必须先在此注册表注册，标注写入者、读取者、语义和恢复策略
- 标记名称不得包含可变数据（如 `【assert_seq_runtime: N】` 应在修复后不再使用）
- context 压缩后丢失的标记，必须按注册表中声明的恢复策略重建，不得使用猜测值

以下为 handbook.md 中所有可被外部文件引用的章节名称。capability SKILL.md 和 adapter 文件通过 `「章节名」` 格式引用这些章节。章节名的增删改必须同步更新此注册表。

| `评审规范` | 问题严重级别和结论判定规则 | qa-case-review |

**约束：**
- 重命名章节 → 必须同步更新此注册表和所有引用该章节的外部文件

- 删除章节 → 必须先检查外部引用，移除所有依赖后再删除
- 审计工具通过比对此外部引用和注册表验证跨文件引用的完整性

## 计数一致性规范

所有 story 文件中的计数字段必须在实际内容写入后、从文件内容统计得出，**禁止使用预估数**：

| 字段 | 统计方式 |
|------|---------|
| `assert_seq` | `max(BL 最大 assert ID, CL 最大 assert ID)` |
| `tc_count` | `grep -c "^用例 ID：" {TC文件}` |
| `pending_count` | `BL pending 数 + CL pending 数` |
| BL 文件头"断言：N 条" | `grep -c "\[assert-" {BL文件}` |
| TC 文件头"用例总数：N 条" | `grep -c "^用例 ID：" {TC文件}` |

qa-functional-test 写入完成后，**立即**执行上述统计验证，将实际值写入 index.json 和文件头。

## 文件操作规范

### JSON 文件（index.json）
- 结构修改（增删字段）→ 必须 Read → Python `json.load()` → 修改 → `json.dump()` → Write
- **禁止**对 JSON 文件使用行级 Edit 操作（极易破坏 JSON 结构）
- 写入后必须 `json.load()` 验证合法性

### 内容文件（.md）
- 必须先 Read 再 Write（禁止跳过 Read 直接覆盖）
- 批量修改脚本执行后必须验证格式完整性：
  - TC 文件：确认 TC 总数未变、每条 TC 含 7 个必填字段
  - BL/CL 文件：确认断言总数未变、所有断言行末有 `| upd:` 标记
  - 确认无重复节头（`grep "^## " | sort | uniq -d`）
  - 确认文件头与章节间有空行分隔

## 通用性约束

qalore 的 capability 和 practices 文件是通用测试框架，服务于任意项目。**禁止**在以下文件中出现项目名、产品名、公司名或业务术语（如具体系统名、模块名、入口名）：

| 范围 | 文件 |
|------|------|
| capability | 所有 SKILL.md、所有适配器文件（`adapters/*.md`） |
| practices | 所有 `common/*.md`、`tech-stacks/*.md`、`schemas/*.md`、`practices-bootstrap.md` |
| practices/index.json | 顶层字段和注册表条目 |

**例外：**
- `practices/index.json` 的 changelog 中通过 `story:` 前缀引用的故事文件路径是历史数据，不在本约束范围内
- 格式示例中的占位符（如 `{项目名}`、`{模块名}`）不视为污染

任何 capability 或 practices 文件的作者、修改者在写入或新增示例、描述、代码片段时，**必须使用通用占位符或抽象描述**，不得直接嵌入执行时接触到的具体项目名称。

## 评审规范

适用于所有类型的测试用例评审，通用评判标准，与技术栈无关。

### 问题严重级别

| 级别 | 定义 | 处置建议 |
|------|------|---------|
| **阻断** | 用例无法执行，或执行后无法判断是否通过 | 必须修复后才能纳入测试计划 |
| **需改进** | 覆盖有漏洞或断言表述模糊，存在漏测风险 | 建议在本轮测试前修复 |
| **建议** | 质量可提升，不影响当前执行 | 下次迭代优化 |

### 结论判定规则

| 条件 | 结论 |
|------|------|
| 阻断 = 0 | 通过 |
| 阻断 > 0，需改进 ≤ 3 | 需修复后通过 |
| 阻断 > 0，需改进 > 3 | 不通过 |

# qalore 审计规范

每次审计按十七步流水线执行，一步通过才进入下一步。每步的检查方法可独立用于日常增量审计。

---

## 十七步审计流水线

### 步骤1：JSON 结构完整性（F）

**目标**：所有 JSON 可解析，结构一致，无静默错误。特别注意结构体交界处。

```
1. json.load() 全部 JSON 文件
2. version == changelog[0].version
3. changelog 条目数单调递增
4. 无重复 key（Python json.load 静默取最后值，需人工检查 editor diff）
5. 边界专项：每次 changelog 头部插入后，验证 [{...},{...}] 的 },{ 边界完整
   — changelog 条目之间的 },{ 是最脆弱的编辑点
6. 版本号连续性（日期格式专用）：
   — 对同一日期的版本序列（如 2026-06-12-vN），检查 v 值从最大值连续递减到 1，无跳号
   — 不连续 → 标记缺失版本号（如 v12→v10 跳过 v11）
7. changelog 条目 version 字段必填检查：
   — 遍历 changelog 所有条目，逐一检查 `version` 字段存在性
   — 缺失 → 标记「无法索引的 changelog 条目」
```

**适用范围**：`practices/index.json`、全部 `story/{项目}/index.json`。

**执行纪律**：每次 `Edit` 后立即 `json.load()` 验证——不等一轮审计结束再统一检查。JSON 损坏最多发生在 changelog 头部插入后的 `},{` 边界，立即验证可将断裂半径限制在单次编辑内。

---

### 步骤2：注册表 ↔ 文件系统双向校验（B）

**目标**：注册表中的每个条目对应磁盘文件，磁盘上的每个文件在注册表中有条目。

```
正向（注册表 → 磁盘）：
  遍历 index.json 中 common 和 tech_stacks 下所有 file 字段
  → 逐一 os.path.exists()
  
反向（磁盘 → 注册表）：
  遍历 practices_path 下所有 .md 文件
  → 逐一检查是否在注册表中
  → 不在 = 未注册（遗漏或废弃文件）
```

**适用范围**：`practices/index.json` 的 `common` / `tech_stacks` 注册段 + `practices_path` 磁盘文件。

---

### 步骤3：Schema ↔ 数据双向比对（A）

**目标**：Schema 定义的字段在数据中存在且格式正确；数据中存在的字段在 Schema 中有定义。

```
正向（Schema → 数据）：
  对 Schema 中每个字段：取至少 1 个真实数据实例
  → 字段存在？类型匹配？值域合法？
  → 枚举型字段：检查所有实例

反向（数据 → Schema）：
  取 1 个完整数据实例，列出所有字段路径
  → 逐一在 Schema 中找到对应定义
  → 找不到 = Schema 遗漏
```

**适用范围**：`schemas/story-index.schema.md` Schema ↔ 全部 `story/{项目}/index.json`。

**枚举型格式字段专项**（适用于 Schema 定义了内部格式模式的 string 字段）：
```
  — 对 Schema 中定义了格式模式的 string 字段（如 {X}::{Y}、YYYY-MM-DD、TC-{PREFIX}-{seq}）
  — 提取所有数据实例，正则验证格式合规率
  — 合规率 < 100% → 逐项列出不合规实例及其所属模块
  — 特别注意部分正确但格式不完整的条目（如「模块名」存在但缺少「::功能点名」）——这类条目比完全缺失更隐蔽，仅靠"字段存在"检查无法发现
```

### 步骤4：计数字段完整性校验（C）

**目标**：所有可从源数据推导的计数字段与源一致。

```
对每个计数字段：
  1. 从源数据重新计算实际值
  2. 与存储值比对
  3. 不一致 → 数据漂移，标记差异

计数字段清单：
  - pending_count ← BL.md + CL.md 中 [pending] 行数
  - tc_count       ← TC.md 中 用例 ID：TC- 出现次数
  - assert_seq     ← BL.md + CL.md 中最大 assert ID 序号
  - BL 头部增量    ← BL changelog 声称的断言数变更（如"31→47"）vs BL 文件新旧版本断言数差异
                   若 changelog 无计数声明 → 标记「changelog 缺少计数声明」
```

**适用范围**：全部 `story/{项目}/index.json` 的 `status` 字段 + 对应的内容文件。

---

### 步骤5：格式合规率采样（D）

**目标**：对每条格式规则，统计所有应遵守实例的合规比例。

```
对每条格式规则：
  1. 列出所有应遵循该规则的实例（文件、条目、章节）
  2. 逐个检查合规性
  3. 报告合规率 = 合规数 / 总数
  4. 合规率 != 100% → 逐项列出不合规实例

关键规则清单：
  - changelog 断言格式           ← 所有 *changelog.md 文件
  - TC 来源标记规则              ← 所有测试用例.md 中备注含"实现待确认"或位于代码独立节
  - mm_short_id 规则             ← 所有 story index.json 模块条目
  - 置信度标注单一权威           ← story-formats.md 全文（查重复定义）
  - 交叉引用章节名               ← 所有 SKILL.md 中 「...」引用
  - 通用性合规                   ← 所有 capability SKILL.md + practices .md 文件（不含 index.json changelog）
                                 搜索项目名/产品名/业务术语（如具体公司名、系统名、模块名），
                                 命中 → 标记「通用框架含项目定制内容」
                                 注意：practices/index.json changelog 中的 story: 路径引用是历史数据，不在此规则范围内
                                 注意：practices 格式示例中的占位符（如 {项目名}、{模块名}）不视为污染
	  - subagent prompt 模板完整性   ← cloud-sync.md「subagent 执行约束」章节
	                                 检查：硬约束是否覆盖全部 practices 规则
	                                 检查：标识符速查是否与同文件「字段 Identifier 速查表」一致
	                                 检查：API 模板是否与同文件「API 能力矩阵」一致
	                                 检查：模板占位符是否在 capability 的 prompt 构造指令中被全部引用
	                                 任一不一致 → 标记「subagent prompt 模板不完整」
```

**判断原则**：找到"至少一个正确例子" ≠ 规则被遵守。必须统计全部实例。

---

### 步骤6：元数据 ↔ 行为对齐（E）

**目标**：元数据描述（when_to_load、字段所有权、状态字段）与实际系统行为一致。

```
对每条元数据：
  1. "谁写的" → 对照代码中实际写入该字段的 capability
  2. "何时加载" → 对照 gateway 和各 capability 的 Read 调用
  3. "状态是什么" → 对照实际文件存在性和内容

关键元数据清单：
  - index.json when_to_load     ← 对照 gateway practices 加载 节
  - practices-bootstrap 字段所有权表 ← 对照各 adapter/capability 写入代码
  - story index.json status.*   ← 对照实际文件是否存在
```

---

### 步骤7：原子变更验证（G）

**目标**：规范中用"缺一不可""必须同步"描述的写入对，在数据中逐对验证。

```
从字段所有权表和写入规程提取所有原子对：
  BL.md 写入  → pending_count 同步更新
  CL.md 写入  → pending_count 同步更新
  TC.md 写入  → tc_count 同步更新
  assert 分配 → assert_seq 同步更新
  内容文件写入 → changelog 同步追加
  跨模块波及   → 1-hop 模块 BL 同步更新
    — 若模块 A 的 BL changelog 含删除/废弃条目，且模块 B（∈ A.business_related ∪ A.depends_on）
      的 BL 文件中含引用被删除对象的断言，则 B 的 BL changelog 应有同期的 3.2 更新条目
    — 无 → 标记「跨模块原子性断裂」

对每个原子对：
  → changelog 有对应日期的条目 → 内容文件同日期有变更？
  → pending_count 的变更量 = BL/CL 中 [pending] 净增减？
  → tc_count 的变更量 = TC 文件中用例 ID 净增减？
```

**适用范围**：全部 story/{项目}/index.json 的计数字段 + 对应的 changelog 和内容文件。

**与步骤4（C）的区别**：步骤4 检查"当前值是否等于源推导值"（静态快照），本步骤检查"两个字段是否在同一个变更事件中一起更新了"（动态一致性）。

---

### 步骤8：示例即契约检查（H）

**目标**：文档中的每个具体示例都能在真实数据中找到匹配实例。示例不一致比没有文档更危险。

```
提取所有 practices 文件中的具体示例：
  代码块中的 JSON/XML/markdown 片段
  "如 xxx"、"例：xxx" 后面的值
  ID 格式示例（如 p001、TC-PIPE-001）
  初始化/最小格式类示例

对每个示例：
  → 找至少 1 个真实数据实例与该示例格式一致
  → 找不到 → 示例过时 或 数据偏离规范
  → 两者都可能是正确的，但不可能同时正确

特殊规则：
  → 初始化/最小格式类示例 → 直接用于 bootstrap 验证（能否用此模板创建合法文件）
  → 格式规范类示例 → 必须与实际数据一致
```

**适用范围**：全部 practices .md 文件中的示例 ↔ 全部 story 数据文件；practices 内部自引用示例（如 bootstrap 最小格式 ↔ practices/index.json 实际格式）。

**与步骤5（D）的区别**：步骤5 检查"数据是否遵循规则"，本步骤检查"文档示例是否真实"——方向相反。规则正确 + 示例错误 = 规范本身有 bug。

---

### 步骤9：注册表同质性检查（I）

**目标**：同一个注册表/清单中的所有条目属于同一语义类别。类别混杂会导致标记含义不一致。

```
对每个注册表/清单：
  1. 提取所有条目，从行为推导其语义类别（不依赖标签文字）
  2. 分类标准：用户可路由触发 / 基础设施自动触发 / 计划中未创建 / 已废弃
  3. 同一清单中出现 ≥2 个类别 → 清单不同质，标记
  4. 对标记的清单：建议按类别拆分为独立表格或章节
```

**适用范围**：gateway capability 列表、practices/index.json 中的文件注册表、story/index.json 的模块列表。

**判断原则**：标签上写「✅ 可用」但一个指"用户可触发"、另一个指"hook 自动运行"、第三个指"文件未创建"——这就是同质性断裂。关键不是标签的样子，是标签背后的行为类别。

---

### 步骤10：状态通道盘点（J）

**目标**：所有组件间的隐式通信通道都有明确的契约（写入者、读取者、格式、恢复策略）。

```
1. 扫描所有组件文件，提取所有共享状态通道：
   — context 标记（【...】格式）
   — 临时文件 / 环境变量
   — 跨 capability 传递的内存状态
2. 对每个通道检查四要素：
   — 写入者：谁创建/更新这个状态？
   — 读取者：谁消费这个状态？
   — 格式：数据的 schema 是什么？有没有可变数据嵌入？
   — 恢复策略：状态丢失（context 压缩、文件被删）后如何重建？
3. 任一要素缺失 → 标记「契约不完整」
4. 特别标记「存储不可恢复数据」的通道（如递增计数器、用户确认结果）→ 高风险
5. 有恢复策略的通道，验证策略是否真的可执行（不是"从 context 重新读"这种循环依赖）
```

**适用范围**：所有 `【...】` context 标记、capability 间传递的交接块、practices 文件的 loaded 标记。

**与哲学9（会话边界即状态边界）的关系**：本步骤是哲学9 的工程化落地——哲学9 诊断了跨会话状态丢失的必然性，本步骤提供系统方法发现所有缺乏契约的状态通道。

---

### 步骤11：消费者-生产者版本契约（K）

**目标**：每个消费 practices 文件的组件声明了它期望的最低版本，且该版本与当前 practices 版本兼容。

```
对每个消费者-生产者关系：
  1. 生产者是否有版本号？
     — practices: index.json 的 version 字段
     — story 内容文件: index.json 的 prd_version / last_updated
  2. 消费者是否声明了最低版本要求？
     — capability SKILL.md: practices_min_version
  3. 消费者未声明 → 标记「隐式依赖」（格式变更时无告警）
  4. 消费者声明的版本 > 生产者当前版本 → 标记「不兼容」
  5. 消费者声明的版本 ≤ 生产者当前版本 → 通过
```

**适用范围**：
- practices ↔ capability（如 qa-functional-test 消费 cases.md）
- capability ↔ capability（如 qa-functional-test 消费 qa-understand 交接块）
- story 文件 ↔ practices（如 TC 文件格式依赖 cases.md 规范版本）
- 基础设施组件 ↔ practices（如 qa-token-report 由 Hook 触发，其版本依赖同样应显式声明）

**强制执行可达性检查**（版本声明的校验路径必须存在）：
```
对每个声明了 practices_min_version 的消费者：
  → 追溯到负责校验此声明的组件（当前为 gateway 的环境验证阶段）
  → 验证该校验组件是否会在运行中读取此消费者的 SKILL.md
  → 不会被读取 → 标记「孤立版本声明：声明存在但永远不会被校验」
  → 注意：校验机制中的"遍历所有 capability" ≠ "遍历被触发的 capability"——前者覆盖基础设施，后者只覆盖用户能力
```

**与步骤1（F）的区别**：步骤1 检查单个文件的版本自洽（version == changelog[0].version），本步骤检查跨组件的版本兼容——两个文件各自自洽，但放在一起可能不兼容。

### 步骤12：权威-副本一致性检查（L）

**目标**：所有存在"权威源→副本"关系的文件对，副本内容与权威源一致。副本不是便利——是待偿还的同步债务。

```
1. 扫描 practices 生态，识别所有"权威-副本"关系：
   — 字段定义：schemas/story-index.schema.md → practices-bootstrap.md（schema 节）
   — 规范内容摘要：handbook-audit.md → index.json（summary 字段）
   — 格式示例：practices/index.json 实际格式 → bootstrap 最小格式示例
   — 注册表条目：index.json 各条目 → 外部引用这些条目的文字描述
   — 章节标题计数：handbook-*.md 实际步骤/哲学数量 → 标题中的数字
   — 标记状态：context 标记注册表（handbook.md）→ 各 SKILL.md 中对标记的引用

2. 对每个关系：
   — 提取权威源的当前状态（字段类型、步骤数量、版本格式等）
   — 提取副本的对应描述
   — 不一致 → 标记「副本过期」

3. 特别标记：
   — 描述中嵌入的计数（"六步""十条"等）—— 这些是文本层面的缓存
   — 副本中与权威源矛盾的字段类型/约束（如 number vs string）
   — 副本中使用了与权威源不一致的示例格式
```

**适用范围**：practices 内部文件间的权威-副本关系（schema↔bootstrap、handbook↔index.json summary、bootstrap 示例↔practices 实际格式）、capability SKILL.md 中对 handbook 章节和 context 标记的引用文字。

**与步骤3（A）的区别**：步骤3 检查「规范定义（Schema）vs 实际数据（story JSON）」，本步骤检查「权威文档 vs 其参考副本」——两方都是 practices 内的文档，不是数据文件。与步骤5（D）的关系：步骤5 检查数据是否遵循规则，步骤8（H）检查文档示例是否在数据中有匹配——本步骤更底层：检查同一事物的两份文档描述是否一致。

**与哲学13（副本即债务）的关系**：本步骤是哲学13 的工程化落地——哲学13 诊断了副本的必然漂移性，本步骤提供系统方法发现所有权威-副本不一致。

### 步骤13：Changelog 声明真实性验证（M）

**目标**：changelog 中声称的变更在对应文件中确实已被应用。changelog 条目中的行为声明是承诺，不是收据——未兑现的 changelog 比没有 changelog 更危险。

```
对 changelog 采样（最近 N 条，或变更类型含"修复/更新/改为/新增/移除"的条目）：
  1. 提取 changelog 中的可验证行为声明：
     — 标题/名称变更声明（如「A→B」「标题X→Y」）
     — 计数变更声明（如「N→M步」「X→Y条」）
     — 字段类型/值变更声明（如「number→string」「格式X→Y」）
     — 新增/删除节或文件的声明
  2. 对每条声明，读取对应文件的当前实际状态
  3. 声明 ≠ 实际 → 标记「changelog 声明未兑现」
     — 例如：changelog 说标题改成了 B，文件中标题仍是 A
     — 例如：changelog 说计数从 N 改到了 M，文件中实际计数是 K
  4. 特别标记：声明中嵌入了具体数字、名称、路径的条目——这类声明可自动验证，也最容易出错

判断原则：
  — changelog 条目的写入时刻与文件修改时刻之间存在时间差——changelog 可能写于修改之前（预设承诺），
    也可能写于修改之后（追记）。审计不追究时间差的原因，只验证最终文件状态是否与声明一致。
```

**适用范围**：`practices/index.json` changelog 全部条目 ↔ 对应 changed_files 的实际文件内容；`story/{项目}/index.json` changelog（如有）同样适用。

**与步骤7（G）的区别**：步骤7 检查「两个字段是否在同一个变更事件中一起更新了」（A 和 B 的同步性），本步骤检查「changelog 声称做的事，文件是否真的做了」（声明和现实的对应性）。这是两个独立维度——一对原子变更可以都写了（通过步骤7），但 changelog 声称的内容和实际写入的内容可以不同（被步骤13 捕获）。

**与哲学14（Changelog 是承诺不是收据）的关系**：本步骤是哲学14 的工程化落地——哲学14 诊断了 changelog 声明的易假性，本步骤提供系统方法验证每一条可验证声明。

---

### 步骤14：基础设施健康检查（N）

**目标**：skill 自身的可执行文件（脚本、配置）不包含阻碍可移植性或正确性的缺陷。审计不仅覆盖 practices/story 数据生态，也覆盖 skill 的运行环境。

```
1. 硬编码路径检查：
   — 扫描所有可执行文件（.py / .bat / .sh / .ps1）
   — 匹配 C:\Users\...、/home/...、/Users/... 等绝对路径模式
   — 命中 → 标记「硬编码用户路径：不可移植」

2. 环境假设检查：
   — Python 版本假设（如 3.8+ 特性）
   — OS 特定调用（如 win32api、osascript）
   — 文件系统布局假设（如假定 practices_path 在 D: 盘）

3. 错误处理检查：
   — 脚本是否静默吞异常（try/except pass 或 exit 0 在错误路径上）
   — 外部命令调用是否有超时机制

4. 配置外部化检查：
   — 密码/token/key 是否硬编码
   — 路径是否可配置（通过环境变量或配置文件）vs 硬编码
```

**适用范围**：skill 包中所有可执行代码文件（`capability/qa-token-report/read_usage.py`、`probe_hook.py`、`run_silent.bat` 等）、Hook 配置（`settings.json` 中引用的命令路径）、技能包外依赖（如 Python 解释器路径）。

**与其他步骤的关系**：本步骤是审计流水线中唯一检查 skill 自身运行环境的步骤，其余步骤全部聚焦 practices/story 数据生态。当 skill 在非原始开发环境中部署时（如不同用户、不同机器、不同 OS），本步骤发现的问题将直接导致功能不可用。

### 步骤15：跨模块引用一致性（O）

**目标**：当某个模块的 BL 变更涉及删除/废弃/重命名对象时，所有通过 `business_related` 或 `depends_on` 显式关联的模块中引用该对象的断言被同步更新。

```
触发条件（从 BL changelog 中检测以下关键词的条目）：
  · 删除 / 废弃 / 移除 / 不再 / 取消
  · 重命名 / 改为 / 合并
  · 角色名、功能点名、入口名的变更声明

1. 从触发条目中提取被变更的对象标识（角色名/功能点名/入口名/模块名）
2. 对每个被变更对象：
   — 扫描当前模块 business_related + depends_on 中列出的所有模块 BL 文件
   — 搜索引用该对象的断言行（文本匹配 或 [模块::功能点] 标签匹配）
   — 存在引用 → 检查该模块的 BL changelog 在同日期或之后是否有对应 3.2 更新条目
   — 无对应更新 → 标记「跨模块波及遗漏：{模块}::{断言ID} 引用了被删除的 {对象}」
3. 对每个被删除/废弃对象：
   — 扫描当前模块自身的 BL 文件
   — 搜索引用该对象的断言行（排除本次变更中已标记 3.3 删除的）
   — 存在未被删除的引用 → 标记「模块内波及遗漏：{断言ID} 仍引用已废弃的 {对象}」

判断原则：
  — 标记为「遗漏」的条目可能是合理的（如确认过不需要更新），审计不判定对错，
    只标记供人工确认。但同一变更中 ≥3 处遗漏 → 标记「系统性问题」
  — 此步骤不检查"变更是否应该发生"——只检查"变更发生后，引用方是否跟进了"
```

**适用范围**：所有 story 模块的 BL changelog + BL 文件 + business_related/depends_on 字段。

**与步骤7（G）的区别**：步骤7 检查"同一模块内的写入对是否同步"（BL 写入 ↔ pending_count 更新），本步骤检查"跨模块的、由业务引用关系决定的写入对是否同步"（模块 A 删除对象 ↔ 模块 B 更新引用）。步骤7 的原子对是规范强制定义的（固定的 A-B 对），本步骤的原子对是数据驱动的（谁引用了被删除的对象，谁就需要更新）。

**与步骤13（M）的区别**：步骤13 检查"changelog 声称改了 X，文件是否真的改了 X"（声明 vs 现实），本步骤检查"文件改了 X，引用 X 的其他文件是否也改了"（变更 vs 波及）。

**执行方式**：本步骤依赖文本搜索和人工判断，不要求全自动化。最小执行方式：取最近一次涉及删除/废弃的 BL changelog 条目，对其执行步骤 1-3。完整执行：对所有模块的最近 N 条此类条目执行。

### 步骤16：外部系统一致性（P）

**目标**：story 中的 TC 数据与外部系统（如云效 TestHub）同步后保持一致；同步过程遵守 `cloud-sync.md` 的方法规范。

本步骤含两个方向：数据一致性（方向 A）和方法合规性（方向 B）。

**方向 A：数据一致性（story ↔ 云效）**

```
1. 逐模块读取云效 directory list API，获取每个模块的 TC 数
2. 与 story index.json 中对应模块的 tc_count 比对
3. 对于每个模块：
   — count 不一致 → 标记具体差值
   — 读取云效 TC 列表，检查 type 字段缺失率（必须为 0）
   — 检查 tag 字段缺失率（必须为 0）
   — 检查新增 TC 的 tag 分布是否与预期版本号对齐（如 V2.1.2 版本 TC 应带 V2.1.2 标签）
4. 任一检查不通过 → 标记模块为「不一致」，列出具体差异条目
```

**方向 B：方法合规性（云效同步过程）**

```
1. cloud-sync.md 文件存在性检查：
   — os.path.exists({practices_path}/tech-stacks/functional/cloud-sync.md)
   — 不存在 → 标记「云效通道规范文件缺失」

2. 同步前差异分析检查：
   — 本次会话若执行了云效同步，是否在操作前输出了差异分析清单（新增/更新/删除明细）？
   — 未输出 → 标记「缺少同步前差异分析」

3. 全删重建检测：
   — 检测云效模块 TC 数是否在同一会话窗口内先降为 0 再恢复到目标值
   — 存在 → 标记「违规：全删重建」
   — 说明：云效不支持更新 API，全删重建是绕过限制的错误方式——应只对缺失 TC 做新增、对失效 TC 做删除

4. 批量删除 Dry-Run 检查：
   — 若执行了批量删除，删除前是否输出了 dry-run 清单并获得用户确认？
   — 未输出 → 标记「批量删除缺少 dry-run」

5. 同步后验证执行检查：
   — 同步完成后是否执行了逐模块 count / type 缺失 / tag 分布的校验？
   — 未执行 → 标记「缺少同步后验证」

6. 更新操作合规性：
   — 是否存在通过 API 执行 TC 更新操作（而非由用户在 UI 手动编辑）？
   — 检测方法：同模块内同 TC ID 的新建 TC 出现多次（间隔时间短）
   — 存在 → 标记「违规：使用 API 更新 TC（API 不支持更新）」
```

**适用范围**：story 全模块 index.json + 云效 TestHub（或任何外部 TC 管理系��）。方向 A 依赖云效 API 可访问性；方向 B 依赖会话行为日志。

**与其他步骤的关系**：本步骤是审计流水线中唯一检查外部系统（而非 practices/story 数据生态）的步骤。步骤 4（C）检查 story 内部计数字段自洽，本步骤检查 story 数据与外部系统的对应关系。步骤 1（F）检查 JSON 结构，本步骤检查 JSON 中的数值是否与外部系统对齐。

**与哲学 2（合规率优先于存在性）的关系**：方向 A 的 type/tag 检查直接贯彻哲学 2——不满足于「至少有一个正确」，而是统计全部实例的合规率。

### 步骤17：subagent 分层合规检查（Q）

**目标**：涉及 subagent 委托的 practices 和 capability 文件严格遵守分层——practices 只含规则和数据（"是什么"），capability 只含执行方法和调度决策（"怎么做"）。

```
1. 定位 subagent 相关内容：
   — 搜索 practices 文件中含「subagent」的章节
   — 搜索 capability SKILL.md 文件中含「subagent」的章节

2. practices 侧检查：
   — practices 中是否包含执行方法声明（如"使用单例模式""调用时不加延时"）
     → 是 → 标记「practices 越界：含执行方法」
   — practices 中是否包含项目特定数据（如具体的 spaceId、assignedTo、目录 ID）
     → 是 → 标记「practices 越界：含项目特定数据」（通用性约束）

3. capability 侧检查：
   — capability 中是否直接嵌入了 practices 规则原文（而非引用）
     → 是 → 标记「capability 越界：嵌入 practices 规则」
   — capability 中对 practices 的引用是否具体到章节名
     → 未指定具体章节 → 标记「引用粒度不足：应精确到### 标题」
   — capability 的触发条件是否覆盖 practices 中定义的全部硬约束类型
     → 遗漏 → 标记「触发条件不完整：遗漏 practices 中的{约束类型}」

4. 双向引用闭合：
   — practices 中声明的"本章节由 {capability} 引用"是否与实际引用它的 capability 匹配
   — capability 中引用的 practices 章节是否在 practices 文件中实际存在
   — 引用存在但章节标题不匹配 → 标记「引用断裂」
```

**适用范围**：`cloud-sync.md` ↔ `qa-functional-test/SKILL.md` 的 subagent 相关章节（当前阶段）；后续扩展至其他 capability 的 subagent 委托。

**与哲学19（通用框架不得下沉）的关系**：本步骤检查 practices 侧是否混入了项目特定数据——这是哲学19 在 subagent 维度的具体化。

**与哲学20（执行隔离可靠性）的关系**：本步骤是哲学20 的工程化落地——哲学20 定义了分层契约，本步骤提供系统方法验证分层是否被遵守。

## 二十条设计哲学

以下哲学来自 29 项审计问题的根因分析，是设计 practices 和 story 体系时的指导原则。

### 哲学1：双向验证原则

> 任何清单（注册表、Schema、引用关系）只做单向检查是不完整的。必须正向（定义 → 存在）和反向（存在 → 定义）双向闭合。

**来源**：第 1-4 轮审计漏掉 `handbook-practices-ops.md` 未注册——因为只查了"注册表里的文件都存在"，没查"磁盘上的文件都在注册表里"。

### 哲学2：合规率优先于"存在性"

> 验证一条规则时，找到"至少一个正确例子"不等于规则被遵守。必须统计所有应遵守实例的合规比例。

**来源**：changelog 格式问题直到第 7 轮才被发现——前 6 轮看了 NL2SQL（标准格式）就通过了，没统计 15/20 用旧格式。

### 哲学3：Schema 与数据异步漂移

> Schema 和实际数据各自独立演化。任何一方的变更不会自动同步到另一方。审计的本质工作就是重新对齐这两条独立的演进线。

**来源**：几乎所有问题都是"当前数据 vs 当前 Schema"的对齐检查——不是找谁错了，是找谁变了。

### 哲学4：可推导字段不得信任

> 任何能从源数据计算出来的存储字段（count、status boolean），都是缓存。缓存会过期。审计必须从源重新计算。

**来源**：`pending_count` 是典型的缓存——每次 pending → resolved 转换都应该更新，但总有遗漏。

### 哲学5：元数据是代码的投影

> `when_to_load`、字段所有权、文件注册表——这些元数据不是独立的文档，而是系统实际行为的投影。行为变了投影必须变。审计的任务是检查投影是否失真。**投影缺失同样是失真——当一个 capability 消费 practices 文件却不声明最低版本时，版本依赖就是隐式的。隐式依赖和错误的 when_to_load 是同一类问题：投影与行为不同步。**

**来源**：第 12 号问题——`when_to_load` 说"写入时加载"但 gateway 实际"每次都加载"，元数据与行为不同步。本次审计问题 7——capability 消费 practices 文件但无 `practices_min_version` 声明，版本依赖是隐式的。

### 哲学6：变更原子性

> 规范中用"缺一不可""必须同步""同时完成"描述的写入操作，在实际数据中必然存在断裂。原子性不是靠文档承诺保证的——是靠审计逐对验证的。

**来源**：handbook.md 中 6 处"缺一不可"约束，但 15 个 changelog 曾格式错误、2 个 pending_count 曾漂移。每一次都是原子性断裂。

### 哲学7：示例即测试用例

> practices 文档中的 `如 p001`、`例：TC-PIPE-001 → p001`、最小 JSON 片段——这些不只是解释，它们是精确的格式测试用例。如果真实数据中找不到一个匹配实例，那么规范和实现之间至少有一个是错的。

**来源**：3 次发现示例与实际数据不一致（`p1` vs `agt`、`p001` 首字母 vs 全前缀、`date` 字段不存在于真实 changelog）。每次都发现得晚，因为示例被当作"只是解释"而非"必须匹配的契约"。

### 哲学8：结构的脆弱点位于编辑边界

> JSON 结构损坏（问题 9）不是随机的——每次都出在"两个版本条目的交界处"。Edit 工具的 old_string 边界、手写 JSON 的 `},` 边界、嵌套对象的缩进边界——这些是结构最脆弱的位置。审计必须在这些边界上特别仔细。

**来源**：问题 9（v3/v2 合并处缺 `}`）、浏览器扩展问题 7（practices-bootstrap.md 删除 `last_updated` 后多一个 `}`）。两次都是"在两个结构体之间插入/删除内容"时破坏边界。

### 哲学9：会话边界即状态边界

> 每个 capability 执行是一个独立的写入会话。跨会话的状态一致性没有编译器保证——全靠人工维护的字段（pending_count、tc_count、changelog 格式）。两次写入之间的时间窗口越长，状态漂移的概率越高。审计本质上是在补偿跨会话状态管理的缺失——**每次审计发现的"数据不一致"都是某个过去会话写入后被另一个会话遗忘更新的证据**。
>
> **推论——状态通道即接口**：如果跨会话状态（context 标记、交接块、临时文件）是不可避免的通信手段，那么这些隐式通道需要和显式 API 一样的契约：谁写、谁读、什么格式、丢失后如何恢复。没有契约的 context 标记 = 没有签名的函数调用——能工作，但随时会断裂。步骤10（状态通道盘点）是这个推论的工程化落地。

**来源**：4 项问题（12/13/14/15）的共同模式——A 会话写入数据，B 会话修改了规范或状态，A 会话的数据变成过期状态，直到 C 会话（审计）才发现。不是"写错了"，是"写对了但后来变了"。本次审计问题 3+4——context 标记协议没有 schema，`assert_seq_runtime` 在 context 压缩后丢失导致 ID 冲突。

### 哲学10：清单的同质性

> 一个清单中的所有条目应该属于同一抽象层级和同一行为类别。混合用户能力、基础设施和未来计划在同一张"可用"列表中，会导致标记词语在不同条目上含义不同——对用户能力 "✅" 指"可被路由触发"，对基础设施 "✅" 指"hook 自动运行"，对计划条目 "✅" 永远不会出现但它的名字和标记在同一个表格里。审计时，不要相信标签——从行为推导类别，按类别验证同质性。

**来源**：本次审计问题 1——qa-token-report 的 "✅ 可用" 与 qa-understand 的 "✅ 可用" 共享同一标记但含义不同。问题 5——Phase 2 死路径也在同一列表中。步骤9（注册表同质性检查）是这个哲学的工程化落地。

### 哲学11：一个文档一个角色

> 当一个文件同时承担多个角色（用户指南 + Schema 权威 / 初始化模板 + 字段定义来源），任一角色的变更都会引入另一角色的风险。重写用户指南以改善可读性 = 可能破坏下游对 Schema 的引用。两个角色需要两份文档，各有一份职责。审计时，对每个文件问一个问题：如果把这个文件重写到只剩一个角色，下游谁会断裂？

**来源**：本次审计问题 6——practices-bootstrap.md 同时承担用户初始化指南和 story/index.json schema 权威两个角色。本次审计将其 schema 职责提取到独立的 `schemas/story-index.schema.md`。

### 哲学12：描述即缓存

> 数据字段不是唯一会过期的缓存——自然语言描述同样会。标题中的计数（「六步」「十条」）、注册表中的摘要字段（summary）、指向其他文档内容的文字描述——这些都是被写死在文本里的缓存。当源内容变更时，这些描述性缓存不会自动更新，过期概率与 `pending_count` 完全相同。审计必须像重新计算计数字段一样，重新验证描述性文本与源的对应关系。

**来源**：本次审计问题 2（handbook-audit.md 标题「六步审计流水线」vs 实际 11 步内容、标题「十条设计哲学」vs 实际 11 条）和问题 3（index.json 中 handbook-audit 的 summary 字段说「六步…和五条设计哲学」vs 实际 11+11）。两者都不是数据字段，是自然语言描述——但过期机制和 `pending_count` 漂移完全相同。步骤1 的版本号连续性检查（第6项）和步骤12 的「描述中嵌入的计数」特别标记是本哲学的工程化落地。

**与哲学4 的关系**：哲学4 覆盖了「可从源数据推导的存储字段」（数据层面的缓存），本哲学将其扩展到「可从源内容推导的自然语言描述」（文本层面的缓存）。同一个根因——缓存未随源自动更新——在两个不同介质中表现为不同的问题形态。

### 哲学13：副本即债务

> 当一个内容存在两份（权威源 + 参考副本），两份都会独立演化。权威源的每次变更都在增加副本的过期风险。副本不是便利——是待偿还的同步债务。审计时，对每个「描述同一事物」的文件对，假设副本已过期，直到证明它同步为止。**解决副本债务有两条路径：消除副本（将引用改为指向权威源），或建立显式同步契约（声明谁在权威源变更后负责更新副本，纳入步骤7 的原子变更检查范围）。**

**来源**：本次审计问题 1（`practices-bootstrap.md` 的 `depth` 类型 `number` 是 `schemas/story-index.schema.md` 中 `string` 的过期副本）和问题 7（bootstrap 版本示例 `"1.0.0"` 与 practices/index.json 实际 `"2026-06-12-v12"` 格式不一致）。两者的共同模式：权威源被更新了，副本留在原地无人同步。步骤12（权威-副本一致性检查）是本哲学的工程化落地。

**与哲学11 的关系**：哲学11（一个文档一个角色）解决同一文件承担多角色导致的冲突——处方是「拆分」。本哲学解决反向问题：拆分之后，两份文档描述同一事物（非角色冲突，是纯冗余），副本如何必然漂移——处方是「消除副本或建立同步契约」。哲学11 创造了权威-副本关系，哲学13 管理这种关系的风险。

### 哲学14：Changelog 是承诺，不是收据

> Changelog 条目中声称的变更与文件实际状态之间的鸿沟，是比数据漂移更隐蔽的错误——数据漂移至少有两个来源可以交叉比对，而「声称已修但未修」只有一个来源在说谎。审计时，假设每个 changelog 条目中带具体数字/名称/路径的声明都是待验证的承诺，直到文件内容证明它已被兑现。一个未兑现的 changelog 声明比没有 changelog 更危险——它制造了问题已被修复的假象，且这个假象可能持续多个版本周期直到被步骤13 捕获。

**来源**：本次审计问题 1——v13 changelog 声称「审计流水线标题十一步→十二步」，这个声明在 changelog 里存在了整整一个版本周期而无人发现它未被执行。根因：写 changelog 时预设了修复已完成（或写了 changelog 后忘记改对应文件），且不再有验证步骤确认。

**与哲学12（描述即缓存）的关系**：哲学12 说「描述是缓存在文本里的快照，源变了描述不会自动更新」——这是描述滞后于源。本哲学是反向的——「changelog 里的行为声明指向了一个可能从未被写入的目标状态」——这是声明超前于源。两种错误对称：一个描述过去但过去已变，一个描述未来但未来未到。步骤13（Changelog 声明真实性验证）是本哲学的工程化落地。

### 哲学15：契约的可执行性与声明同样重要

> 一个被声明但永远不会被校验的版本契约（如 `practices_min_version`），本质上是一条注释——它出现在结构化字段里，满足机器的解析要求，但永远不会触发任何阻断或告警，因为校验它的代码路径不覆盖这个组件。审计时，对每个声明了最低版本或契约约束的消费者，必须同时验证「在系统运行时的某个代码路径中，这个声明会被读取并与生产者的实际版本进行比较」。声明范围 ≠ 强制执行范围 = 契约不完整。声明是契约的文字，强制执行路径是契约的签名——两者缺一不可。

**来源**：本次审计问题 4——qa-token-report 在 SKILL.md 中声明了 `practices_min_version: "2026-06-12-v11"`，所有格式都正确。但 gateway 的环境验证阶段只校验「被路由表触发的 capability」，由 Hook 触发的 infrastructure capability 永远不会被纳入版本比较。声明存在，但声明永远不被读取——等于没有声明。修复方案是让 gateway 遍历「所有已注册 capability」而非「被触发的 capability」。

**与哲学5（元数据是代码的投影）的关系**：哲学5 说「`when_to_load` 描述的行为必须与代码实际加载行为一致」。本哲学是该原则在版本契约维度的具体化——不仅元数据（如 when_to_load）要准确投影代码行为，版本契约（如 practices_min_version）的 enforceability 也必须可追溯到一段会读取它的代码。哲学5 的"投影失真" + 本哲学的"校验路径缺失" = 同一类问题在不同维度的表现。步骤11（K）的「强制执行可达性检查」是本哲学的工程化落地。

### 哲学16：检验左移

> 质量检查的时机越接近信息产生点，修复成本越低，错漏半径越小。写入前从待写入内容直接统计计数字段，比写入后用 grep 发现偏差再 Edit 修复，少了一次完整文件的上下文往返。修复半径从"多轮对话后"压缩到"单次 Write 调用内"。

**来源**：本次执行 P9/P10/P11——写入完才发现计数不一致，被迫多轮 Edit。根因：计数是在事后校验的，不是在生产时确定的。

**与哲学4 的关系**：哲学4 说"可推导字段不得信任——从源重新计算"。检验左移是哲学4 的工程优化——不是"从源重新计算"，而是"在源还在手上的时候就计算好，带着结果一起写入"。处方的区别是被动重算 vs 主动同步写入。

### 哲学17：最小安全基线

> 自动化系统的默认配置必须覆盖该领域最常见、影响最大的失效模式。1-hop 上下游不是"更好的理解"——是"基本的安全网"。在 token 预算和上下文膨胀的成本约束下，默认读取范围不能低于能发现已注册上下游模块中的波及断裂。最小基线一旦确定，不可为了节省 token 而降低——因为降低基线的省 token vs 遗漏波及的修复成本不对称（1 次遗漏 ≈ 多轮对话重建上下文）。

**来源**：本次执行中，遗漏入口的波及影响和删除角色的连锁影响——共同根因：默认没读上下游模块。如果有 1-hop 安全基线，两个遗漏在波及分析阶段就会被发现。

**与哲学1 的关系**：哲学1（双向验证）要求注册表↔磁盘双向闭合。最小安全基线是同一原则在"读取范围"维度的应用——不能只看目标模块，必须同时看引用它的模块。哲学1 保证了"清单的完整"，本哲学保证了"理解的完整"。

### 哲学18：纵挖 + 横切

> 单一维度的深入分析（纵向：每个功能点追问 7 个问题）产生完整性错觉。完整的理解需要一次独立的横向扫描——不是对每个功能点再多问一个问题，而是对整体产出问一个不同方向的问题：谁被波及了？哪里还没被覆盖？纵向分析保证每个功能点被充分测试，横向扫描保证没有功能点被遗漏。

**来源**：本次 P4 和 P5——7 问全部充分执行，但所有功能点分析完成后，没有机制检查"是不是还有入口没覆盖"或"被删除的角色有哪些已有断言仍在引用它"。

**与哲学2 的关系**：哲学2（合规率优先于存在性）检查"所有实例是否符合规则"（单维度全量），本哲学检查"是否存在未被第一维度覆盖的第二维度"（多维度）。哲学2 是"做对每一道题"，本哲学是"确认没有题被漏掉"。

### 哲学19：通用框架不得下沉

> 通用测试框架的内容一旦包含具体项目名、产品名或业务术语，就失去了通用性——下一次用于不同项目时，这些硬编码的示例会成为误导或噪音。框架文件的作者有责任在每一次提交示例、描述、代码片段时自问：「这个例子换一个项目还能用吗？」不能 → 用抽象占位符替代。审计通过扫描全部 capability 和 practices 文件来验证这一点——不是查"写得好不好"，是查"有没有不该出现的东西"。

**来源**：本次执行发现 capability 适配器文件的示例中出现了具体项目名和入口名，practices 格式规范文件的路径示例中出现了具体项目目录名。这些是在迭代过程中自然沉淀的——框架作者脑中装着当前项目，写示例时顺手用了当前项目的名字。需要在规范层面明确禁止，并在审计流程中加入检查。

**与哲学11 的关系**：哲学11（一个文档一个角色）说"每个文件只承担一种职责"。本哲学是该原则的跨角色维度——通用框架文件和项目 story 文件承担不同的角色。将项目内容写入通用文件 = 一个文件承担了两种角色（通用指南 + 项目记录），违反了角色单一原则。

### 哲学20：执行隔离可靠性

> 长对话 context 中，自然语言规则对模型的约束力存在衰减——距离越远、context 越拥挤，规则越容易被跳过。将硬约束规则从 conversation context 迁移到 subagent 的 prompt（system prompt 级别优先级）中，是将"软建议"升级为"硬约束"的机制。但分层必须精确：practices 定义被隔离的规则（"隔离什么"），capability 定义隔离的时机和方法（"何时隔离、如何隔离"）。审计时，对每个 subagent 委托的 practices-capability 对，检查三层闭合：practices 声明的硬约束 ⊆ subagent prompt 中复制的规则 ⊆ capability 的触发条件覆盖了需要隔离的场景。任一 ⊆ 断裂 → 隔离失效。**并发隔离的前提是状态无竞争——共享状态（如 index.json）必须在隔离区外（主线）统一更新。**

**来源**：本窗口 qalore 迭代——74 条 TC 同步中前缀规则被 2 次忽略、批量删除前 dry-run 被跳过、同步后验证不完整。根因：practices 在长 context 中被模型遗忘。subagent 方案将规则从 conversation 注入 system prompt，同时定义了精确的分层契约。

**与哲学6（变更原子性）的关系**：哲学6 说"缺一不可的写入对在数据中必然存在断裂"。本哲学将该原则扩展到"subagent 的 prompt 构造"——practices 硬约束 + API 模板 + 操作列表 = 三个必须全部包含的段，和原子写入对一样，缺一段即断裂。步骤17（subagent 分层合规检查）的"三层闭合"检查是本哲学的工程化落地。

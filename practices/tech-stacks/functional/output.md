# 产物输出规范

---

## 是否输出 .mm

| 用户描述信号 | 输出策略 |
|------------|---------|
| 含「输出用例 / 生成用例 / 出用例 / 测试文件 / 全量」 | 输出 .mm |
| 含「沉淀 / 存档 / 记录 / 只更新 / 只提炼」 | 不输出 .mm，仍更新 TC 文件和 changelog |
| 无明确信号 | 询问用户后决定 |

## 输出范围

| 输入信号 | 默认输出范围 |
|---------|------------|
| 用户描述含「全量」 | 全量（所有模块、所有版本累积用例） |
| 完整版本 PRD + 含「输出用例」 | 版本级（当前版本跨模块用例） |
| 单 feature 文档 / 口述单个功能 | Feature 级 |
| 口述多个 feature 跨模块 | 版本级 |
| 无需求文档 + 指定模块 | 模块级 |
| 无法匹配 | 询问用户：「全量 / 版本级 / 模块级 / Feature 级」 |

用户在功能层级上额外限定时，取更窄范围。产出文件数量由确认的输出范围决定，与处理了几个模块无关。多模块任务时，所有模块完成设计和 story 写入后，统一执行一次产物输出，中间不产出任何 .mm 文件。

## 默认产物格式

功能测试用例产物默认生成 FreeMind `.mm` 格式文件，可直接导入 XMind / FreeMind / MindManager。

用户明确指定格式时（如「给我 md 格式」），按用户要求执行，不受此默认值约束。

文件头固定格式：

```xml
<map version="1.0.1">
<!-- FreeMind 格式，可直接导入 XMind / FreeMind / MindManager -->
<!-- 生成日期：{YYYY-MM-DD} | 输出范围：{scope} | 用例总计：{n} 条 -->
<node ID="root" TEXT="{项目名}">
```

## 脑图结构规则

**根节点固定为项目名称，层级从模块向功能点路径逐层展开，TC 节点展开至步骤与预期：**

```
{项目名}（根节点，ID="root"）
└── {模块名}（ID="{模块缩写}"，POSITION="right"，STYLE="bubble"）
    └── {功能点N级}（ID="{模块缩写}_fp{n:03d}"，STYLE="fork"）
        └── TC：{[仅需求]/[仅代码]}{用例名称}（ID="{tc_id}"，STYLE="fork"）
            ├── 优先级：{P0/P1/P2}（ID="{tc_id}_p"，STYLE="fork"）
            ├── 版本：{版本号}（ID="{tc_id}_v"，STYLE="fork"，无版本时显示"版本："，字段必须存在）
            ├── 前提：{前置条件}（ID="{tc_id}_pre"，STYLE="fork"）
            └── 操作步骤：（ID="{tc_id}_steps"，STYLE="fork"）
                ├── 1. {步骤1描述}（ID="{tc_id}_s1"，STYLE="fork"）
                │   └── 预期：{预期内容}（ID="{tc_id}_s1_e"，STYLE="fork"，触发性步骤才有此节点）
                ├── 2. {纯输入步骤描述}（ID="{tc_id}_s2"，STYLE="fork"，无预期子节点）
                └── 3. {步骤3描述}（ID="{tc_id}_s3"，STYLE="fork"）
                    └── 预期：{预期内容}（ID="{tc_id}_s3_e"，STYLE="fork"）
```

层级与 ID 映射：

| 层级 | 内容 | ID 规则 | STYLE |
|------|------|---------|-------|
| 根节点 | 项目名称 | `root` | — |
| 第二级 | 模块名 | `{模块缩写}` 如 `pipe`、`img` | `bubble` |
| 中间层 | `功能点` 字段按 `>` 拆分的各级节点 | `{模块缩写}_fp{n:03d}`，模块内递增 | `fork` |
| TC 节点 | `TC：{[仅需求]/[仅代码]}{标题}` | 取 TC ID 数字部分 + 模块 mm_short_id 前缀，如 TC-PIPE-001 → `p001`；mm_short_id 来自 story/index.json，新模块首次生成时由 qa-functional-test 写入 | `fork` |
| TC 子节点 | 优先级 / 版本 / 前提 / 操作步骤： | `{tc_id}_p` / `{tc_id}_v` / `{tc_id}_pre` / `{tc_id}_steps` | `fork` |
| 步骤节点 | `{n}. {步骤描述}` | `{tc_id}_s{n}` | `fork` |
| 预期节点 | `预期：{内容}`（触发性步骤的子节点，非每步都有）| `{tc_id}_s{n}_e` | `fork` |

**节点书写规则：**

1. 所有 `<node>` 必须带 `ID` 属性，全局唯一
2. 根节点不设 `STYLE`；模块节点（第二级）必须带 `POSITION="right"` 和 `STYLE="bubble"`；第3级起所有子节点统一使用 `STYLE="fork"`
3. 子节点必须独立一行，**禁止 inline**：

```xml
<!-- 错误 -->
<node ID="p001_s2" TEXT="2. 步骤"><node ID="p001_e1" TEXT="预期：xxx"/></node>

<!-- 正确 -->
<node ID="p001_s2" TEXT="2. 步骤">
  <node ID="p001_e1" TEXT="预期：xxx"/>
</node>
```

4. TEXT 属性值中以下字符必须转义：

| 字符 | 转义为 |
|------|--------|
| `&` | `&amp;` |
| `<` `>` | `&lt;` `&gt;` |
| `"` / `"` / `"` | `&quot;` |

**叶节点挂载原则：TC 节点只挂到功能点路径的最末节点，中间节点不挂用例。**

## TC 来源标记

TC 节点 TEXT 中，`TC：` 后可标注用例来源，帮助阅读者判断置信度：

| 标记 | 含义 | 对应 story TC 信号 |
|------|------|------------------|
| `[仅需求]` | 用例仅基于需求文档，代码尚未确认 | TC 备注含"实现待确认" |
| `[仅代码]` | 用例仅基于代码分析，PRD 无对应描述 | TC 位于 `## 代码独立` section |
| 无标记 | 需求+代码双源交叉验证，置信度最高 | 既非代码独立 section，也无实现待确认备注 |

标记紧接 `TC：` 之后、用例标题之前：`TC：[仅需求]用例标题` / `TC：[仅代码]用例标题`。

## 输出文件命名规则

产物输出范围分为四个层级（**版本维度**与**功能层级**正交）：

| 范围层级 | 含义 | 触发场景示例 |
|---------|------|------------|
| **全量** | 全项目所有模块所有版本累积用例 | "全量用例" |
| **版本级** | 当前版本所有新增/更新用例（跨模块） | 阅读某版本完整 PRD + "输出用例" |
| **模块级** | 指定模块全部累积用例（跨版本） | "图生图模块的用例" |
| **Feature 级** | 单个 feature 的新增/更新用例 | 阅读单 feature 文档 / 口述某功能更新 |

**文件命名对照：**

| 输出范围 | 文件名格式 |
|---------|-----------|
| 全量 | `{项目名}-全量功能测试用例.mm` |
| 版本级 | `{项目名}-{版本号}-功能测试用例.mm` |
| 模块级 | `{项目名}-{模块名}-功能测试用例.mm` |
| Feature 级 | `{项目名}-{模块名}-{feature名}-功能测试用例.mm` |
| 单功能点（原有） | `{项目名}-{模块名}-{功能点名}-功能测试用例.mm` |

多级功能点路径取最末级节点名称，例：`新增管线 > 预览图上传 > 格式校验` → 文件名取 `格式校验`。

## .mm 生成后验证

.mm 文件生成后必须执行以下检查，不通过则重新生成：

| 检查项 | 方法 | 阈值 |
|--------|------|------|
| XML 合法性 | `ET.parse()` 解析不抛异常 | 必须通过 |
| TC 节点数 | 统计 `<node TEXT="TC：` 节点数 | 等于 `sum(index.json tc_count)` |
| 零步骤 TC | 检查每个 TC 节点的 `_steps` 子节点 | 必须为 0 |
| 步骤覆盖率 | 统计 `_s{n}` 步骤节点总数 | ≥ TC 数 × 1.5 |
| 预期覆盖率 | 统计 `_s{n}_e` 预期子节点数 | ≥ 步骤节点的 80% |
| 模块节点数 | 统计根节点下一级 `<node>` 数 | 等于 modules 数 |

不通过时的处理：
- 零步骤 TC → 检查生成脚本的 `parse_tc()` 循环逻辑（最常见根因：循环结束后未追加 last step）
- 步骤覆盖率不足 → 可能是步骤解析正则未匹配到步骤行
- XML 不合法 → 检查 TEXT 属性中的 `&` `<` `>` `"` 是否正确转义

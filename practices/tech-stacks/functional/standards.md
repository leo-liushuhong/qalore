# 功能测试规范

## 用例格式（完整字段定义）

每条测试用例必须包含以下所有字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 用例 ID | 是 | 格式：`TC-{需求ID}-{序号}`，全局唯一 |
| 标题 | 是 | 简明描述，不超过 30 字 |
| 功能点 | 是 | 路径表达式，格式见「功能点路径规范」 |
| 版本 | 否 | 对齐需求版本号；为空表示未关联版本；规则见「版本标记规则」 |
| 前置条件 | 是 | 执行前必须满足的系统状态或数据状态 |
| 测试步骤 | 是 | 编号步骤，每步一个操作，明确具体 |
| 预期结果 | 是 | 明确可验证，禁止使用「正常」等模糊词 |
| 优先级 | 是 | P0 / P1 / P2 / P3 |
| 测试维度 | 是 | 正向 / 边界 / 异常 / 上下游 |

**用例示例：**

```
用例 ID：TC-AUTH-001
标题：用户使用正确账号密码登录成功
功能点：登录 > 账号密码登录
版本：v1.9
前置条件：用户账号已注册且状态正常
测试步骤：
  1. 打开登录页面
  2. 输入正确的用户名
  3. 输入正确的密码
  4. 点击登录按钮
预期结果：跳转至首页，顶部显示用户昵称
优先级：P0
测试维度：正向
```

### 版本标记规则

| 情况 | 处理方式 |
|------|---------|
| 需求文档、对话、代码中明确写了版本号 | 使用该版本号 |
| 需求文档未写版本号 | 保留用例已有版本号；若为空则保持空 |
| 更新已有用例，对话 / 文档 / 代码均无明确版本标记 | 保留用例已有版本号；若为空则保持空 |

## 功能点路径规范

### 格式

```
功能点：{一级功能点} > {二级功能点} > {三级功能点} > ...
```

- 使用 `>` 作为层级分隔符
- 层级数量不限，按需扩展
- 每级名称使用业务语言，不超过 10 字

### 叶节点挂载原则

**用例只挂载到路径的最末节点，中间节点不挂用例。**

错误示例（同一功能点既挂用例又有子节点）：
```
功能点：新增管线            ← 有用例 TC-A
功能点：新增管线 > 名称校验  ← 也有用例 TC-B
```

正确示例：
```
功能点：新增管线 > 必填校验  ← 所有必填类用例统一挂此节点
功能点：新增管线 > 名称校验  ← 名称规则类用例挂此节点
```

### 脑图映射规则

路径中每个 `>` 对应脑图的一个层级，最末节点下列出该路径的所有用例标题。

示例映射：
```
管线管理（模块根节点）
└── 新增管线
    └── 预览图上传
        ├── 大小校验
        │   ├── TC-005 上传恰好 5MB 的预览图成功
        │   └── TC-006 上传超过 5MB 的预览图被拒绝
        └── 格式校验
            └── TC-007 上传非 PNG/JPG 格式预览图被拒绝
```

### 路径扩展时机

满足以下任一条件时，考虑在当前层级下新增子节点：
- 同一功能点下用例超过 5 条，且存在明显的子分类
- 需求细化带来新的子功能，原路径粒度不足以区分

## 优先级定义

| 级别 | 定义 | 典型场景 |
|------|------|---------|
| P0 | 核心业务流程不可用 / 数据损坏风险 / 上下游阻断 | 登录失败、支付无法完成、数据写入异常 |
| P1 | 功能降级但主流程可用 / 边界异常 / 非核心功能故障 | 错误提示文案不对、边界值处理错误、非主路径异常 |
| P2 | 体验问题 / 极低频场景 / 文案错误 | 页面加载偏慢、按钮文案错别字、极端输入无响应 |
| P3 | 极低优先级 / 锦上添花 / 不影响使用的优化点 | 样式微调、快捷键、极端输入下的友好提示改进 |

**PRD 优先级优先原则：** 需求文档对业务场景已标注优先级时，用例优先级**强制对齐 PRD 标注**，不由 testcraft 自行推断。仅当 PRD 未标注时，才按上表规则判断。

## 测试维度

| 维度 | 说明 | 设计要点 |
|------|------|---------|
| 正向路径 | 标准流程，输入合法，预期成功 | 覆盖主干业务流程的完整执行链路 |
| 边界条件 | 输入边界值、极值、空值、最大长度 | 关注数值边界（最大/最小/±1）和字符串长度 |
| 异常场景 | 错误输入、权限不足、网络超时、并发冲突 | 覆盖错误处理路径和异常恢复机制 |
| 上下游影响 | 此功能变化对依赖它的其他模块的影响 | 梳理模块间依赖，关注接口契约变化 |

## 跨模块 TC 归属规则

上下游影响类 TC 挂载在**行为可观测的模块**（通常是下游/受影响方）：

| 场景 | TC 归属 | 示例 |
|------|---------|------|
| A 模块操作 → B 模块展示/数据发生变化 | 写在 B 模块 | 管线版本删除 → 图生图再来一张提醒 → TC 写在图生图 |
| A 模块权限变更 → B 模块访问受限 | 写在 A 模块（权限控制方）| 权限管理 → 管线管理不可访问 → TC 写在权限管理 |

同时在上游模块业务逻辑文件「上下游依赖」节，用 `→ 见 TC-XXX-NNN` 标注关联用例 ID。

## 命名规则

### 用例 ID

格式：`TC-{需求ID}-{序号}`

| 字段 | 规则 |
|------|------|
| 需求 ID | 来自项目需求管理系统；无系统时用模块英文缩写（见下方「TC 前缀注册规则」） |
| 序号 | 3 位数字，从 001 开始，同需求 ID 内唯一递增 |

示例：`TC-AUTH-001`、`TC-PAY-003`、`TC-ORDER-012`

#### TC 前缀注册规则

新模块首次建立时，按以下步骤确定并注册 TC 前缀：

1. 取模块英文意译的首字母缩写（2-5 个字母，全大写）
2. 确认与同项目已有前缀不冲突
3. 写入 `{story_path}/{项目名}/index.json` 该模块的 `tc_prefix` 字段
4. **注册后不可变更**（变更会导致历史 ID 断链）

| 已注册示例 | 模块名 | TC 前缀 | .mm 短 ID |
|---------|-------|---------|----------|
| AIGC平台 | 管线管理 | PIPE | `p{序号}` |
| AIGC平台 | 图生图 | IMG | `i{序号}` |
| AIGC平台 | AI创作-工具 | TOOL | `t{序号}` |
| AIGC平台 | 资源库-我的资产 | ASSET | `a{序号}` |

`.mm` 短 ID 规则：取 TC 前缀首字母小写 + 序号数字，例：`TC-PIPE-001` → `p001`。

### 文件命名

| 文件 | 命名格式 |
|------|---------|
| 测试用例 | `{模块名}-功能-测试用例.md` |
| 变更日志 | `{模块名}-功能-测试用例.changelog.md` |

## 质量底线

### 覆盖率规则

覆盖来源分两个层级，优先级不同：

**第一层级：PRD 业务场景（必须全量覆盖）**
产品文档中明确描述的每一个业务场景，必须输出对应的测试用例，不得遗漏。

**第二层级：测试方法衍生场景（默认最简）**
边界值、异常路径、上下游影响等由测试方法推导出的场景，默认每类只输出最典型的 1 条。用户明确要求「尽量覆盖」时，扩展至全量输出。

| 来源 | 默认输出策略 | 用户要求全覆盖时 |
|------|------------|----------------|
| PRD 明确描述的业务场景 | 全量输出（强制） | — |
| 测试方法衍生场景 | 每类最多 1 条最典型 | 尽量全量输出 |

**第三层级：上下游依赖覆盖（强制）**

若模块业务逻辑文件的「上下游依赖」节列出了依赖方向，测试用例设计必须针对每个列出的依赖方向至少输出 1 条上下游维度用例，验证该模块的配置或行为在依赖方向上生效。

| 触发条件 | 覆盖要求 |
|---------|---------|
| 业务逻辑文件「上下游依赖」节为空或不存在 | 不强制 |
| 业务逻辑文件「上下游依赖」节列出了依赖方向 | 每个依赖方向至少 1 条上下游用例（强制） |

#### 判断说明

**PRD 明确描述** = 需求文档中有明确文字描述的功能行为、交互逻辑、字段规则或系统响应，包括功能列表项、场景说明、校验规则等。

**测试方法衍生场景** = 测试工程师根据测试方法论自行推导出的场景，需求文档中无对应的文字描述。

| 示例场景 | 分类 | 依据 |
|---------|------|------|
| "版本名称最多 50 个字符" | PRD 明确描述 | PRD 写了字段规则 |
| 输入 51 字符被拒绝 | PRD 明确描述 | 上述规则的直接对应行为 |
| 输入特殊字符的行为 | 衍生场景 | PRD 未描述，测试方法推导 |
| 上传后网络超时的处理 | 衍生场景 | PRD 未描述，异常场景推导 |

### 用例质量标准

- 预期结果必须**明确可验证**，禁止使用「正常显示」「符合预期」等模糊表述
- 前置条件必须完整，确保用例可独立执行
- 测试步骤每步只描述一个操作，逐步可追踪
- 用例 ID 全局唯一，格式遵循本文件「命名规则」章节

## story 写入规则

### changelog 格式

```markdown
## v{n}（{YYYY-MM-DD}）
- 新增：{用例ID} {标题}（原因：{需求新增描述}）
- 更新：{用例ID} {变更字段及内容}（原因：{需求变更描述}）
- 删除：{用例ID} {标题}（原因：{失效原因，说明哪条需求导致该用例不再成立}）
```

### changelog 写入顺序

**新版本条目插入到文件标题行之后、所有旧版本之前（倒序），最新变更永远在最前面。**

```markdown
# {模块名} 功能测试用例 变更日志   ← 标题行，固定在最上方

## v{最新}（{YYYY-MM-DD}）        ← 新条目插在这里
- ...

## v{次新}（{YYYY-MM-DD}）
- ...

## v1（{YYYY-MM-DD}）             ← 最旧版本沉底
- ...
```

每次写入必须同步执行以下两步，缺一不可：

| 步骤 | 操作 |
|------|------|
| 1 | 在文件标题行之后**头部插入**新 `## v{n}（日期）` 块，含所有变更条目 |
| 2 | 保持旧版本条目**原样不动**，不得删除或重排已有内容 |

## 产物输出规范

### 默认产物格式

功能测试用例产物默认生成 FreeMind `.mm` 格式文件，可直接导入 XMind / FreeMind / MindManager。

用户明确指定格式时（如「给我 md 格式」），按用户要求执行，不受此默认值约束。

文件头固定格式：

```xml
<map version="1.0.1">
<!-- FreeMind 格式，可直接导入 XMind / FreeMind / MindManager -->
<!-- 生成日期：{YYYY-MM-DD} | 输出范围：{scope} | 用例总计：{n} 条 -->
<node ID="root" TEXT="{项目名}">
```

### 脑图结构规则

**根节点固定为项目名称，层级从模块向功能点路径逐层展开，TC 节点展开至步骤与预期：**

```
{项目名}（根节点，ID="root"）
└── {模块名}（ID="{模块缩写}"，POSITION="right"）
    └── {功能点N级}（ID="{模块缩写}_fp{n:03d}"）
        └── TC：{用例名称}（ID="{tc_id}"）
            ├── 优先级：{P0/P1/P2}（ID="{tc_id}_p"）
            ├── 版本：{版本号}（ID="{tc_id}_v"，无版本时显示"版本："，字段必须存在）
            ├── 前提：{前置条件}（ID="{tc_id}_pre"）
            └── 操作步骤（ID="{tc_id}_steps"）
                ├── 1. {步骤1描述}（ID="{tc_id}_s1"）
                └── 2. {步骤2描述}（ID="{tc_id}_s2"）
                    └── 预期：{预期结果}（ID="{tc_id}_e1"，挂在最后一步下）
```

层级与 ID 映射：

| 层级 | 内容 | ID 规则 |
|------|------|---------|
| 根节点 | 项目名称 | `root` |
| 第二级 | 模块名 | `{模块缩写}` 如 `pipe`、`img` |
| 中间层 | `功能点` 字段按 `>` 拆分的各级节点 | `{模块缩写}_fp{n:03d}`，全局递增 |
| TC 节点 | `TC：{标题}` | `p{序号}` / `i{序号}` / `t{序号}` / `a{序号}`（取 TC ID 数字部分，前缀按模块：PIPE→p、IMG→i、TOOL→t、ASSET→a） |
| TC 子节点 | 优先级 / 版本 / 前提 / 步骤容器 | `{tc_id}_p` / `{tc_id}_v` / `{tc_id}_pre` / `{tc_id}_steps` |
| 步骤节点 | `{n}. {步骤描述}` | `{tc_id}_s{n}` |
| 预期节点 | `预期：{内容}` | `{tc_id}_e1` |

**节点书写规则：**

1. 所有 `<node>` 必须带 `ID` 属性，全局唯一
2. 模块节点（第二级）必须带 `POSITION="right"`
3. 子节点必须独立一行，**禁止 inline**：

```xml
<!-- 错误 -->
<node ID="p001_s2" TEXT="2. 步骤"><node ID="p001_e1" TEXT="预期：xxx"/></node>

<!-- 正确 -->
<node ID="p001_s2" TEXT="2. 步骤">
  <node ID="p001_e1" TEXT="预期：xxx"/>
</node>
```

4. TEXT 属性值中以下字符必须转义：

| 字符 | 转义为 |
|------|--------|
| `&` | `&amp;` |
| `<` `>` | `&lt;` `&gt;` |
| `"` / `"` / `"` | `&quot;` |

**叶节点挂载原则：TC 节点只挂到功能点路径的最末节点，中间节点不挂用例。**

> 注意：当前 story 格式的预期结果为合并输出（非逐步），生成脑图时挂载在最后一个步骤节点下。

### 输出文件命名规则

产物输出范围分为四个层级（**版本维度**与**功能层级**正交）：

| 范围层级 | 含义 | 触发场景示例 |
|---------|------|------------|
| **全量** | 全项目所有模块所有版本累积用例 | "全量用例" |
| **版本级** | 当前版本所有新增/更新用例（跨模块） | 阅读某版本完整 PRD + "输出用例" |
| **模块级** | 指定模块全部累积用例（跨版本） | "图生图模块的用例" |
| **Feature 级** | 单个 feature 的新增/更新用例 | 阅读单 feature 文档 / 口述某功能更新 |

**文件命名对照：**

| 输出范围 | 文件名格式 |
|---------|-----------|
| 全量 | `{项目名}-全量功能测试用例.mm` |
| 版本级 | `{项目名}-{版本号}-功能测试用例.mm` |
| 模块级 | `{项目名}-{模块名}-功能测试用例.mm` |
| Feature 级 | `{项目名}-{模块名}-{feature名}-功能测试用例.mm` |
| 单功能点（原有） | `{项目名}-{模块名}-{功能点名}-功能测试用例.mm` |

多级功能点路径取最末级节点名称，例：`新增管线 > 预览图上传 > 格式校验` → 文件名取 `格式校验`。

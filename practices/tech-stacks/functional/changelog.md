# 变更记录规范

---

## 断言 changelog 格式

适用于 `{模块名}-功能-业务逻辑.changelog.md` 和 `{模块名}-功能-代码逻辑.changelog.md`。

```markdown
# {模块名} 功能{业务逻辑/代码逻辑} 变更日志   ← 标题行，文件创建时写入，固定在最上方

## {YYYY-MM-DD}（{来源描述，如：PRD v2.0 / 代码阅读 / 人工修正}）
- 功能点/组件「{name}」新增：assert-{PREFIX}-{seq1}～{seq2}（{n} 条，{一句话描述覆盖内容}）
- 功能点/组件「{name}」更新：assert-{PREFIX}-{seq}（{具体变化，如：大小限制从 10MB 改为 5MB}）
- 功能点/组件「{name}」删除：assert-{PREFIX}-{seq1}、{seq2}（原因：{功能点被移除/逻辑被替代}）
```

**触发规则：内容文件（业务逻辑.md / 代码逻辑.md）任何改动都必须写 changelog，无论是否发版。**

**写入顺序：新条目插入到文件标题行之后、所有旧条目之前（倒序），最新变更永远在最前面。**

---

## 用例 changelog 格式

```markdown
## v{n}（{YYYY-MM-DD}）
- 新增：{用例ID} {标题}（原因：{需求新增描述}）
- 更新：{用例ID} {变更字段及内容}（原因：{需求变更描述}）
- 删除：{用例ID} {标题}（原因：{失效原因，说明哪条需求导致该用例不再成立}）
```

---

## changelog 写入顺序

**新版本条目插入到文件标题行之后、所有旧版本之前（倒序），最新变更永远在最前面。**

```markdown
# {模块名} 功能测试用例 变更日志   ← 标题行，固定在最上方

## v{最新}（{YYYY-MM-DD}）        ← 新条目插在这里
- ...

## v{次新}（{YYYY-MM-DD}）
- ...

## v1（{YYYY-MM-DD}）             ← 最旧版本沉底
- ...
```

每次写入必须同步执行以下两步，缺一不可：

| 步骤 | 操作 |
|------|------|
| 1 | 在文件标题行之后**头部插入**新 `## v{n}（日期）` 块，含所有变更条目 |
| 2 | 保持旧版本条目**原样不动**，不得删除或重排已有内容 |

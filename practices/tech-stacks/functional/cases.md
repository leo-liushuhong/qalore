# 功能测试用例规范

---

## 用例格式（完整字段定义）

每条测试用例必须包含以下所有字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 用例 ID | 是 | 格式：`TC-{PREFIX}-{序号}`，全局唯一；PREFIX 为模块英文缩写，见「命名规则」章节 |
| 标题 | 是 | 简明描述，不超过 30 字 |
| 功能点 | 是 | 路径表达式，格式见「功能点路径规范」 |
| 版本 | 否 | 对齐需求版本号；为空表示未关联版本；规则见「版本标记规范」 |
| 前置条件 | 是 | 执行前必须满足的系统状态或数据状态 |
| 测试步骤 | 是 | 编号步骤，每步一个操作；触发系统可观察响应的步骤必须紧跟「→ 预期：{内容}」；纯输入步骤可省略预期 |
| 优先级 | 是 | P0 / P1 / P2 / P3 |
| 测试维度 | 是 | 正向 / 边界 / 异常 / 上下游 |

**触发性步骤判断标准（以下类型必须写内联预期，不得省略）：**

| 动作类型 | 典型示例 | 应写的预期内容 |
|---------|---------|-------------|
| 点击按钮/图标打开 UI | 点击"编辑"、点击"更多图标" | 抽屉/弹窗/菜单弹出，内容描述 |
| 点击按钮关闭 UI | 点击"确认"、点击"取消" | 弹窗关闭，数据变化描述 |
| 切换开关/选项 | 开启随机开关、切换项目 | 开关/控件状态变化 |
| 文件选择/上传完成 | 选择图片文件、上传管线文件 | 上传成功/失败，预览区变化 |
| 触发数据刷新 | 等待搜索结果、选择版本 | 列表/区域内容刷新描述 |
| 鼠标悬浮/交互反馈 | 悬浮卡片、悬浮按钮 | 视觉状态变化描述 |

**纯输入步骤（以下类型可省略预期）：**
- 在文本框输入内容（无即时校验触发时）
- 填写表单字段
- 保持参数不变

**用例示例：**

```
用例 ID：TC-AUTH-001
标题：用户使用正确账号密码登录成功
功能点：登录 > 账号密码登录
版本：v1.9
前置条件：用户账号已注册且状态正常
测试步骤：
  1. 打开登录页面
     → 预期：显示登录表单，包含用户名输入框、密码输入框和「登录」按钮
  2. 输入正确的用户名「testuser」
  3. 输入正确的密码「password123」
  4. 点击「登录」按钮
     → 预期：页面跳转至首页，顶部导航显示用户昵称「testuser」
优先级：P0
测试维度：正向
```

## 版本标记规范

**版本来源**：版本信号可来自任意输入——需求文档的版本标注、代码 release tag、文档头部版本号等，不限于单一来源。

**新需求的版本标记**：输入包含新增功能点时，若检测到版本信号，以该版本标记新增用例；未检测到版本信号时，版本留空。

**更新已有用例**：保留用例原有版本号，不因更新操作而变更。

**无法识别版本时**：版本字段留空，不猜测，不假设。

## 功能点路径规范

### 格式

```
功能点：{一级功能点} > {二级功能点} > {三级功能点} > ...
```

- 使用 `>` 作为层级分隔符
- 层级数量不限，按需扩展
- 每级名称使用业务语言，不超过 10 字

### 叶节点挂载原则

**用例只挂载到路径的最末节点，中间节点不挂用例。**

错误示例（同一功能点既挂用例又有子节点）：
```
功能点：新增管线            ← 有用例 TC-A
功能点：新增管线 > 名称校验  ← 也有用例 TC-B
```

正确示例：
```
功能点：新增管线 > 必填校验  ← 所有必填类用例统一挂此节点
功能点：新增管线 > 名称校验  ← 名称规则类用例挂此节点
```

### 脑图映射规则

路径中每个 `>` 对应脑图的一个层级，最末节点下列出该路径的所有用例标题。

### 路径扩展时机

满足以下任一条件时，考虑在当前层级下新增子节点：
- 同一功能点下用例超过 5 条，且存在明显的子分类
- 需求细化带来新的子功能，原路径粒度不足以区分

## 优先级定义

| 级别 | 定义 | 典型场景 |
|------|------|---------|
| P0 | 核心业务流程不可用 / 数据损坏风险 / 上下游阻断 | 登录失败、支付无法完成、数据写入异常 |
| P1 | 功能降级但主流程可用 / 边界异常 / 非核心功能故障 | 错误提示文案不对、边界值处理错误、非主路径异常 |
| P2 | 体验问题 / 极低频场景 / 文案错误 | 页面加载偏慢、按钮文案错别字、极端输入无响应 |
| P3 | 极低优先级 / 锦上添花 / 不影响使用的优化点 | 样式微调、快捷键、极端输入下的友好提示改进 |

**PRD 优先级优先原则：** 需求文档对业务场景已标注优先级时，用例优先级**强制对齐 PRD 标注**，不由 qalore 自行推断。仅当 PRD 未标注时，才按上表规则判断。

## 测试维度

| 维度 | 说明 | 设计要点 |
|------|------|---------|
| 正向路径 | 标准流程，输入合法，预期成功 | 覆盖主干业务流程的完整执行链路 |
| 边界条件 | 输入边界值、极值、空值、最大长度 | 关注数值边界（最大/最小/±1）和字符串长度 |
| 异常场景 | 错误输入、权限不足、网络超时、并发冲突 | 覆盖错误处理路径和异常恢复机制 |
| 上下游影响 | 此功能变化对依赖它的其他模块的影响 | 梳理模块间依赖，关注接口契约变化 |

## 跨模块 TC 归属规则

上下游影响类 TC 挂载在**行为可观测的模块**（通常是下游/受影响方）：

| 场景 | TC 归属 | 示例 |
|------|---------|------|
| A 模块操作 → B 模块展示/数据发生变化 | 写在 B 模块 | 管线版本删除 → 图生图再来一张提醒 → TC 写在图生图 |
| A 模块权限变更 → B 模块访问受限 | 写在 A 模块（权限控制方）| 权限管理 → 管线管理不可访问 → TC 写在权限管理 |

同时在对应的【上下游】断言行末追加 `→ 见 TC-{PREFIX}-{seq}`，建立断言与用例的双向关联。

## 命名规则

### 用例 ID

格式：`TC-{PREFIX}-{序号}`

| 字段 | 规则 |
|------|------|
| PREFIX | 模块英文意译首字母缩写（见「TC 前缀规范」），无需求管理系统时直接使用模块缩写 |
| 序号 | 3 位数字，从 001 开始，同 PREFIX 内唯一递增 |

示例：`TC-AUTH-001`、`TC-PAY-003`、`TC-ORDER-012`

#### TC 前缀规范

TC 前缀是模块英文意译的首字母缩写（2-5 个大写字母），在项目范围内唯一，**注册后不可变更**（变更导致历史 ID 断链）。

注册过程由 qa-functional-test 在首次设计用例时执行，结果写入 story 的 index.json。

| 已注册示例 | 模块名 | TC 前缀 | .mm 短 ID |
|---------|-------|---------|----------|
| AIGC平台 | 管线管理 | PIPE | `p{序号}` |
| AIGC平台 | 图生图 | IMG | `i{序号}` |
| AIGC平台 | AI创作-工具 | TOOL | `t{序号}` |
| AIGC平台 | 资源库-我的资产 | ASSET | `a{序号}` |

`.mm` 短 ID 规则：取 TC 前缀全小写 + 序号数字，例：`TC-PIPE-001` → `p001`、`TC-AGT-001` → `agt001`。多模块项目中前缀首字母可能冲突时，可使用前缀全小写确保唯一性。实际值以 index.json `mm_short_id` 为准。

### 文件命名

| 文件 | 命名格式 |
|------|---------|
| 测试用例 | `{模块名}-功能-测试用例.md` |
| 变更日志 | `{模块名}-功能-测试用例.changelog.md` |

## 质量底线

### 覆盖率规则

覆盖来源分两个层级，优先级不同：

**第一层级：PRD 业务场景（必须全量覆盖）**
产品文档中明确描述的每一个业务场景，必须输出对应的测试用例，不得遗漏。

**第二层级：测试方法衍生场景（默认最简）**
边界值、异常路径、上下游影响等由测试方法推导出的场景，默认每类只输出最典型的 1 条。用户明确要求「尽量覆盖」时，扩展至全量输出。

| 来源 | 默认输出策略 | 用户要求全覆盖时 |
|------|------------|----------------|
| PRD 明确描述的业务场景 | 全量输出（强制） | — |
| 测试方法衍生场景 | 每类最多 1 条最典型 | 尽量全量输出 |

**第三层级：上下游依赖覆盖（强制）**

若模块业务逻辑.md 中存在带 `[模块::功能点]` 标签的【上下游】断言，测试用例设计必须针对每个跨模块引用至少输出 1 条上下游维度用例。

| 触发条件 | 覆盖要求 |
|---------|---------|
| 业务逻辑.md 中无带 `[模块::功能点]` 标签的【上下游】断言 | 不强制 |
| 业务逻辑.md 中存在带 `[模块::功能点]` 标签的【上下游】断言 | 每个跨模块引用至少 1 条上下游用例（强制）|

#### 判断说明

**PRD 明确描述** = 需求文档中有明确文字描述的功能行为、交互逻辑、字段规则或系统响应。

**测试方法衍生场景** = 测试工程师根据测试方法论自行推导出的场景，需求文档中无对应文字描述。

| 示例场景 | 分类 |
|---------|------|
| "版本名称最多 50 个字符" | PRD 明确描述 |
| 输入 51 字符被拒绝 | PRD 明确描述（上述规则的直接对应行为）|
| 输入特殊字符的行为 | 衍生场景 |
| 上传后网络超时的处理 | 衍生场景 |

### 用例质量标准

**步骤可执行性**：每个测试步骤必须是单一、具体、可操作的动作，明确指出操作对象和操作方式。禁止使用「正常使用」「操作该功能」「进行相应操作」等无法执行的描述。

**逐步预期完整性**：触发系统可观察响应的步骤（页面跳转、UI 状态变化、系统反馈、数据变更）必须在该步骤下紧跟「→ 预期：{内容}」。纯输入步骤（填写字段、选择选项）可省略。

**预期内容可观察性**：每处预期必须描述具体可感知的变化（UI 变化、返回内容、状态变化、数据变更等）。禁止使用「系统正常响应」「操作成功」「功能正常」等无法判断通过/失败的描述。

**用例独立性**：单条用例只验证一件事，预期结果不应包含多个不相关的可观察点。

**前置条件充分性**：前置条件必须足以独立复现测试场景，需明确角色/权限、数据状态、系统配置等。禁止使用「已登录」（未指定角色）、「有数据」（未说明数据状态）等不完整描述。

**用例 ID 唯一性**：全局唯一，格式遵循「命名规则」章节，设计前扫描已有最大序号，新 ID 从 N+1 递增。

## TC 语言规范

### 用户视角优先

测试步骤和预期结果必须从用户可观察的角度描述，禁止暴露实现细节：

| 正确（用户视角） | 错误（工程语言） |
|------|------|
| 点击"发布"按钮 | 调用 POST /api/agent/{id}/publish |
| 页面顶部弹出黄色警告"仅超级管理员可创建智能体" | ElMessage.warning("仅超级管理员可创建智能体") |
| 选中项高亮显示（浅蓝紫色背景） | .agent-block.active 背景为 #e0e7ff |
| 退出登录→清除登录状态→页面跳转到登录页 | authService.clearAuth()+router.replace('/login') |

### 灰盒注解格式

如需标注工程上下文，使用 `（工程描述）` 格式附加在场景描述之后：

```
点击"发布"按钮（调用 POST /api/agent/{id}/publish，AgentController.publish()→setStatus("published")）→ Agent状态标签从"草稿"变为"已发布"
访问 /agents 页面（加载 AgentList.vue，调用 GET /api/agent/list）→ 左侧栏展示Agent头像列表
```

### 禁止暴露的实现细节

| 类别 | 禁止模式 | 示例 |
|------|---------|------|
| 组件名 | `ElMessage` `AgentBaseSetting` `MarkdownAgentContainer` | 禁止出现在步骤和预期中 |
| CSS 类名 | `.agent-block` `.chat-sidebar` `.preset-chip` `.empty-flow-stage` | 禁止出现在步骤和预期中 |
| 内部变量 | `showDashedBorder` `sidebarCollapseKey` `autoScroll` `collapsedTimelines` | 禁止出现在步骤和预期中 |
| 服务调用 | `authService.xxx()` `router.push()` `AgentService.xxx()` | 仅允许在灰盒注解 `（）` 中出现 |
| API 端点 | `POST /api/agent` `GET /api/agent/list` | 仅允许在灰盒注解 `（）` 中出现 |

## TC 覆盖验证

qa-functional-test 产出 TC 后，必须执行覆盖检查：

1. 遍历 BL 中所有 `## 功能点：{name}` 节
2. 将 TC 的 `功能点：` 字段第一级与 BL 节名做模糊匹配（包含关系）
3. 无匹配 → 标记为 `UNCOVERED`，警告用户补充
4. "代码独立断言"节豁免覆盖检查

覆盖报告格式：

```
| 模块 | BL 功能点 | TC 覆盖 | 状态 |
|------|----------|---------|------|
| 智能体配置 | 知识库配置 | 0 条 | UNCOVERED |
| 系统管理 | Prompt模板管理 | 0 条 | UNCOVERED |
```

## 断言规则（强制）

每条 TC 的每个 `→ 预期：` 必须紧跟至少一个 `→ 断言：`。

断言规则由 qa-functional-test 在设计 TC 时**同步产出**，不可后补。
无断言规则的 TC 视为不可执行，qa-execution 将标记 SKIP。

### 格式

```
→ 预期：{自然语言描述，人可读}
→ 断言：{断言类型}({参数})
→ 断言：{断言类型}({参数})
```

- 一个 `→ 预期：` 可对应多个 `→ 断言：`
- `→ 断言：` 行紧跟在对应的 `→ 预期：` 行之后
- 参数含空格或特殊字符时用双引号包裹

### 断言类型

断言类型**仅在 `execution.md` 中定义**（权威源），此处列出名称供 TC 设计时查阅：

| 类型 | 用途 | 示例 |
|------|------|------|
| `element-exists` | UI 元素存在 | `element-exists(.new-chat-btn)` |
| `element-not-exists` | UI 元素不存在 | `element-not-exists(.new-chat-btn)` |
| `element-count` | UI 元素数量 | `element-count(.group-label, 5)` |
| `element-text` | UI 元素文本 | `element-text(.sidebar-title, "历史记录")` |
| `element-width` | UI 元素宽度 | `element-width(.chat-sidebar, 380)` |
| `message-contains` | Element UI 消息 | `message-contains("请输入请求消息！")` |
| `url-contains` | 页面 URL | `url-contains("/login")` |
| `api-status` | API HTTP 状态 | `api-status(/api/agent/.*/sessions, 200)` |
| `api-field-exists` | API 响应字段 | `api-field-exists(/api/agent/.*/sessions, "total")` |
| `api-field-value` | API 响应字段值 | `api-field-value(/api/agent/.*/sessions, "total", 9)` |

### 告示例

```markdown
测试步骤：
1. 输入框为空点击发送     → 预期：弹出黄色警告"请输入请求消息！"
   → 断言：message-contains("请输入请求消息！")

2. 展开历史记录栏         → 预期：侧栏宽度380px，标题为"历史记录"
   → 断言：element-width(.chat-sidebar, 380)
   → 断言：element-text(.sidebar-title, "历史记录")
```

### TC 产出验证

qa-functional-test 产出 TC 后必须验证（执行后验证第 5 项）：
- 每条 TC 的每个 `→ 预期：` 至少对应一个 `→ 断言：`
- 断言类型在 execution.md 定义的 10 种范围内

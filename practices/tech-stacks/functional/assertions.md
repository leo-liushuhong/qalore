# 可测试断言规范

可测试断言是从信息源（需求文档、代码等）中提炼的、可被测试用例直接使用的最小可验证单元。

---

## 格式

```
[assert-{缩写}-{seq}] 【维度】（{置信度标注}）对象 · 条件 → 可观察预期 | upd:{版本/日期}
```

**上下游断言跨模块格式（目标为其他模块时）：**
```
[assert-{缩写}-{seq}] 【上下游】对象 · 条件
  → [目标模块::目标功能点/组件] 影响描述 | upd:{版本/日期}
```

**多源模式来源标注（synthesis 区分来源时使用）：**
- 文本适配器：在断言行末 `| upd:` 之后追加 `[来源:text]`
- 代码适配器（多仓库）：在断言行末 `| upd:` 之后追加 `[来源:{仓库名}]`，如 `[来源:frontend]`
- 代码适配器（单仓库多源）：在断言行末 `| upd:` 之后追加 `[来源:code]`
```
[assert-{缩写}-{seq}] 【维度】对象 · 条件 → 预期 | upd:{日期} [来源:text]
[assert-{缩写}-{seq}] 【维度】对象 · 条件 → 预期 | upd:{日期} [来源:{仓库名}]
```

---

## 维度标签

| 来源 | 维度标签 |
|------|---------|
| 需求视角 | 功能 / 状态 / 权限 / 输入约束 / 交互反馈 / 上下游 / 异常 |
| 代码视角 | 边界值 / 状态转换 / 异常处理 / 上下游 |

同一断言只使用一个维度标签。维度标签驱动测试用例分类：【交互反馈】→ 交互类 TC，【状态】→ 展示类 TC，以此类推。

---

## 置信度标注

由 synthesis 写入，单源提炼时无标注。完整定义见 `story-formats.md`「置信度标注规范」章节，不在此重复。

---

## 粒度规范

- 每条断言描述一个独立可验证的事实，粒度等于「一个步骤可以验证的一件事」
- 同一对象的多个可观察结果写多条断言，不合并
- 待确认项：信息源未描述的维度标注「未描述」写入待确认项，不猜测，不跳过

---

## 断言质量标准

**条件**：必须具体可触发，禁止「正常情况下」「一般」「通常」等模糊触发描述。

**预期**：必须可观察，禁止「系统正常响应」「操作成功」「功能正常」等无法判断通过/失败的表述。

**对象**：必须指向具体的系统元素（字段、按钮、接口、状态等），不泛指。

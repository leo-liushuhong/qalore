# Story 文件格式规范

testcraft 沉淀的 story 文件格式定义。所有读取或写入这些文件的 capability 遵循此规范。

---

## 业务逻辑文件：`{模块名}-功能-业务逻辑.md`

由 qa-understand 文本适配器生成，记录从需求视角提炼的可测试断言。

```markdown
# {模块名} 功能业务逻辑
> 需求来源版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}

## 功能概述
{一句话：面向谁、解决什么问题、核心职责}

## 可测试断言
> 格式：【维度】需求对象 · 条件 → 可观察预期
> 维度标签：功能 / 状态 / 权限 / 输入约束 / 交互反馈 / 上下游 / 异常

【功能】{需求对象} · {触发条件} → {可观察的预期结果}
【状态】{需求对象} · {状态条件} → {该状态下的可观察表现}
...

## 待确认项（PRD未描述，需与 PM 确认）
- 【{维度}】{需求对象}：{追问内容}，输入未描述
```

---

## 代码逻辑文件：`{模块名}-功能-代码逻辑.md`

由 qa-understand 代码适配器生成，记录从代码视角提炼的可测试断言。

```markdown
# {模块名} 功能代码逻辑
> 代码路径：{path} | 入口：{entry} | 阅读深度：depth={N} | 读取日期：{YYYY-MM-DD} | 代码参考：{branch@commit（可选）}

## 技术概述
{一句话：核心实现思路、主要技术手段}

## 代码来源
| 文件/路径 | 入口函数/类 | 阅读深度 | 读取日期 |

## 可测试断言（代码维度）
> 格式：【维度】代码单元 · 条件 → 可观察预期
> 维度标签：输入约束 / 权限 / 边界值 / 状态转换 / 上下游 / 异常处理

【输入约束】{参数/字段} · {违反约束的输入} → {报错类型/错误消息/响应码}
【状态转换】{当前状态} · {触发动作} → {目标状态/副作用}
...

## 待确认项（代码行为不明确）
- 【{维度}】{代码单元}：{具体不明确的内容}
```

---

## 测试用例文件：`{模块名}-功能-测试用例.md`

由 qa-functional-test 生成，用例字段格式遵循 `tech-stacks/functional/cases.md`「用例格式」章节。

```markdown
# {模块名} 功能测试用例
> TC 前缀：{PREFIX} | 用例总数：{n} 条 | 最新版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}
```

文件包含该模块的全部累积用例条目，按新增顺序排列，不按功能点分组（分组由脑图产物承担）。

**头部更新规则：** 每次写入测试用例后同步更新头部的用例总数、最新版本和最后更新日期。

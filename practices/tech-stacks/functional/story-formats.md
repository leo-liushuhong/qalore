# Story 文件格式规范

testcraft 沉淀的 story 文件格式定义。所有读取或写入这些文件的 capability 遵循此规范。

---

## 断言 ID 规范

每条可测试断言都有唯一 ID，格式：`[assert-{模块缩写}-{3位序号}]`

- 模块缩写与 TC 前缀一致（如 IMG、PIPE）
- 序号在模块内全局递增，跨文件（业务逻辑.md 和代码逻辑.md 共享同一序号空间）
- text adapter 生成断言时分配 ID，从 001 开始；已有文件时从最大序号 +1 开始
- code adapter 生成断言时分配 ID，从当前模块已有最大序号 +1 开始（读业务逻辑.md 或 index.json 的 assert_seq 获取）
- synthesis 识别同一场景后，将 code_logic.md 中对应断言的 ID 改写为与 business_logic.md 相同的值，并在业务逻辑.md 对应断言行追加置信度标注

**同一 assert ID 在两个文件中出现 = 同一场景的不同视角**，是跨文件关联的唯一依据。

**置信度标注**（由 synthesis 写入业务逻辑.md，跨会话持久化）：

| 标注 | 含义 |
|------|------|
| `（✓）` | 两个视角互相印证，已验证 |
| `（待代码确认）` | PRD 有描述，代码中未找到对应实现 |
| `（仅实现视角）` | 仅代码有，PRD 未描述，写在业务逻辑.md 末尾补充节 |
| `⚠️` | 文本与代码存在冲突，需确认 |

无置信度标注 = 单源提炼（仅文本适配器运行过，尚未与代码视角对照）。

## 业务逻辑文件：`{模块名}-功能-业务逻辑.md`

由 qa-understand 文本适配器生成，记录从需求视角提炼的可测试断言。

```markdown
# {模块名} 功能业务逻辑
> 需求来源版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}

## 功能概述
{一句话：面向谁、解决什么问题、核心职责}

## 可测试断言
> 格式：[assert-{模块缩写}-{seq}] 【维度】（{置信度标注}）需求对象 · 条件 → 可观察预期
> 置信度标注由 synthesis 写入，单源提炼时无标注；维度标签：功能 / 状态 / 权限 / 输入约束 / 交互反馈 / 上下游 / 异常

[assert-IMG-001] 【功能】（✓）{需求对象} · {触发条件} → {可观察的预期结果}
[assert-IMG-002] 【状态】{需求对象} · {状态条件} → {该状态下的可观察表现}
[assert-IMG-003] 【输入约束】（待代码确认）{需求对象} · {条件} → {预期}
[assert-IMG-004] 【权限】⚠️ {需求对象} · {条件} → 文本：{描述A} / 代码：{描述B}

## 代码补充断言（仅实现视角，PRD 未描述，由 synthesis 追加）
[assert-IMG-005] 【边界值】（仅实现视角）{代码单元} · {边界值} → {处理结果}

## 待确认项（PRD未描述，需与 PM 确认）
- [pending]   【{维度}】{需求对象}：{追问内容}，输入未描述
- [resolved]  【{维度}】{需求对象}：{追问内容} → {确认结论}（{YYYY-MM-DD} {来源}）
              → 已转化为正式断言 assert-{模块缩写}-{seq}
- [dismissed] 【{维度}】{需求对象}：{追问内容} → 无需覆盖（{YYYY-MM-DD} {原因}）
```

## 代码逻辑文件：`{模块名}-功能-代码逻辑.md`

由 qa-understand 代码适配器生成，记录从代码视角提炼的可测试断言。

```markdown
# {模块名} 功能代码逻辑
> 代码路径：{path} | 入口：{entry} | 阅读深度：depth={N} | 读取日期：{YYYY-MM-DD} | 代码参考：{branch@commit（可选）}

## 技术概述
{一句话：核心实现思路、主要技术手段}

## 代码来源
| 文件/路径 | 入口函数/类 | 阅读深度 | 读取日期 |

## 可测试断言（代码维度）
> 格式：[assert-{模块缩写}-{seq}] 【维度】代码单元 · 条件 → 可观察预期
> 与业务逻辑.md 共享同一 assert ID 空间；synthesis 对齐后，同一场景两文件 ID 相同
> 维度标签：输入约束 / 权限 / 边界值 / 状态转换 / 上下游 / 异常处理

[assert-IMG-001] 【权限】{代码单元} · {条件} → {实现侧结果}   ← 与业务逻辑.md assert-IMG-001 同一场景
[assert-IMG-005] 【边界值】{参数} · {边界值} → {处理结果}     ← 仅代码视角，无对应业务断言
...

## 待确认项（代码行为不明确）
- [pending]   【{维度}】{代码单元}：{具体不明确的内容}
- [resolved]  【{维度}】{代码单元}：{不明确内容} → {确认结论}（{YYYY-MM-DD} {来源}）
              → 已转化为正式断言 assert-{模块缩写}-{seq}
- [dismissed] 【{维度}】{代码单元}：{不明确内容} → 无需覆盖（{YYYY-MM-DD} {原因}）
```

## 测试用例文件：`{模块名}-功能-测试用例.md`

由 qa-functional-test 生成，用例字段格式遵循 `tech-stacks/functional/cases.md`「用例格式」章节。

```markdown
# {模块名} 功能测试用例
> TC 前缀：{PREFIX} | 用例总数：{n} 条 | 最新版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}
```

文件包含该模块的全部累积用例条目，按新增顺序排列，不按功能点分组（分组由脑图产物承担）。

**头部更新规则：** 每次写入测试用例后同步更新头部的用例总数、最新版本和最后更新日期。

**assert ID 关联：** 每条用例的「功能点」字段后可追加 `[assert-{ID}]` 标注，记录该用例覆盖的断言 ID，供后续知识图谱查询使用（可选，有则填写）。

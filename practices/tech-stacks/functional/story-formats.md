# Story 文件格式规范

qalore 沉淀的 story 文件格式定义。所有读取或写入这些文件的 capability 遵循此规范。

---

## 置信度标注规范

synthesis 层写入，单源提炼时无标注。

| 标注 | 含义 |
|------|------|
| `（✓）` | 文本 + 所有代码源均确认，结论一致 |
| `（✓,部分代码确认）` | 文本 + 部分代码源确认，其余代码源未覆盖 |
| `（待代码确认）` | 仅文本描述，无任何代码源实现 |
| `（代码独立）` | 仅单一代码源发现，文本未描述 |
| `（代码独立,多源）` | 多个代码源均发现，文本未描述 |
| `⚠️` | 文本与代码源结论矛盾，需确认 |
| `⚠️(代码)` | 多个代码源描述同一场景但结论矛盾，需确认 |

无标注 = 单源提炼，尚未与其他来源对照。

## 断言 ID 规范

每条可测试断言都有唯一 ID，格式：`[assert-{模块缩写}-{3位序号}]`

- 模块缩写与 TC 前缀一致（如 IMG、PIPE）
- 序号在模块内全局递增，跨文件（业务逻辑.md 和代码逻辑.md 共享同一序号空间）
- text adapter 生成断言时分配 ID，从 001 开始；已有文件时从最大序号 +1 开始
- code adapter 生成断言时分配 ID：读取 context 中 `【qa-understand-mode】` 标记（由 qa-understand 调度层写入）；值为 `multi` 时从 `【assert_seq_runtime: N】` 续接；值为 `single` 或不存在时从 index.json 的 `assert_seq` 或业务逻辑.md 的最大序号获取
- synthesis 识别同一场景后，将 code_logic.md 中对应断言的 ID 改写为与 business_logic.md 相同的值，并在业务逻辑.md 对应断言行追加置信度标注

**同一 assert ID 在两个文件中出现 = 同一场景的不同视角**，是跨文件关联的唯一依据。

置信度标注详见本文件「置信度标注规范」章节（文件顶部）。

**多源代码断言来源标注**（code adapter 在多源模式下写入）：
- `[来源:仓库名]` 标注在断言行末，紧跟 `| upd:` **之后**
- 例：`[assert-IMG-005] 【边界值】... | upd:2026-05-14 [来源:frontend]`
- synthesis 依此区分哪些断言来自哪个代码源

## 业务逻辑文件：`{模块名}-功能-业务逻辑.md`

由 qa-understand 文本适配器生成，记录从需求视角提炼的可测试断言。按功能点分节，每节维护当前最新状态。

```markdown
# {模块名} 功能业务逻辑
> 需求来源：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD} | 功能点：{n} 个 | 断言：{n} 条

## 功能概述
{一句话：面向谁、解决什么问题、核心职责}

## 功能点：{feature1名}
> last-upd:{v1.2 / YYYY-MM-DD}

> 断言格式：[assert-{缩写}-{seq}] 【维度】（{置信度标注}）需求对象 · 条件 → 可观察预期 | upd:{版本/日期}
> 置信度标注由 synthesis 写入，单源提炼时无标注；多源模式下行末追加 `[来源:text]`（供 synthesis 来源识别）；维度标签：功能 / 状态 / 权限 / 输入约束 / 交互反馈 / 上下游 / 异常

[assert-IMG-001] 【功能】（✓）{需求对象} · {触发条件} → {可观察的预期结果} | upd:v1.2
[assert-IMG-002] 【状态】{需求对象} · {状态条件} → {该状态下的可观察表现} | upd:v1.2
[assert-IMG-006] 【上下游】{需求对象} · {触发条件}
  → [目标模块::目标功能点] {跨模块影响描述} | upd:v1.2

## 功能点：{feature2名}
> last-upd:{v2.0}

[assert-IMG-003] 【输入约束】（待代码确认）{需求对象} · {条件} → {预期} | upd:v2.0
[assert-IMG-004] 【权限】⚠️ {需求对象} · {条件} → 文本：{描述A} / 代码：{描述B} | upd:v2.0

## 代码独立断言（代码发现的系统行为，PRD 未描述，由 synthesis 追加）
> last-upd:{YYYY-MM-DD}

[assert-IMG-005] 【边界值】（代码独立）{代码单元} · {边界值} → {处理结果} | upd:{YYYY-MM-DD}

## 待确认项（PRD未描述，需与 PM 确认）
- [pending]   【{维度}】{需求对象}：{追问内容}，输入未描述
- [resolved]  【{维度}】{需求对象}：{追问内容} → {确认结论}（{YYYY-MM-DD} {来源}）
              → 已转化为正式断言 assert-{模块缩写}-{seq}
- [dismissed] 【{维度}】{需求对象}：{追问内容} → 无需覆盖（{YYYY-MM-DD} {原因}）
```

## 业务逻辑 changelog 文件：`{模块名}-功能-业务逻辑.changelog.md`

记录业务逻辑.md 每次变动，只追加，永不删除。格式见 `tech-stacks/functional/changelog.md`「断言 changelog 格式」章节。

## 代码逻辑文件：`{模块名}-功能-代码逻辑.md`

由 qa-understand 代码适配器生成，记录从代码视角提炼的可测试断言。按组件分节，每节维护当前最新状态。

```markdown
# {模块名} 功能代码逻辑
> 最后更新：{YYYY-MM-DD} | 组件：{n} 个 | 断言：{n} 条

## 技术概述
{一句话：核心实现思路、主要技术手段}

## 代码来源
| 文件/路径 | 入口函数/类 | 阅读深度 | 读取日期 |

## 组件：{组件1名}
> last-upd:{YYYY-MM-DD / v1.2}

> 断言格式：[assert-{缩写}-{seq}] 【维度】代码单元 · 条件 → 可观察预期 | upd:{版本/日期}
> 与业务逻辑.md 共享同一 assert ID 空间；synthesis 对齐后，同一场景两文件 ID 相同
> 维度标签：输入约束 / 权限 / 边界值 / 状态转换 / 上下游 / 异常处理

[assert-IMG-001] 【权限】{代码单元} · {条件} → {实现侧结果} | upd:v1.0
[assert-IMG-005] 【边界值】{参数} · {边界值} → {处理结果} | upd:{YYYY-MM-DD}
[assert-IMG-007] 【上下游】{代码单元} · {触发条件}
  → [目标模块::目标组件] {跨模块调用描述} | upd:{YYYY-MM-DD}
[assert-IMG-008] 【边界值】{参数} · {边界值} → {处理结果} | upd:2026-05-14 [来源:frontend]

## 组件：{组件2名}
> last-upd:{YYYY-MM-DD}

[assert-IMG-006] 【上下游】{代码单元} · {条件} → {副作用} | upd:{YYYY-MM-DD}

## 待确认项（代码行为不明确）
- [pending]   【{维度}】{代码单元}：{具体不明确的内容}
- [resolved]  【{维度}】{代码单元}：{不明确内容} → {确认结论}（{YYYY-MM-DD} {来源}）
              → 已转化为正式断言 assert-{模块缩写}-{seq}
- [dismissed] 【{维度}】{代码单元}：{不明确内容} → 无需覆盖（{YYYY-MM-DD} {原因}）
```

记录代码逻辑.md 每次变动，只追加，永不删除。格式见 `tech-stacks/functional/changelog.md`「断言 changelog 格式」章节。

## 测试用例文件：`{模块名}-功能-测试用例.md`

由 qa-functional-test 生成，用例字段格式遵循 `tech-stacks/functional/cases.md`「用例格式」章节。

```markdown
# {模块名} 功能测试用例
> TC 前缀：{PREFIX} | 用例总数：{n} 条 | 最新版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}
```

文件包含该模块的全部累积用例条目，按新增顺序排列，不按功能点分组（分组由脑图产物承担）。

**头部更新规则：** 每次写入测试用例后同步更新头部的用例总数、最新版本和最后更新日期。

**assert ID 关联：** 每条用例的「功能点」字段后可追加 `[assert-{ID}]` 标注，记录该用例覆盖的断言 ID，供后续知识图谱查询使用（可选，有则填写）。

---

## 评审记录文件：`{模块名}-功能-评审记录.md`

由 qa-case-review 在用户同意后追加写入（文件不存在时新建），仅在 story 模式和联动模式下可用。

```markdown
## {YYYY-MM-DD} 评审记录
- 评审用例数：{n} 条 | 规范版本：{practices version}
- 阻断：{n} 条 / 需改进：{m} 条 / 建议：{k} 条
- 结论：{通过 / 需修复后通过 / 不通过}
- 阻断问题摘要：
  - {TC-ID}：{一句话描述}
  - {TC-ID}：{一句话描述}
```

写入规则：追加到文件末尾，新条目在旧条目之后；永不删除历史条目。

---

## 统一交接块格式：`【测试意图已提炼】`

qa-understand 向 qa-functional-test / qa-case-review 传递理解成果的跨 capability 数据契约。格式如下：

```
【测试意图已提炼】
项目：{项目名}
模块：{模块名}
信息源：{文本 / 代码（{仓库名}）/ 混合（文本 + 代码源A（{仓库名}） + 代码源B（{仓库名}）...）}

可测试断言：
  [assert-{PREFIX}-{seq}] 【{维度}】（{置信度标注}）{对象} · {条件} → {可观察预期}
    ↳ {补充说明（有额外信息时追加，无则省略）}

待确认项（有则输出，无则省略）：
  【{维度}】{对象}：{具体不明确内容}

回归候选（有则输出，无则省略）：
  · {模块名}（code_paths 重叠）：分析路径与该模块代码路径交叉，建议验证受影响功能点
  · {模块名}（正向依赖）：本模块 depends_on 中包含该模块，本次修改可能影响其接口
  · {模块名}（反向依赖）：该模块的 depends_on 包含本模块，本次修改可能影响该调用方
  · {模块名}（业务关联）：本模块 business_related 中包含该模块的功能点
```

**置信度标注规则（与置信度标注规范章节一致）：**

| 标注 | 含义 |
|------|------|
| `（✓）` | 文本 + 所有代码源确认 |
| `（✓,部分代码确认）` | 文本 + 部分代码源确认 |
| `（待代码确认）` | 仅文本，无代码源实现 |
| `（代码独立）` | 仅单一代码源发现 |
| `（代码独立,多源）` | 多个代码源均发现，文本未描述 |
| `⚠️` | 文本与代码源结论矛盾 |
| `⚠️(代码)` | 多个代码源描述同一场景但结论矛盾 |
| 无标注 | 单源提炼，尚未与其他来源对照 |

**消费规则（qa-functional-test / qa-case-review）：**
- context 中存在此块 → 以此为主输入，优先于 story 中的文件
- context 中不存在 → 读 story 中的文件（见各 capability 的输入降级规则）

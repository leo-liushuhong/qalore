qalore

AI 测试工作的持久记忆层

解决 AI 跨会话无持久记忆的根本问题——
让每次测试分析都不从零开始，让测试知识随项目一起生长。

🧠 跨会话知识积累 🎯 用户影响优先 🔍 7问/五镜头结构保底 🔗 PRD→断言→TC 全链路

核心价值

原生 AI 测试 vs qalore

原生 AI 每次都是第一天上班。qalore 让 AI 的测试工作有记忆、有积累、有方向。

😓 原生 AI 的困境

✗每次重新分析，无法从上次中断处继续
✗测试偏"验证性"——系统是否工作，而非用户会在哪里受伤
✗覆盖随机性高，容易遗漏关键测试维度
✗多模块无全局视图，团队无法共享测试知识
✗PRD→TC 黑盒，无法知道用例来自哪条需求

✨ qalore 的解法

✓story 系统持久化，下次从断点继续，无重复劳动
✓以"用户最先感受到什么损失"为测试优先级锚点
✓7问/五镜头结构保底，不遗漏任何测试维度
✓index.json 统一管理进度，story 是共享知识库
✓断言链路显式记录，每条 TC 有明确的需求来源

核心思考哲学

不是工具，是思考框架

qalore 的核心竞争力是三套互补的认知框架。任何模型按这套框架执行，大方向上的测试质量都能得到保障。

🧭
读信息时（PRD / 代码 / 口述）

三驱动认知框架

读任何信息源时，始终用一套驱动体系保持认知方向——确保每次阅读都是目标驱动的，而不是信息堆砌。

💡 假设优先 读前先假设「模块做什么、谁触发、影响什么」。读是在验证假设，不是从零积累信息。
🔄 模型驱动 每读一段，问：这证实还是推翻了假设的哪个部分？
📋 证据驱动 每条论断必须有来源（PRD 文字 / 代码实现）。「我认为」不是断言，是待验证假设。
🔗 边界驱动 每个交互都有两端。遇到边界（外部 API、跨模块依赖）时，判断另一端是否已知。
🎯 认知目标：完整理解功能行为（为验证类测试打基础）+ 识别潜在风险点（为异常类测试指方向）——两者都不能偏废。
📖
读大文档 / 大文件时

渐进式阅读策略

大文档不是从头读到尾，而是两阶段策略。不限工具类型，任何 agent 自行选择实现方式。

Phase 1
导航（建地图） 扫结构，定位置
成本极低
→
Phase 2
切片精读 按问题读语义单元
有问题才继续

代码的语义单元 = 函数体  ·  文档的语义单元 = 功能点段落

🎯
设计测试用例时

用户影响优先

测试优先级不由断言系统顺序决定，而由一个问题决定：

如果这个模块今天在生产中出问题，用户最先感受到什么损失？

以这个答案定义 2-3 个优先场景，多角度覆盖。再用断言集合补全系统性覆盖。
⚠️ 和「待代码确认」断言自动获得最高优先级——这是系统中已知的不确定点，最容易出 bug。

快速上手

三步开始使用

qalore 当前以 Claude Code skill 形式提供；思考框架本身不绑定工具，任何 AI 均可按其执行。

1

创建配置文件

告诉 qalore 你的规范库和知识库在哪里：

// ~/.claude/qalore-config.json
{
  "practices_path": "/path/to/your/practices",
  "story_path": "/path/to/your/story"
}
2

初始化 practices 规范库

参考 practices-bootstrap.md 创建你的测试规范文件。定义断言格式、用例标准、覆盖规则——一次建设，所有项目复用。

3

用自然语言触发能力

以下描述在 Claude Code 中会自动识别并调用对应的 qalore 能力：

# 理解需求 → 提炼断言 → 写入 story
"帮我分析登录模块的需求文档，沉淀到 story"

# 阅读代码 → 提炼断言
"帮我阅读 src/auth/ 的代码，更新 story"

# 需求 + 代码同时分析（多源模式）
"读登录模块的 PRD 和代码，更新 story"

# 从 story 生成测试用例
"给登录模块出测试用例"

# 评审已有用例
"评审登录模块的测试用例"

内置能力

三个核心 Capability

每个 capability 装备一种专业领域的思考方式，不是执行步骤清单——框架保证方向，模型贡献深度。

🔬

qa-understand

从需求文档、代码或两者组合中，独立提炼可测试断言，写入 story。支持多仓库同时分析，synthesis 层自动聚合置信度，标注矛盾点。

PRD → 断言 代码 → 断言 多源聚合 置信度判断
📝

qa-functional-test

以用户影响为优先级锚点，从断言集合设计测试用例。可选输出 FreeMind 脑图（.mm），支持多模块任务的自主节奏调度，TC 序号唯一不重复。

断言 → TC 用户影响优先 .mm 脑图 增量更新
🔍

qa-case-review

从结构层（字段完整性）、覆盖层（断言是否都有 TC）、归因层（问题根因在哪里）三个层次评审测试用例质量，输出三级问题报告。

结构检查 覆盖验证 问题归因 三级报告

Story 设计原则

测试知识如何被持久化

story 是项目测试知识的"物化"——让跨会话、跨模型、跨团队协作成为可能。

1

只维护当前态，历史追踪入 changelog

断言按功能点分节，变更时原地更新，不追加历史。这样每个节都是可直接定位的锚点。变更原因写入 .changelog.md，正常任务完全不加载，只在溯源时才读。

2

三层加载，越常用越精简

越频繁访问的层，信息密度越高、体积越小——这是节省 token 的根本机制。

Layer 1  index.json
每次必读，极小，全局概览
Layer 2  内容文件 .md
按需读，只读当前模块
Layer 3  changelog 文件
正常任务不读，只在溯源时用

↑ 条长代表文件体积，不代表访问频率——越小越精简，越频繁访问

3

跨模块引用用结构化标签，自动聚合不手动维护

模块间的依赖关系不手动填写，而是从断言中的 [模块::功能点] 标签自动聚合到 index.json，防止手动维护产生幽灵数据。

[assert-ORDER-012] 【上下游】订单创建 · 支付成功后
  → [库存::扣减] 触发库存扣减事务 | upd:v2.0

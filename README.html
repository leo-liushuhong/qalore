qalore

AI 驱动的测试任务网关

qalore 是一套运行于 AI 助手之上的测试能力框架。它不替代 AI 的智力——而是为 AI 提供测试工作所需的记忆、结构和执行纪律，让测试知识可以在模块之间、 会话之间、模型之间持续积累。

🚀 快速开始 了解网关设计
Why

为什么需要 qalore

原生 AI 在测试任务上存在结构性局限——不是能力不够，而是缺少持久化的测试知识基础设施。qalore 从根本上填补这个空白。

核心问题❌ 原生 AI✅ qalore
跨会话记忆每次重新分析，重复劳动，无法从上次断点继续Story 知识库持久化所有测试资产，下次对话直接从状态文件恢复
覆盖完整性依赖提示词质量，遗漏维度是常态7问追问 + 五镜头结构化框架，系统性覆盖正向/边界/异常/上下游四维度
测试思维方向验证性：系统是否按预期工作对抗性：用户在什么情况下会受伤？从哪里开始出问题？
需求到用例可追溯输入 → 黑盒 → 输出，中间推理不可见断言链路显式记录，每条用例可追溯到具体断言和原始来源
多模块全局管理无全局视图，不知道哪些测了哪些没测index.json 统一管理每个模块的覆盖状态和进度
团队协作每人独立操作，知识无法共享Story 是共享知识库，一次沉淀全员复用
Gateway

网关设计：三大职责

网关是整个 qalore 系统的调度中枢。每次任务启动时，它按固定顺序完成三件事，确保执行环境可靠、意图清晰、上下文完备。

🔧

职责一：环境验证

读取 ~/.claude/qalore-config.json 获取路径配置；校验 practices 和 story 目录存在且可访问；验证 practices 版本一致性（version == changelog[0].version）；对全部已注册 capability 逐一检查其 practices_min_version 与当前 practices 版本的兼容性。任一环节失败则停止，给出明确的修复指引。

🧭

职责二：意图识别

从用户描述中识别项目名（须字符串完全相等才可自动推断，模糊匹配必须暂停确认）；对照三行路由表判断触发哪些 capability；确定多 capability 的执行顺序（有依赖关系的不得并发启动）；无匹配项时向用户说明并请求澄清，不猜测执行。

💉

职责三：上下文注入

触发任何 capability 前，先读取其 SKILL.md，然后向 context 显式注入变量：practices_path、story_path、确认项目名、practices_version、playwright_mcp_available。多个 capability 共享同一套注入值，各自按需加载 practices 文件。

⚠️ 三条不可违反的约束
  • 路径约束：practices_path 和 story_path 必须从 config 文件读取，不得硬编码、推断或猜测。
  • 能力约束：只有注册在可用列表中的 capability 才能执行，不得用通用测试知识替代未建设的能力。
  • 项目约束：项目名必须由用户明确提供或从对话中唯一识别，不得自行假设。

意图路由表

网关根据用户措辞自动匹配 capability，支持同时匹配多个并按依赖顺序调度。

用户意图信号触发 Capability前置条件

Capabilities

四大核心能力

每个 capability 拥有独立的思考哲学和执行流程，由网关按路由表调度，通过上下文注入获得运行所需变量。

qa-understand 测试意图理解与提炼 ✅ 可用
输入：PRD / 代码 / 混合 思考：三驱动 + 7 问追问 + 波及与覆盖分析 输出：业务逻辑.md + 代码逻辑.md + 交接块

定位：将任意信息源转化为可测试的结构化理解。是 qa-functional-test 和 qa-case-review 的输入生产者。

认知框架（三驱动）：

  • 🔮 假设优先 — 读之前先用一句话假设模块"做什么、谁触发、影响什么"，读是验证假设，不是从零积累。
  • 🧩 模型驱动 — 每读一段问：这证实还是推翻了假设的哪个部分？
  • 📎 证据驱动 — 每条论断必须有来源（PRD 原文 / 代码行），"我认为"只是待验证假设。
  • 🔗 边界驱动 — 每个系统交互都有两端，遇边界判断另一端是否已知，未知且关键则主动追踪。

三个适配器，按需加载：

  • 📄 文本适配器（7 问纵向追问 + 横向波及与覆盖分析） — 对每个功能点追问：正常结果、状态、触发者、输入约束、交互反馈、副作用、失败场景；全部完成后执行波及与覆盖横向扫描。受信息范围层级约束（4 级同心圆，默认层级 1）。
  • 💻 代码适配器（五镜头） — 入口与路径、数据约束、状态与转换、副作用链、错误路径。支持多代码仓库。
  • 🔬 综合层（证据聚合） — 文本和代码独立提炼后，识别"同一场景"、判断置信度、标注冲突。

7 种置信度标注： （✓） （✓,部分代码确认） （待代码确认） （代码独立） （代码独立,多源） ⚠️ ⚠️(代码) 

[assert-AUTH-001] 【权限】（✓）登录接口 · 正确用户名密码 → 返回 JWT token | upd:v1.0
[assert-AUTH-003] 【异常】（待代码确认）登录 · 账号被锁定时 → 返回具体错误码 | upd:v1.0
[assert-AUTH-004] 【上下游】（✓）登录成功 · token 写入后
  → [用户中心::会话管理] 创建活跃会话记录 | upd:v1.0
qa-functional-test 功能测试用例设计与产物输出 ✅ 可用
输入：交接块 / Story 断言 思考：用户影响优先 输出：测试用例.md + .mm 脑图
💡 如果这个模块今天在生产中出问题，用户最先感受到什么损失？

以此定义 2-3 个优先测试场景，为这些场景设计多角度用例。断言集合作为覆盖验证——⚠️ 和 （待代码确认） 的断言额外追加高优先级用例。

输入来源（按优先级）：

  1. Context 中的 【测试意图已提炼】 交接块
  2. Story 中的业务逻辑.md + 代码逻辑.md
  3. 两者均无 → 提示用户先触发 qa-understand

设计规格：

  • 📊 P0–P3 四级优先级 — 从用户影响最大到锦上添花。
  • 🧭 四维测试维度 — 正向 / 边界 / 异常 / 上下游。
  • 🔢 TC 序号自动接力 — 从文件读取最大序号，续接不跳号。
  • 🗺️ FreeMind .mm 脑图 — 可选产物，可直接导入 XMind / FreeMind / MindManager。
  • 🔗 跨模块 TC 关联 — 上下游断言自动回写关联标注。
用例 ID：TC-AUTH-001
标题：正确账号密码登录返回有效 token
功能点：用户登录 > 账号密码登录
前置条件：存在状态正常的用户账号，用户未登录
测试步骤：
  1. 输入正确凭据，点击登录
     → 预期：返回 HTTP 200，响应体含有效 JWT token
优先级：P0  |  测试维度：正向
qa-case-review 用例质量评审 ✅ 可用
输入：TC + 断言基准 思考：溯源式质量判断 输出：评审报告（阻断 / 需改进 / 建议）
🔍 这批用例拿去执行，会漏掉什么？卡在哪里？

评审的目标不是找格式问题，而是回答这个问题。从三个层次递进检查：

  • 🏗️ 结构层 — 字段是否完整？每个触发性步骤是否有对应预期？步骤是否可执行？
  • 📐 覆盖层 — 每个功能点是否覆盖了正向/边界/异常/上下游四维度？每条断言是否有对应 TC？
  • 🎯 归因层 — 发现问题时追溯根源：是 qa-functional-test 的执行层遗漏，还是 qa-understand 的理解层遗漏？
评审模式： 联动模式（同会话）— 以交接块为基准三层完整比对 · 独立模式（跨会话）— 以 Story 中的断言文件为基准 · 临时模式（粘贴内容）— 无基准，只做结构和覆盖层检查
qa-execution 测试用例执行 ✅ 可用
输入：TC 文件 + 执行环境 URL 引擎：Playwright MCP + CDP Network MCP 输出：执行报告（PASS / FAIL / SKIP）

定位：通过浏览器自动化执行测试用例，产出步骤级结构化执行报告。依赖 Playwright MCP 提供浏览器控制能力，CDP Network MCP 提供网络请求验证能力。网关执行前检查 Playwright MCP 可用性。

断言类型（10 种）：

  • 🌐 UI 断言（7 种） — 元素可见性验证、文本内容验证、元素值验证、列表项验证、截图对比、控制台消息验证、网络请求验证。
  • 🔌 API 断言（3 种） — HTTP 状态码验证、响应体 JSON 验证、响应头验证。
  • 📊 执行报告 — 逐用例记录每个步骤的执行结果，汇总 PASS / FAIL / SKIP 统计，失败用例附加截图和错误详情。
🔧 基础设施能力：qa-token-report
由 Claude Code Stop Hook 自动触发，不由网关路由。从 transcript JSONL 累加本轮 API 调用的 token 数据， 通过 8 个信号词过滤确保只统计 qalore 会话，非 qalore 会话静默退出。
🚧 Phase 2（尚未建设）
自动化测试 · 性能/压力测试 · 安全测试 · 混沌测试
请求未建设能力时，网关会明确告知不支持，并提供扩展路径指引。
Get Started

快速上手

三步完成安装配置，第四步开始使用。

1

安装 Skill 包

将 qalore/ 目录放置到 ~/.claude/skills/qalore/。Claude Code 会自动发现并注册该 skill。

2

创建配置文件

在 ~/.claude/qalore-config.json 中指定 practices 和 story 的绝对路径。首次使用时网关会自动引导完成此步骤。

{
  "practices_path": "/path/to/practices",
  "story_path": "/path/to/story"
}
3

初始化 Practices

参考 skill 包内的 practices-bootstrap.md 创建 practices 目录结构和必需文件。这是所有 capability 的规范来源。

4

开始使用

在 Claude Code 中描述测试任务，网关自动识别意图并调度对应能力：

"帮我分析登录模块的需求文档，沉淀到 story"
"读 src/auth/ 代码，更新 story"
"给登录模块出测试用例"
"评审登录模块的测试用例"
"给登录模块出全量用例并输出脑图"
Knowledge System

规范库与知识库

qalore 的持久化知识体系由两层组成：跨项目复用的 practices（规范层）和按项目维护的 story（数据层）。

📚 Practices（规范库）

一次建设，跨项目复用。capability 通过引用使用，不自行重复定义。

practices/
├── index.json ← 版本管理入口
├── common/
│   ├── handbook.md ← 写入协议、评审规范、通用性约束（每次任务必加载）
│   ├── handbook-practices-ops.md ← practices 元操作
│   └── handbook-audit.md ← 审计规范（15 步 + 19 哲学）
├── schemas/
│   └── story-index.schema.md ← Story 数据 Schema 权威定义
└── tech-stacks/functional/
    ├── assertions.md ← 断言格式规范
    ├── cases.md ← 用例格式规范
    ├── changelog.md ← 变更记录规范
    ├── output.md ← .mm 脑图规范
    ├── story-formats.md ← 文件格式规范
    └── execution.md ← 执行规范（断言 + 规则）

📖 Story（知识库）

按项目-模块两级目录组织，是测试知识的持久化存储。

story/{项目名}/
├── index.json ← 项目全局视图（每次任务必读）
└── {模块名}/
    ├── {模块}-功能-业务逻辑.md ← 需求视角断言
    ├── {模块}-功能-代码逻辑.md ← 代码视角断言
    └── {模块}-功能-测试用例.md ← 测试用例
💡 Story 设计原则：
· 三层加载：index.json 每次必读（极小）→ 内容文件按需读 → changelog 正常情况不读
· 只维护当前态：内容文件原地更新，变更原因写入 changelog
· 上下游自动聚合：depends_on 和 business_related 从断言标签自动重建

按需加载机制

practices 不进向量库——规范需要精确加载，不是语义检索。通过 context 标记实现高效复用。

Practices 文件加载时机负责加载
common/handbook.md每次任务必加载网关预加载
functional/story-formats.md读写 story 文件时各 capability 按需
functional/assertions.md理解需求或代码时qa-understand
functional/cases.md设计或评审用例时qa-functional-test / qa-case-review
functional/changelog.md写入变更记录时qa-understand / qa-functional-test
functional/output.md生成 .mm 脑图时qa-functional-test
functional/execution.md执行测试用例时qa-execution
common/handbook-practices-ops.md修改 practices 时（极低频）网关
Architecture

系统全景

Skill 包（执行引擎）+ Practices（规范层）+ Story（数据层），三层独立协作。

⚙️

Skill 包 — 执行引擎

SKILL.md ← 网关：路由、验证、注入
├── practices-bootstrap.md
├── qalore.config.template.json
└── capability/
    ├── qa-understand/ ← 理解（含 3 个适配器）
    ├── qa-functional-test/ ← 用例设计
    ├── qa-case-review/ ← 用例评审
    ├── qa-execution/ ← 用例执行（Playwright）
    └── qa-token-report/ ← Token 统计（Hook）
📚

Practices — 规范层

跨项目复用的测试规范，包含断言格式、用例格式、变更记录格式、产物输出格式、写入协议、评审规范等。通过版本号管理演进，capability 通过 practices_min_version 声明最低版本依赖。

📖

Story — 数据层

按项目维护的结构化测试知识库。每个模块独立管理业务逻辑、代码逻辑和测试用例三份文件。index.json 提供全局视图——覆盖状态、断言计数、用例数量一目了然。

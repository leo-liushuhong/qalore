面向 AI 的结构化测试知识管理系统

AI 的智力和推理能力 + qalore 的记忆和结构 = 可积累、可追溯、可跨会话延续的测试工程体系

快速开始 了解架构

为什么需要 qalore

原生 AI 在测试任务上存在结构性局限，qalore 从根本上解决这些问题。

核心思考哲学

思考框架不绑定特定工具，任何 AI 模型均可按其执行。模型越强，细节越好；框架保证大方向不走偏。

🔍

三驱动认知框架

假设优先 — 读之前先假设模块「做什么、谁触发、影响什么」，读是验证假设，不是从零积累。

模型驱动 — 每读一段，问：这证实还是推翻了假设的哪个部分？

证据驱动 — 每条论断必须有来源。任何「我认为」都是待验证假设。

边界驱动 — 每个系统交互都有两端。遇边界判断另一端是否已知，未知且关键则主动寻找。

🎯

用户影响优先

如果这个模块今天在生产中出问题，
用户最先感受到什么损失？

以此定义 2-3 个优先测试场景，为这些场景设计多角度用例。置信度为 ⚠️ 和 （待代码确认） 的断言追加高优先级覆盖——这是系统已知的不确定点，也是最容易出 bug 的地方。

📄

渐进式阅读策略

第一阶段：导航 — 用最低代价扫描信息结构，建立「这里有什么、在哪里」的地图。

第二阶段：切片精读 — 每次精读针对一个语义单元，回答假设中的一个开放问题。没有开放问题时停止，不继续读。

快速上手

①

安装

将 qalore/ 目录复制到 ~/.claude/skills/qalore/，Claude Code 自动发现。

②

配置

创建 ~/.claude/qalore-config.json，指定 practices 和 story 路径：

{
  "practices_path": "/path/to/practices",
  "story_path": "/path/to/story"
}
③

使用

在 Claude Code 中描述测试任务即可自动触发：

"帮我分析登录模块的需求文档，沉淀到 story"
"给登录模块出测试用例"
"评审登录模块的测试用例"
"给登录模块出全量用例并输出脑图"

核心能力

3 个建设完成的核心能力 + Token 统计辅助能力。

qa-understand 测试意图理解与提炼

输入：PRD / 代码 / 混合7问追问 + 五镜头 + 多源聚合输出：业务逻辑 + 代码逻辑 + 交接块

7问追问（文本源）：正常结果、状态、触发者、输入约束、交互反馈、副作用、失败场景 —— 每个功能点逐一追问。

五镜头（代码源）：入口与路径、数据约束、状态与转换、副作用链、错误路径 —— 每个代码单元独立分析。

多源证据聚合：文本和代码独立提炼后，综合层识别同一场景、判断置信度、标注矛盾。

[assert-AUTH-001] 【权限】（✓）登录接口
  · 正确凭据 → 返回 JWT token | upd:v1.0

[assert-AUTH-003] 【异常】（待代码确认）
  登录 · 账号锁定时 → 返回错误码 | upd:v1.0

[assert-AUTH-004] 【上下游】（✓）登录成功
  → [用户中心::会话管理] 创建活跃会话 | upd:v1.0

7 种置信度标注： （✓） （✓,部分代码确认） （待代码确认） （代码独立） （代码独立,多源） ⚠️ ⚠️(代码)

qa-functional-test 功能测试用例设计

用户影响优先输入：交接块 / story 断言输出：TC 文件 + .mm 脑图

先问 "用户最先感受到什么损失"，定义优先场景；再用断言集合系统性补全覆盖。支持 P0–P3 四级优先级、正向/边界/异常/上下游四维测试维度，可选输出 FreeMind .mm 脑图。

用例 ID：TC-AUTH-001
标题：正确账号密码登录返回有效 token
功能点：用户登录 > 账号密码登录
前置条件：存在状态正常的用户账号，用户未登录
测试步骤：
  1. 输入正确凭据，点击登录
     → 预期：返回 HTTP 200，响应体含有效 JWT token
优先级：P0  |  测试维度：正向

qa-case-review 用例质量评审

溯源式质量判断输入：TC + 断言基准输出：评审报告（阻断/需改进/建议）
层次检查内容示例
结构层字段完整性、步骤可执行性、预期完整性触发性步骤是否紧跟 → 预期
覆盖层与断言基准比对，每条断言是否有对应 TC边界值断言是否被覆盖
归因层发现缺口时追溯根源用例设计遗漏 vs 断言理解遗漏

qa-token-report Token 使用统计

Stop Hook 自动触发非 qalore 会话静默退出

从 transcript JSONL 累加本轮 API call 的 token 数据，以固定格式打印到对话。8 个信号词过滤确保只统计 qalore 会话。

Story 设计原则

🌐

三层加载

层内容加载策略
L1index.json每次任务必读，极小块
L2内容文件 .md按需读，只读当前模块
L3changelog正常任务绝不读
📝

只维护当前态

内容文件按功能点 / 组件分节。变更时原地更新（3.1新增 / 3.2更新 / 3.3删除），不追加历史。变更原因写入 .changelog.md，只增不删（新条目头部插入）。

🔗

跨模块关联

上下游断言是唯一真相来源。跨模块引用使用结构化标签 [模块::功能点]，index.json 的 depends_on 和 business_related 从断言自动聚合重建，不独立维护。

⚡

AI 自解释格式

[assert-IMG-003] 唯一 ID · | upd:v2.0 更新版本 · > last-upd: 节级标记 · [模块::功能点] 跨模块引用 · [来源:repo] 来源标注 · [pending/resolved/dismissed] 待确认状态

✅

人工确认两道门槛

执行前确认：展示计划，用户确认"做什么"。 写入前确认：展示待沉淀，用户确认"写什么"。两个检查点职责不同，不合并，不跳过。

🔒

Practices 不进向量库

规范需要精确加载，不是语义相似检索。context 标记 【practices:*:loaded】 实现高效按需加载。Story 模块超 50+ 时可叠加向量索引层，不替换现有结构。

实践规范库

practices 是 qalore 的"公司规范"，一次建设跨项目复用。capability 通过引用使用，不自行重复定义。

文件职责何时加载
handbook.mdstory 维护约定、增量原则、变更规则（3.1/3.2/3.3）、写入协议、评审规范每次任务必加载（网关预加载）
story-formats.md置信度标注（7 种唯一权威）、断言 ID/来源规则、业务/代码/TC/评审文件格式、交接块格式读写 story 文件时
assertions.md断言格式语法、维度标签、粒度规范、质量标准理解需求或代码时
cases.md用例格式、优先级定义（P0–P3）、测试维度、覆盖率规则、跨模块 TC 归属、命名规则设计或评审用例时
changelog.mdchangelog 内容格式、写入顺序（头部插入，倒序）写入变更记录时
output.md.mm 产物格式、脑图结构规则、输出范围策略生成脑图产物时
handbook-practices-ops.mdpractices 元操作（index.json 维护规范、practices 写入协议）修改 practices 时（极低频）

# 综合层

> 各来源的提炼已在三驱动认知框架下独立完成；综合层职责仅为聚合证据、判断置信度、产出交接块，不承担系统理解职责。

当所有来源（文本适配器、一个或多个代码适配器）均已完成时加载。

各来源地位对等：文本描述「系统应该做什么」，每个代码源描述「该仓库实际做什么」。综合层的使命不是"比较两个视角"，而是**聚合所有独立来源的证据，判断每个可测试行为的可信程度**。

---

## 核心思考哲学：多源证据聚合

### 前置：识别各断言的来源

进入聚合前，先从所有断言中提取来源标签：
- 文本适配器产出的断言：通过行末 `[来源:text]` 标签识别（多源模式下写入）
- 代码适配器产出的断言（多仓库模式）：通过行末 `[来源:{仓库名}]` 标签识别，如 `[来源:frontend]`、`[来源:backend]`
- 代码适配器产出的断言（单仓库多源模式）：通过行末 `[来源:code]` 标签识别
- 无任何 `[来源]` 标签 → 单源场景下的独立断言，直接按单源产出，无需聚合

### 第一步：识别"同一场景"

遍历所有来源的断言，将描述同一场景的断言归为一组。判断是否同一场景，满足以下任一即是：
1. 触发条件相同 + 受影响对象属同一业务实体
2. 描述同一个可观察的系统状态变化
3. 维度标签相同，操作对象在业务语义上等价（「用户」≈ `user_id`，「标题字段」≈ `title param`）**且触发条件有任何相似之处**（纯粹语义等价但触发条件无交集 → 不合并，保持独立）

无法确定是否同一场景时，保持独立，不强行合并。

### 第二步：按证据一致性分类处理

**情况一：多个独立来源均确认，结论一致**
→ 合并为一条更丰富的断言，置信度由来源数决定：
- 文本 + 所有代码源 → `（✓）`
- 文本 + 部分代码源 → `（✓,部分代码确认）`
- 仅多个代码源（无文本）→ `（代码独立,多源）`
- 合并时：用户侧可观察表述作为主断言，代码实现细节作 `↳` 补充行

**情况二：仅文本有，无任何代码源实现**
→ 独立保留，标注 `（待代码确认）`
→ 提示用户：PRD 描述了此行为，代码中未找到对应实现

**情况三：仅代码源有，文本未描述**
→ 独立保留，与文本断言同等地位
- 单一代码源 → 标注 `（代码独立）`
- 多个代码源均有且一致 → 标注 `（代码独立,多源）`
→ 追加到业务逻辑.md「代码独立断言」节（节不存在时新建）

**情况四：文本与代码源结论矛盾 `⚠️`**
→ 不合并，不产出断言
→ 记录为待确认，展示具体分歧点（见「冲突展示规范」）

**情况五：多个代码源描述同一场景但结论矛盾 `⚠️(代码)`**

先用**合约层检验四步法**判断是否真实冲突，避免把视角差异误判为冲突：

1. **是否描述同一业务动作？** 相同触发条件 + 相同业务对象 → 才可比较；否则不判冲突
2. **边界条件是否一致？** 字段名、值域、必填性 → 有差异即真实冲突
3. **结论方向是否一致？** 对同一输入，各来源都走向"允许"或都走向"拒绝" → 方向相同但表述不同（一个说"禁用按钮"，另一个说"返回400"）是**视角差异，不是冲突**；方向相反才是冲突
4. **是视角自然差异还是真实分歧？** 无法判断时 → 标记待确认

确认为真实冲突 → 不产出断言，记录为 `⚠️(代码)` 待确认，展示具体分歧点

---

## 冲突展示规范

冲突摘要在【待沉淀】上方先行输出，必须说明**具体分歧点**，不能只说"不一致"：

```
【冲突提示】
⚠️（文本-代码冲突）{对象}：
  文本：{描述A}
  代码（{仓库名}）：{描述B}
  分歧点：{具体不同之处}

⚠️(代码)（代码源冲突）{对象}：
  代码源A（{仓库名}）：{描述A}
  代码源B（{仓库名}）：{描述B}
  分歧点：{具体不同之处，如：边界值 X 的处理不同}
  建议：确认哪个仓库的逻辑为准
```

## 产出

**多源场景下的唯一【待沉淀】出口。** 所有适配器均不单独输出【待沉淀】，由本层统一汇总后输出。

**pending_count 合并规则：** 将所有适配器产生的 pending 项数量相加，写入 index.json。各来源 pending 项分别保留在对应文件，不合并文本。

输出统一交接块，格式定义见 `{practices_path}/tech-stacks/functional/story-formats.md`「统一交接块格式」章节：

```
【测试意图已提炼】
项目：{项目名}
模块：{模块名}
信息源：{文本 / 代码（{仓库名}）/ 混合（文本 + 代码源A + 代码源B）}

可测试断言：
  [assert-IMG-001] 【功能】（✓）提交按钮 · 表单完整时点击 → 订单创建成功
    ↳ 实现侧：submit() 写入 DB，发布 order.created 事件
  [assert-IMG-003] 【输入约束】（待代码确认）标题字段 · 超过 50 字符 → 显示校验提示
  [assert-IMG-005] 【边界值】（代码独立,多源）config.retry · 值为 0 → 不重试，直接返回失败
  [assert-IMG-004] 【状态转换】⚠️ 审核状态取消行为
    文本：审核中状态可被取消
    代码（仓库A）：cancel() 对 reviewing 抛出 InvalidStateError

待确认项（有则输出，无则省略）：
  【{维度}】{对象}：{具体不明确内容}

回归候选（无则省略）：
  · {模块名}（code_paths 重叠）：分析路径与该模块代码路径交叉
  · {模块名}（正向依赖）：本模块调用该模块的组件，本次修改可能影响其接口
  · {模块名}（反向依赖）：该模块的 depends_on 包含本模块，本次修改可能影响该调用方
  · {模块名}（业务关联）：本模块业务上影响该模块的功能点
```

**回归候选来源（两路合并）：**
1. **当前 context 的上下游断言**（本次会话新发现）：扫描文本适配器和代码适配器产出的所有【上下游】断言，提取 `[模块::功能点/组件]` 标签，识别本次涉及的跨模块影响
2. **index.json 历史数据**（跨会话积累）：读取 `{story_path}/{项目}/index.json`，获取本模块现有的 `depends_on` 和 `business_related` 字段（历史写入的）

两路合并去重后按上述4种类型输出。

汇总所有适配器的写入声明，输出一张合并的【待沉淀】：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-业务逻辑.md | {story_path}/{项目}/{模块}/ | {新建/3.1新增/3.2更新/3.3删除} | {功能点变更摘要} |
| {模块名}-功能-业务逻辑.changelog.md | {story_path}/{项目}/{模块}/ | 追加 | {YYYY-MM-DD}（{来源描述}）{操作摘要} |
| {模块名}-功能-代码逻辑.md | {story_path}/{项目}/{模块}/ | {新建/3.1新增/3.2更新/3.3删除} | {组件变更摘要} |
| {模块名}-功能-代码逻辑.changelog.md | {story_path}/{项目}/{模块}/ | 追加 | {YYYY-MM-DD}（{来源描述}）{操作摘要} |
| index.json | {story_path}/{项目}/ | {新建条目/patch} | assert_seq={最终值}，pending_count={合计}，depends_on={有更新时}，business_related={有更新时} |

新增/更新断言预览（按文件分组，供用户确认内容正确性）：
{逐行列出全部新增或更新的断言，格式与对应文件相同，含 | upd:{版本/日期} 标记}
```

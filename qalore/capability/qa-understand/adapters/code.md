# 代码适配器

从代码中独立建立对系统可测试行为的理解。代码是出发点，也是终点，不依赖 PRD 框架。

---

## 核心思考哲学：五镜头追问

> 五镜头是代码信息源的认知哲学，与 qa-understand SKILL.md 的三驱动元框架共同构成完整的认知体系：
> 三驱动决定「为什么读、何时停、如何识别边界」；五镜头决定「如何从代码中提取可测试行为」。

读代码的最小单位是**函数体**（完整控制流的最小自包含单元）。对于前端组件，「函数体」指核心处理函数（handleX / useXxx），不是组件渲染函数本身。阅读策略见 qa-understand SKILL.md「渐进式阅读策略」章节。

| 镜头 | 追问 | 产出断言类型 |
|------|------|------------|
| **入口与路径** | 这段代码在什么条件下被触发？主执行路径是什么？有哪些分支？ | 正向/分支断言 |
| **数据约束** | 代码对输入施加了哪些约束？权限检查、格式/范围/长度限制？ | 输入约束/权限断言 |
| **状态与转换** | 涉及哪些状态字段？允许哪些转换？哪些被明确禁止？ | 状态转换断言 |
| **副作用链** | 执行后改变了什么？持久化写入、事件发布、外部调用、缓存失效？**每个外部调用都是一个「边界」——另一端的行为是否在当前模型中？** | 上下游联动断言 |
| **错误路径** | 如何处理异常？失败时系统状态如何？用户侧可观察变化是什么？ | 异常类断言 |

**追问原则：**
- 五镜头对每个接触到的代码单元全部执行，有明确实现 → 记为断言；无明确实现 → 写入待确认项
- 外部依赖（跨服务调用、第三方库）是系统边界，判断每个边界是否闭合：
  - 另一端已知（已读过或 story 中有记录）→ 以 `[模块::组件]` 标注，记录边界关系
  - 另一端在可访问范围内但未读 → 主动读取，最多追一层（找到直接处理函数即止；若第一层仍是代理/中转看不清行为，写入待确认项「链路断点」）
  - 另一端不可访问（外部仓库 / 中间件 / 第三方服务）→ 记录「外部依赖断点」，汇入执行计划确认环节集中向用户提问，处理方式见 handbook.md「沟通规范」
  - 另一端未知且不关键 → 写入待确认项，标注「链路断点」
- 阅读深度由理解需要决定：用户有明确指定时遵从，否则以能理解核心行为为准
- 停止阅读：当前模块所有可测试开放端均有来源证据时停止；已有 code_paths 记录时确认历史路径核心行为均已覆盖；首次阅读无历史 code_paths 时，以当前任务提供的路径为基准
- **代码适配器的认知任务是「独立发现实现事实」，不是「验证 PRD 断言」。若某行为同时出现在代码和 PRD 中，由 synthesis 来识别它们是同一场景——代码适配器只负责发现，不负责对齐。**

## 回归影响思考

理解代码时，自然地思考：这段代码的变更会影响哪些其他模块？扫描三个维度：

1. **code_paths 重叠**：扫描 story 中其他模块的 `code_paths`，与当前分析路径有交集 → 标注为回归候选
2. **depends_on 正向**：当前模块的 `depends_on` 中列出的模块 → 本模块修改可能影响它们提供的接口，标注为回归候选
3. **depends_on 反向**：扫描 story 中其他模块的 `depends_on`，找出依赖当前模块的模块 → 本模块修改可能影响调用方，标注为回归候选

以上任一命中 → 纳入交接块「回归候选」节，注明原因（code_paths 重叠 / 正向依赖 / 反向依赖）

## 跨模块依赖识别

分析代码时，同步识别并写入 index.json：

**depends_on（结构性依赖）：** 本模块代码中调用了哪些其他模块的服务/接口/组件
- 判断标准：出现跨模块的方法调用、RPC 调用、Service 注入、事件消费等
- 记录格式：`{被调用模块名: [组件/服务名列表]}`
- 只记录调用方向（本模块 → 依赖模块）
- **上下游断言是唯一真相**：depends_on 从上下游断言的 `[模块::组件]` 标签聚合，不单独维护

**上下游断言跨模块格式：**
```
[assert-IMG-007] 【上下游】{代码单元} · {触发条件}
  → [目标模块::目标组件] {跨模块调用描述} | upd:{日期}
```

## 断言 ID 分配

规则遵循 story-formats.md「断言 ID 规范」（权威来源）：
- 检查 context 中 `【qa-understand-mode】` 标记（由 qa-understand 调度层在本模块开始时写入）：
  - **值为 `multi`（多源场景）**：从 `【assert_seq_runtime: N】` 读取续接起点，从 N+1 开始递增
  - **值为 `single` 或标记不存在（单源场景）**：从 index.json 的 `assert_seq` 获取当前最大序号，从 +1 开始递增
- 分配完成后更新 context 中的标记：`【assert_seq_runtime: N】`（N = 本次最大序号）
- 多源场景下【待沉淀】由 synthesis 统一输出，本节不单独输出

## 产出

**单源（仅代码适配器运行）：** 输出【待沉淀】声明，格式遵循 `{practices_path}/common/handbook.md`「story 写入协议」：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-代码逻辑.md | {story_path}/{项目}/{模块}/ | {新建/3.1新增/3.2更新/3.3删除} | {组件变更摘要，如：新增「ImageValidator」(3条)，更新「ImageStorageService」(1条)} |
| {模块名}-功能-代码逻辑.changelog.md | {story_path}/{项目}/{模块}/ | 只增不删 | {YYYY-MM-DD}（代码阅读）{操作摘要} |
| index.json | {story_path}/{项目}/ | patch | {变更字段摘要} |

新增/更新断言预览（供用户确认内容正确性）：
{逐行列出本次新增或更新的断言，格式与代码逻辑.md 相同，含 | upd:{日期} 标记}
```

**写入规程：**
- 代码逻辑.md 按「断言变更规则」操作（见 handbook.md）：找到对应组件章节原地覆盖，新组件追加章节，删除的组件整节移除
- 每条被新增或更新的断言行末附 `| upd:{日期}`（代码适配器无 PRD 版本，统一用日期）；组件节头同步刷新 `> last-upd:{日期}`
- **多仓库多源模式**（用户提供多个代码仓库时）：读取 context 中 `【code-source: {仓库名}】` 标记获取当前仓库名，在每条断言行末 `| upd:` 之后追加 `[来源:{仓库名}]`，供 synthesis 区分来源
- **单仓库多源模式**（`【qa-understand-mode: multi】` 且只有一个代码仓库）：在每条断言行末 `| upd:` 之后追加 `[来源:code]`，供 synthesis 区分与文本断言的来源
- **上下游断言跨模块时**：目标用 `[模块::组件]` 标签标注；写入后**重建** index.json 的 `depends_on`——扫描**步骤2合并后的完整内容**（即写入文件的最终版本）中所有 `[模块::组件]` 标签，提取全量跨模块引用，完整替换 index.json 中的 `depends_on` 字段（不追加，防止删除的引用残留形成幽灵数据）
- 内容文件写入后必须同步追加 changelog，格式见 `tech-stacks/functional/changelog.md`「断言 changelog 格式」章节

**index.json 声明字段（单源时）：**
`assert_seq`、`code_paths: [{ "path", "entry", "depth", "last_read" }]`、`depends_on: {"模块名": ["组件列表"]}`（有跨模块上下游断言时更新）、`status.code_logic: true`、`status.code_logic_changelog: true`、`status.pending_count`、`last_updated`

patch 操作说明：「现有 {m} 条」须来自实际 Read 文件后统计，不得从 context 记忆推断。

**多源（文本 + 代码适配器同时运行）：** 本节不输出独立的【待沉淀】。断言 ID 分配照常进行，执行结果（含断言列表和 pending 项）传入 synthesis.md，由其统一输出最终的【待沉淀】。

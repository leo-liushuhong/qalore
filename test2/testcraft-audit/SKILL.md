# testcraft-audit：系统一致性审查方法

---

## 思考核心

系统的每一条信息在流动时，接收方是否准确理解了发送方的意图？

这是所有检查的本质问题，展开为三个子问题：

1. **合约是否一致？** 文件 A 定义的格式，文件 B 消费时是否匹配
2. **新特性是否传播？** 在 A 新增的功能，所有下游消费方是否同步更新
3. **指令是否无歧义？** AI 读到这条指令，是否只有一种合理解读

---

## 思考方法：依赖图驱动，六类问题分类

**核心原则：不全量重读，沿依赖图定点检查。**

当文件 X 变更后：
1. 查 `dependency-map.md`，确定 X 的直接下游消费方
2. 对每个消费方，用 `issue-taxonomy.md` 的六类问题做定点检查
3. 修复后只重新检查该修复的影响路径，不重新审查全系统

---

## 六类问题（详见 issue-taxonomy.md）

| 类型 | 核心判断 |
|------|---------|
| 合约不一致 | producer 定义的格式 ≠ consumer 实际使用的格式 |
| 传播遗漏 | 新增特性未在所有下游消费方中体现 |
| 术语陈旧 | 旧概念名词在架构变更后未替换 |
| 机制模糊 | 同一指令有多种合理解读，AI 可能选错 |
| 结构破损 | 编辑引入重复内容、代码块未闭合、段落错误拼接 |
| 时序错误 | 指令依赖的数据在执行时尚未存在 |

---

## 执行协议（Phase 1 仅人工执行，Phase 2 将自动化）

### Step 1：变更影响分析

每次修改文件 X 后：
```
X 的输出格式变了吗？
  → 找所有消费 X 输出的文件（见 dependency-map.md），检查合约一致性

X 引入了新概念/新字段吗？
  → 找所有相关文件，检查传播覆盖

X 删除/重命名了某概念吗？
  → 在影响路径文件中搜索旧名称，找残留引用
```

### Step 2：定点验证

对影响路径上的每个文件，检查：
- 格式定义和实际使用是否一致
- 新特性是否已被处理（或已声明不处理）
- 是否有残留旧措辞

### Step 3：修复验证环

```
发现问题 → 修复 → 仅重新检查该修复的影响路径 → 直到路径上无新问题
```

**收敛标准**：当一轮检查的影响路径上没有新问题出现，停止。

---

## 严重级别

| 级别 | 定义 | 处理 |
|------|------|------|
| **严重** | 影响功能正确性（AI 会产生错误行为或幻觉）| 必须修复后再继续 |
| **显著** | 设计模糊或闭环缺口（降低任务质量）| 建议同批修复 |
| **次要** | 格式/措辞不一致（不影响行为，影响可读性）| 下次修改时顺带修复 |

## 自迭代路线图

```
Phase 1（当前）：设计沉淀
  - 依赖图显式化（dependency-map.md）
  - 问题分类体系建立（issue-taxonomy.md）
  - 人工执行审查流程

Phase 2：半自动化
  - AI 读取依赖图和问题分类，执行定点检查
  - 人工确认 issue 列表后执行修复
  - 修复后 AI 自动验证影响路径

Phase 3：全自动化
  - 每次 capability / practices 变更后自动触发
  - 产出结构化 issue 报告
  - 常见问题有标准修复模板，可选自动修复
```

---

## 自迭代的前置条件

以下条件满足时，Phase 3 才可靠：

1. **依赖图是显式的** — 写在配置文件里，不在 AI 脑子里
2. **问题类型是穷举的** — 六类是封闭集，检查时不依赖直觉
3. **收敛标准是可判断的** — 不是"感觉差不多"，是"影响路径无新 issue"
4. **修复有标准模板** — 常见问题可按模板处理，不需要从头分析

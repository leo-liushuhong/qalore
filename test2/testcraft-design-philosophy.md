# testcraft 设计哲学

> 修改 capability / practices / story 结构前必读。确认方向与设计核心一致再动手。

---

## 第一部分：设计核心

### 1. AI 是主体，框架是支撑

AI 像一名拥有专业技能、遵守公司规范、熟悉项目背景的员工，接到任务后自主判断如何工作。框架只负责给他工具、设边界、提供上下文，不规定每一个动作。

**这是所有设计决策的出发点。** 违反这条就是把员工变成流水线。

---

### 2. Capability = 思考哲学，不是执行步骤

Capability 的唯一职责是装备 AI 某个专业领域的**思考方式**：
- `qa-understand`：7问追问（文本）+ 五镜头（代码）+ 多源证据聚合（synthesis）
- `qa-functional-test`：最小充分用例集
- `qa-case-review`：溯源式质量判断

**改 capability 前问：这改变了 AI 的思考方式吗？** 加步骤 ≠ 加 capability，只有加思考维度才是。

---

### 3. Practices = 公司规范，不是操作手册

Practices 划定边界，防止失控，但不干涉边界内的智力发挥：
- 规定断言按功能点分节 → 规范，防止知识库结构失控
- 不规定断言写什么 → AI 自由判断
- 规定 changelog 必须写 → 规范，保证可溯源
- 不规定 changelog 措辞 → AI 自行表达

**改 practices 前问：这条规范在防止真实的失控风险吗？** 过度规范 = 削弱智力输出。

---

### 4. Story = 项目上下文，不是任务数据

Story 解决 AI 没有持久记忆但项目知识需要持久存在的矛盾。是项目知识在 AI 认知中的"物化"，让跨会话、跨模型、跨智能体协作成为可能。

**改 story 结构前问：AI 能从这个格式快速理解项目上下文吗？** 以 AI 快速理解为第一优先，不是数据完整性。

---

### 5. 渐进式披露：按上下文流转切割，不按角色切割

唯一判断标准：**这个切割点是否会阻断 AI 完成任务所需的上下文流转？**

按角色切割（分析 Agent / 生成 Agent）→ 错误，上下文跨 Agent 丢失。  
按上下文依赖切割（index.json → 内容文件 → changelog）→ 正确，每层按需加载，流转不中断。

**改加载时机或新增 capability 前问：会让 AI 上下文断流吗？**

---

## 第二部分：设计原则

### Story 三层架构

```
Layer 1  index.json          每次任务必读，极小，给 AI 概览全局
Layer 2  内容文件 (.md)       按需读，只读当前需要的模块
Layer 3  changelog 文件       正常任务绝不读，只在溯源时读
```

越频繁访问的层，信息密度越高、体积越小。这是节省 token 的根本机制。

---

### 内容文件只维护当前态，按功能点 / 组件分节

业务逻辑按 `## 功能点：{name}` 分节，代码逻辑按 `## 组件：{name}` 分节。

变更时按 3.1新增 / 3.2更新 / 3.3删除 原地操作，不追加历史。没有分节就没有定位锚点，只能追加，退化为版本日志。

---

### 变更标记内嵌断言行

```
[assert-IMG-003] 【输入约束】格式校验 · 大小 > 5MB → 拒绝 | upd:v2.0
```

- `> last-upd:v2.0`：节级快速扫描
- `| upd:v2.0`：断言级精确标记

不读 changelog 就能回答"哪个版本更新的"、"是否陈旧"。

---

### Changelog：有变动就写，不区分是否发版

条目头：日期 + 来源描述（有版本写版本，无版本写来源）。Changelog 记录 before state 和 why，正常任务执行不加载。

---

### 跨模块关联：上下游断言是唯一真相

跨模块引用写在上下游断言里，用结构化标签：

```
[assert-IMG-006] 【上下游】图片上传 · 上传成功
  → [内容审核::文件接收] 触发文件接收队列 | upd:v2.0
```

index.json 的 `business_related` 和 `depends_on` **从上下游断言聚合重建**，不独立维护。写入内容文件后扫描步骤2合并后的完整内容，完整替换字段（不追加，防止幽灵数据）。

| 字段 | 来源 | 回答的问题 |
|------|------|-----------|
| `code_paths` 重叠 | 代码适配器扫描 | 哪些模块共享代码路径 |
| `depends_on` | 代码上下游断言聚合 | 本模块调用了哪些模块的哪些组件 |
| `business_related` | 业务上下游断言聚合 | 本模块业务上影响了哪些模块的哪些功能点 |

---

### synthesis：多源证据聚合，不是二元比较

synthesis 的使命：聚合所有独立来源的证据，判断每个可测试行为的可信程度。

**来源识别**：文本适配器 = 文本；代码适配器 = 通过 `[来源:仓库名]` 标签区分各仓库。

**置信度由来源数和一致性决定（7种）：**

| 标注 | 触发条件 | 是否产出断言 |
|------|---------|------------|
| `（✓）` | 文本 + 所有代码源确认 | 产出 |
| `（✓,部分代码确认）` | 文本 + 部分代码源确认 | 产出 |
| `（待代码确认）` | 仅文本，无代码实现 | 产出 |
| `（代码独立）` | 仅单一代码源 | 产出 |
| `（代码独立,多源）` | 多个代码源均有，无文本 | 产出 |
| `⚠️` | 文本与代码源结论矛盾 | **不产出，待确认** |
| `⚠️(代码)` | 多个代码源描述同一场景但结论矛盾 | **不产出，待确认** |

---

### 冲突判定：合约层四步法

判断是真实冲突还是视角自然差异，防止误判：

1. **是否同一业务动作？** 不同动作 → 不判冲突
2. **边界条件是否一致？** 字段名/值域/必填性不同 → 真实冲突
3. **结论方向是否一致？** "禁用按钮" vs "返回400" 描述同一结果 → 视角差异，不冲突；同输入一个"允许"一个"拒绝" → 真实冲突
4. **无法判断？** → 标记待确认，展示具体分歧点，不猜测

---

### 回归候选：两路合并

触发回归候选的来源：
1. **当前 context 的上下游断言**（本次会话新发现的跨模块引用）
2. **index.json 历史数据**（之前会话写入的 depends_on / business_related）

两路合并去重，按4种类型输出：code_paths 重叠 / 正向依赖 / 反向依赖 / 业务关联。

---

### AI 自解释格式优先

读到这一行，不需要加载任何其他文件就能理解：
- `[assert-IMG-003]` → 唯一 ID
- `| upd:v2.0` → 更新版本
- `> last-upd:v2.0` → 节级最后更新
- `[模块::功能点]` → 跨模块引用
- `[来源:仓库名]` → 代码来源
- `[pending/resolved/dismissed]` → 待确认项状态

---

### 人工确认两道门槛

1. **执行前确认**：展示计划，用户确认"做什么"
2. **写入前确认**：展示待沉淀，用户确认"写什么"

两个检查点职责不同，不合并，不跳过。

---

### Practices 不进向量库，Story 未来可加索引层

Practices 是规则，需要精确加载，不是语义相似检索。context 标记（`【practices:handbook.md:loaded】`）已是高效按需加载，不需要向量基础设施。

Story 在模块数量超过 50+ 时，可在 git+markdown 基础上叠加向量索引层（用于跨模块语义检索），不替换现有结构。

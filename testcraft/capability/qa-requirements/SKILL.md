# qa-requirements：需求理解与结构化提炼

## 前置说明

以下变量由 testcraft 在调用前显式注入，此处直接使用：

| 变量 | 来源 | fallback |
|------|------|---------|
| `{practices_path}` | testcraft 步骤 6 注入 | 自动恢复：按 testcraft 步骤 0-4 重建 context |
| `{story_path}` | testcraft 步骤 6 注入 | 自动恢复：按 testcraft 步骤 0-4 重建 context |
| `{确认项目名}` | testcraft 步骤 3 注入 | 暂停，向用户重新确认 |

**若任一变量不可用（context 丢失）：**
- `practices_path` / `story_path` → 自动恢复：读 `~/.claude/testcraft-config.json` 重新获取路径，按 testcraft 步骤 0-4 重建 context，然后继续执行本任务
- `确认项目名` → 暂停，向用户提问：「请确认当前任务属于哪个项目？」
- **不得用猜测值继续执行**

## 需求来源

支持以下任意形式的需求输入：

```
· 用户直接粘贴的需求文档内容
· 用户口头描述的功能需求
· 本地文档文件路径
· 外部文档 URL（通过 WebFetch 读取）
```

需求内容不足时，暂停并向用户提问，不得自行假设补全。

## 前置状态检查

检查路径：`{story_path}/{确认项目名}/{模块名}/`

```
目录不存在 → 全新模块，从零提炼
目录已存在 → 读取现有文件：
  · {模块名}-功能-业务逻辑.md  ← 了解当前业务状态
  · {模块名}-功能-代码逻辑.md  ← 了解当前技术状态（若存在）
  → 与新需求对比，识别新增/变更/删除部分
  → 仅提炼变化部分，不重写未变内容
```

## 可测试性分解协议

**核心原则：需求是测试的线索，不是测试的答案。** 提炼的目标不是"复述需求写了什么"，而是"推导出什么东西可以被验证"。

### 识别需求对象

阅读输入（口述或文档）后，先识别其中包含的所有**需求对象**——每一个有独立可验证行为的功能单元，例如：一个按钮、一条规则、一个状态、一个跳转、一个字段校验、一个权限限制。

### 对每个需求对象执行7个追问（强制全部执行）

对识别到的每一个需求对象，必须逐一完成以下7个追问：

| # | 追问 | 追问目的 |
|---|------|---------|
| Q1 | **正常结果是什么？** 用户或系统能观察到什么可测量的变化？ | 功能类断言 |
| Q2 | **它有哪些状态？** 每个状态下用户能看到/感知到什么？（loading / empty / error / disabled / selected / hover…） | 状态类断言 |
| Q3 | **谁可以触发它？** 需要什么前提条件？不满足前提时会怎样？ | 权限类 / 前提类断言 |
| Q4 | **输入是什么？** 格式、范围、长度、必填规则是什么？违反时有什么反馈？ | 输入约束类断言 |
| Q5 | **用户每个动作之后，系统给什么反馈？** hover、点击、拖拽、提交…每个动作之后 UI 有何变化？ | 交互反馈类断言 |
| Q6 | **它改变了什么？** 影响哪些其他模块或数据？依赖哪些其他模块提供什么？ | 上下游联动类断言 |
| Q7 | **什么情况下它会失败？** 失败后系统如何响应？用户看到什么？数据如何处理？ | 异常类断言 |

**追问规则：**
- 7个追问对每个需求对象**全部执行**，不得跳过
- 输入中有描述 → 记录为对应维度的断言
- 输入中没有描述 → **不得跳过**，标注为「PRD未描述」写入待确认项
- 一个追问可产生多条断言，按实际发现数量记录

---

## 提炼方法

**将7问追问的结果按以下原则组织输出：**

1. **有答案的追问结果** → 写入「可测试断言」，标注对应维度标签（Q1~Q7 分别对应功能/状态/权限/输入约束/交互反馈/上下游/异常）
2. **无答案的追问维度** → 写入「待确认项」，注明是哪个追问维度、缺失什么信息
3. **多个需求对象共享的规则** → 提升为模块级断言，单独记录
4. **仅影响当前模块内部的规则** → 归入对应需求对象的断言
5. **跨模块影响** → 归入 Q6（上下游）维度

> **代码逻辑（{模块名}-功能-代码逻辑.md）** 由 Phase N 的 qa-read-code 能力通过读取业务代码生成，当前版本不写入此文件。

## 输出格式

**{模块名}-功能-业务逻辑.md：**

```markdown
# {模块名} 功能业务逻辑

## 功能概述

{一句话描述：面向谁、解决什么问题、核心职责}

## 可测试断言

> 格式：【维度】需求对象 · 条件 → 可观察预期
> 维度标签：功能 / 状态 / 权限 / 输入约束 / 交互反馈 / 上下游 / 异常

【功能】{需求对象} · {触发条件} → {可观察的预期结果}
【状态】{需求对象} · {状态条件} → {该状态下的可观察表现}
【权限】{角色} · {操作} → {允许 / 拒绝，以及拒绝时的表现}
【输入约束】{字段/输入项} · {输入内容} → {校验结果 / 反馈方式}
【交互反馈】{需求对象} · {用户动作} → {系统给用户的即时反馈}
【上下游】{本模块操作} → {对模块X的影响 / 依赖模块Y提供的{具体功能}}
【异常】{失败条件} → {处理方式 / 用户看到什么 / 数据如何处理}

## 待确认项（PRD未描述，需与 PM 确认）

- 【{维度}】{需求对象}：{追问内容}，输入未描述
```

**断言写法说明：**
- 每条断言是一个独立可验证的事实，粒度等于"一个步骤可以验证的一件事"
- 同一需求对象的多个可观察结果，写多条断言，不合并
- 维度标签驱动 qa-functional-test 生成对应类型的用例：【交互反馈】→交互类TC，【状态】→展示类TC，以此类推

## context 交接块（供 qa-functional-test 直接使用）

提炼完成后输出结构化的 context 交接块，供 qa-functional-test 设计用例时使用：

```
【业务逻辑已提炼】
（内部传递机制，人类只需关注后续 story 写入确认；此块由 qa-functional-test 消费）
项目：{项目名}
模块：{模块名}

功能概述：{一句话描述}

可测试断言：
  【功能】{需求对象} · {条件} → {预期}
  【状态】{需求对象} · {状态条件} → {可观察表现}
  【权限】{角色} · {操作} → {结果}
  【输入约束】{字段} · {输入} → {校验结果}
  【交互反馈】{需求对象} · {用户动作} → {系统反馈}
  【上下游】{操作} → {影响/依赖}
  【异常】{失败条件} → {处理方式}

待确认项：
  【{维度}】{需求对象}：{追问内容}，PRD未描述
```

**多模块任务**：先处理完**所有模块**，再统一输出全部交接块。完整视野有利于 qa-functional-test 设计跨模块用例。

**输出时机**：全部交接块输出后，立即展示 story 写入确认表格。用户确认后：qa-requirements 写入业务逻辑文件 + qa-functional-test 从交接块设计用例，两者并发执行。

## 写入 story（强制展示确认）

写入基础路径：`{story_path}/{确认项目名}/{模块名}/`

执行前展示全部待写入内容，等待人类确认后写入：

| 文件 | 完整路径 | 操作 |
|------|---------|------|
| `{模块名}-功能-业务逻辑.md` | `{story_path}/{确认项目名}/{模块名}/` | 新建 或 增量 patch |
| `index.json` | `{story_path}/{确认项目名}/` | 新模块时追加条目；已有模块时更新 `last_updated` |

**写入规则：**
- 业务逻辑.md：增量 patch，**禁止全量覆盖**，保留未变更内容
- index.json：只追加，禁止删除已有条目
- 未变更的文件：不写入，不在确认内容中展示

**新增模块时 index.json 条目格式：**

```json
"{模块名}": {
  "description": "{模块一句话描述，说明该模块是什么、面向谁、核心职责}",
  "tech_stacks": [],
  "last_updated": "{YYYY-MM-DD}"
}
```

> **注意：** 此为初始条目格式（qa-requirements 首建）。qa-functional-test 后续追加 `tc_prefix`、`mm_short_id` 字段，写入时只追加新字段，禁止覆盖已有内容。

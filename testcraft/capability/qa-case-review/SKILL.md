  输出结构化问题报告，验证用例健壮性。不产生持久化产物，报告仅在对话中呈现。

  支持两种输入模式：story 模式（读 story TC 文件，完整三层检查）；

  支持联动模式：qa-functional-test 完成后直接以 context 中刚生成的用例触发。

# qa-case-review：用例评审

## 前置说明

以下变量由 testcraft 在调用前显式注入，此处直接使用：

| 变量 | 来源 | fallback |
|------|------|---------|
| `{practices_path}` | testcraft 注入 | 自动恢复：读 `~/.claude/testcraft-config.json` 重建 context |
| `{story_path}` | testcraft 注入 | 自动恢复：读 `~/.claude/testcraft-config.json` 重建 context |
| `{确认项目名}` | testcraft 注入 | 临时模式下不强制要求；story 模式下暂停向用户确认 |

**若任一变量不可用（context 丢失）：**
- `practices_path` / `story_path` → 自动恢复：读 `~/.claude/testcraft-config.json` 重新获取路径，重建 context，然后继续执行本任务
- `确认项目名` → 临时模式下不强制要求；story 模式下暂停确认
- **不得用猜测值继续执行**

## 输入模式与评审基准（一次性判断，结果写入 context）

先判断输入模式，再确定评审基准，两步合并执行：

```
── 联动模式 ──────────────────────────────────────────────────
触发条件：用户请求「生成并评审」，qa-functional-test 已完成用例设计
输入来源：context 中刚生成的用例
评审基准：context 中的【测试意图已提炼】交接块（含置信度标注，信息最丰富）
检查范围：L1 + L2a + L2b + L3（完整三层）

── story 模式 ──────────────────────────────────────────────────
触发条件：用户指定模块名 + 未粘贴 TC 内容 + 非联动触发
输入来源：{story_path}/{确认项目名}/{模块名}/{模块名}-功能-测试用例.md
评审基准（按可用性依次）：
  · 业务逻辑.md 存在 → 意图层基准（含置信度标注，跨会话可读）
  · 代码逻辑.md 存在 → 实现层基准
  · 两者均不存在 → 仅做 L1 + L2a + L3
检查范围：L1 + L2a + L2b（有基准时）+ L3

── 临时模式 ──────────────────────────────────────────────────
触发条件：用户粘贴了 TC 内容（对话中存在用例格式文本）
输入来源：对话中用户粘贴的内容
评审基准：无
检查范围：L1 + L2a + L3（跳过 L2b）
项目/模块信息可选：有则在报告中标注，无则以「临时评审」标注
```

**story 模式下 TC 文件不存在：**
→ 停止，输出「{模块名} 暂无测试用例文件，无法执行评审」。

同时读取规范文件（优先复用 context 中已有内容）：
```
context 中存在 【practices:assertions.md:loaded】 → 直接复用，0 token
不存在 → Read({practices_path}/tech-stacks/functional/assertions.md)
         加载后写入：【practices:assertions.md:loaded】

context 中存在 【practices:cases.md:loaded】 → 直接复用，0 token
不存在 → Read({practices_path}/tech-stacks/functional/cases.md)
         加载后写入：【practices:cases.md:loaded】
```

handbook.md（评审规范）已由 testcraft 网关预加载，直接使用。

## 三层检查框架

### L1 结构合规性（机械校验）

对照 `{practices_path}/tech-stacks/functional/cases.md`「用例格式」章节逐条扫描：

| 检查项 | 判定规则 | 问题级别 |
|--------|---------|---------|
| 必填字段完整性 | 用例 ID / 标题 / 前置条件 / 测试步骤 / 优先级 / 版本，任一缺失 | 阻断 |
| 逐步预期完整性 | 触发系统可观察响应的步骤（页面跳转、UI 状态变化、系统反馈、数据变更）缺少「→ 预期：」内联预期 | 阻断 |
| TC ID 格式 | 必须符合 `TC-{PREFIX}-{3位序号}` 规则 | 阻断 |
| TC ID 重复 | 同文件内存在相同 ID | 阻断 |
| TC ID 连续性 | 序号存在跳号（非删除导致） | 需改进 |
| 优先级合法值 | 仅允许 P0 / P1 / P2 / P3 | 阻断 |

### L2 覆盖完整性

**L2a：四维度覆盖（始终执行）**

从 TC 标题和步骤中识别功能点列表，对每个功能点检查四个维度：

| 维度 | 判定逻辑 | 缺失时级别 |
|------|---------|----------|
| 正向路径 | 是否存在该功能点的成功执行用例（P0/P1） | 阻断 |
| 边界条件 | 是否存在边界值/极值/空值相关用例 | 需改进 |
| 异常场景 | 是否存在错误输入/权限不足/超时等异常用例 | 需改进 |
| 上下游影响 | 功能点涉及跨模块影响时，是否有对应联动用例 | 需改进 |

**L2b：断言库多层对照（基准可用时执行）**

以可用基准对照 TC 文件，按可用层次进行比对：

**意图层（业务逻辑.md 或交接块中的文本断言）：**
```
有对应 TC    → 意图已覆盖
无对应 TC    → 标注「意图未覆盖」，列入需改进
```

**实现层（代码逻辑.md 或交接块中的代码断言，若存在）：**
```
有对应 TC    → 实现行为已验证
无对应 TC    → 标注「实现行为未验证」，列入建议
```

**交叉比对（交接块存在时额外执行）：**
```
意图有但 TC 未覆盖    → 追溯：是大纲设计时遗漏，还是用例编写时遗漏？
实现与意图冲突的断言   → TC 是否选择了正确版本进行验证？
```

问题归因标注：
- 「意图→TC 直接缺失」→ 用例编写层遗漏（qa-functional-test 执行问题）
- 「意图存在但综合层未纳入」→ 理解层遗漏（qa-understand 综合层问题）

### L3 断言质量（逐条评判）

遵循 `{practices_path}/tech-stacks/functional/cases.md`「用例质量标准」章节，对每条用例评判步骤可执行性、逐步预期完整性、预期内容可观察性、用例独立性、前置条件充分性。
断言本身的质量标准遵循 `{practices_path}/tech-stacks/functional/assertions.md`「断言质量标准」章节。

## 输出格式

```
【用例评审报告】
项目：{确认项目名}  模块：{模块名}  评审时间：{YYYY-MM-DD}
评审用例数：{n} 条  规范版本：{practices version}

■ 阻断问题（{n} 条，必须修复）
  [{TC-ID 或 覆盖项}] {问题描述}

■ 需改进（{n} 条）
  [{TC-ID 或 覆盖项}] {问题描述}

■ 建议优化（{n} 条）
  [{TC-ID 或 覆盖项}] {建议内容}

总结：{n} 条阻断 / {m} 条需改进 / {k} 条建议
结论：{通过 / 需修复后通过 / 不通过}
```

**结论判定规则：** 遵循 `{practices_path}/common/handbook.md`「评审规范 · 结论判定规则」章节。

**报告末尾固定输出修复引导（有阻断问题时）：**

```
如需修复阻断问题：
  执行「修复 {模块名} 以下用例的阻断问题：{TC-ID-001, TC-ID-005, ...}」
  → testcraft 将触发 qa-functional-test 对上述 TC 执行精准 patch
```

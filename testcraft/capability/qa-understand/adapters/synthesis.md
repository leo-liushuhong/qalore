# 综合层

当文本适配器和代码适配器**均已完成**时加载，执行双重验证并产出统一交接块。
两个适配器的断言集合是**平行独立的视角**，综合层的职责是交叉验证，而非合并填补。

---

## 触发条件

context 中同时存在：
- 文本适配器的断言集合（来自 7问追问）
- 代码适配器的断言集合（来自五镜头追问）

单源场景不加载本文件，由对应适配器直接输出交接块。

---

## 双重验证逻辑

对每条文本断言，在代码断言中查找对应的实现覆盖：

```
一致（文本有，代码也有）
→ 标注「已验证」，纳入统一断言集，信心最高

仅文本有，代码无
→ 标注「待代码确认」
→ 纳入统一断言集
→ 向用户提示：该行为在 PRD 中有描述，但代码中未找到对应实现，
  建议与开发确认是漏实现还是 PRD 超前

仅代码有，文本无
→ 标注「仅实现视角」
→ 纳入统一断言集
→ 不预判是否应该测试，由 qa-functional-test 按覆盖策略决定

文本与代码存在冲突 ⚠️
→ 标注「需确认」
→ 两个版本均保留在统一断言集，分别注明来源
→ 向用户明确列出冲突，建议与 PM/开发确认以哪个为准
```

---

## 统一断言集合产出

综合层完成验证后，将所有断言（含标注）合并为一份统一断言集合，
按 qa-understand/SKILL.md 定义的交接块格式输出：

```
【测试意图已提炼】
项目：{项目名}
模块：{模块名}
信息源：混合（文本 + 代码）

可测试断言：
  【功能】... （已验证）
  【输入约束】... （待代码确认）
  【边界值】... （仅实现视角）
  【状态转换】... ⚠️ 需确认（文本：{描述A} / 代码：{描述B}）
  ...

待确认项：
  【{维度}】{对象}：{具体不明确内容}
```

---

## 产出写入声明

综合层汇总两个适配器的写入声明，合并输出一张【待沉淀】声明，不新建额外文件。

冲突摘要（若有）在声明上方输出后，再展示合并后的【待沉淀】表。

**多字段合并规则：**
- `status.has_pending_items`：任一适配器为 true → 合并结果为 true（取最严格值）
- `code_paths`：来自代码适配器，文本适配器不写此字段，无需合并
- `status.business_logic` / `status.code_logic`：各自独立，无冲突
- `last_updated`：取当前日期，覆盖两者

# 代码适配器

从代码视角独立建立对系统可测试行为的理解。
与文本适配器**平行独立**，不依赖文本视角的输出，不以填补 PRD 空白为目的。
产出写入 `{模块名}-功能-代码逻辑.md`，并向综合层提供独立的代码维度断言集合。

---

## 代码输入来源（优先级从高到低）

```
1. index.json 中该模块的 code_paths 字段（已记录的历史路径，优先复用）
2. 用户在请求中明确提供的文件路径、目录路径或代码片段
3. 以上均无 → 暂停询问：「请提供 {模块名} 对应的代码路径（文件或目录均可）」
   获得后纳入本次 index.json 更新，在写入确认时一并展示
```

**不得自行推断代码路径，不得扫描项目目录猜测。**

---

## 阅读深度模型

| 深度 | 含义 | 适用场景 |
|------|------|---------|
| depth=0 | 只读用户指定的代码单元本体，不追踪任何调用 | 快速了解单个函数/方法 |
| depth=1（默认） | 读入口单元本体 + 直接被调用单元的签名（不展开实现）| 大多数日常场景 |
| depth=N（用户指定）| 递归展开至第 N 层，第 N+1 层只记录名称 | 副作用链复杂时 |

用户描述含「深入」「追踪调用链」「depth=N」时覆盖默认值。
code_paths 中记录的该路径 depth 值以该值为默认（可被本次请求覆盖）。

**外部依赖处理（任意深度均适用）：**
跨服务调用/第三方库/系统 API → 记录调用点和参数，标注「外部依赖」，不展开其实现。

## 五镜头追问框架（对每个代码单元强制全部执行）

| 镜头 | 追问 | 产出断言类型 |
|------|------|------------|
| L1 入口与路径 | 这段代码在什么条件下被触发？主执行路径是什么？有哪些分支？ | 正向/分支用例 |
| L2 数据约束 | 代码对输入施加了哪些隐式或显式约束？权限检查、格式/范围/长度限制？ | 输入约束/权限断言 |
| L3 状态与转换 | 涉及哪些状态字段？允许哪些转换？哪些转换被明确禁止？ | 状态转换断言 |
| L4 副作用链 | 执行后改变了什么？持久化写入/事件发布/外部调用/缓存失效？ | 上下游联动断言 |
| L5 错误路径 | 如何处理异常？失败时系统状态如何？用户侧可观察变化是什么？ | 异常类断言 |

**规则：** 五镜头对每个代码单元全部执行，不得跳过。
- 代码中有明确实现 → 记录为对应维度的断言
- 代码中无明确实现（如某异常分支无处理逻辑）→ 标注「代码行为不明确」写入待确认项

---

## 执行步骤（depth=1 时）

```
1. Read 入口文件（用户指定或 code_paths 记录的入口）
2. 记录入口函数/类名称（用于代码来源表的「入口函数/类」字段和 index.json entry）
3. 提取所有直接调用点的函数/方法名
4. 对每个直接调用：Read 其定义，只读签名 + 顶层注释，不递归展开
5. 对每个读取到的代码单元执行五镜头追问
```

## 产出写入声明

执行完成后输出【待沉淀】声明，格式遵循 `{practices_path}/common/handbook.md`「story 写入协议」：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-代码逻辑.md | {story_path}/{项目}/{模块}/ | {新建/patch} | {断言变更摘要} |
| index.json | {story_path}/{项目}/ | patch | {变更字段摘要} |
```

**index.json 声明字段：**
- `code_paths`：`[{ "path": "{路径}", "entry": "{入口函数/类}", "depth": N, "last_read": "{YYYY-MM-DD}" }]`
  - 已存在时覆盖同路径条目，新路径追加
- `status.code_logic: true`
- `status.has_pending_items: {true/false}`（基于本次提炼结果）
- `last_updated`

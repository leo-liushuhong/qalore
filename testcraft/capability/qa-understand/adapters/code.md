# 代码适配器

从代码中独立建立对系统可测试行为的理解。代码是出发点，也是终点，不依赖 PRD 框架。

---

## 核心思考哲学：五镜头追问

拿到任何代码后，用以下五个镜头审视它。目标不是理解代码写了什么，而是推导出代码中哪些行为可以被验证。

| 镜头 | 追问 | 产出断言类型 |
|------|------|------------|
| **入口与路径** | 这段代码在什么条件下被触发？主执行路径是什么？有哪些分支？ | 正向/分支断言 |
| **数据约束** | 代码对输入施加了哪些约束？权限检查、格式/范围/长度限制？ | 输入约束/权限断言 |
| **状态与转换** | 涉及哪些状态字段？允许哪些转换？哪些被明确禁止？ | 状态转换断言 |
| **副作用链** | 执行后改变了什么？持久化写入、事件发布、外部调用、缓存失效？ | 上下游联动断言 |
| **错误路径** | 如何处理异常？失败时系统状态如何？用户侧可观察变化是什么？ | 异常类断言 |

**追问原则：**
- 五镜头对每个接触到的代码单元全部执行，有明确实现 → 记为断言；无明确实现 → 写入待确认项
- 外部依赖（跨服务调用、第三方库）记录调用点，不展开其实现
- 阅读深度由理解需要决定：用户有明确指定时遵从，否则以能理解核心行为为准

## 回归影响思考

理解代码时，自然地思考：这段代码是否也被其他模块依赖？
- 扫描 story 中其他模块的 `code_paths`，检查是否与当前分析路径有交集
- 有交集的模块 → 标注为回归候选，纳入交接块「回归候选」节

---

## 断言 ID 分配

规则遵循 story-formats.md「断言 ID 规范」（权威来源）：
- 检查 context 中 `【qa-understand-mode】` 标记（由 qa-understand 调度层在本模块开始时写入）：
  - **值为 `multi`（多源场景）**：从 `【assert_seq_runtime: N】` 读取续接起点，从 N+1 开始递增
  - **值为 `single` 或标记不存在（单源场景）**：从 index.json 的 `assert_seq` 获取当前最大序号，从 +1 开始递增
- 分配完成后更新 context 中的标记：`【assert_seq_runtime: N】`（N = 本次最大序号）
- 多源场景下【待沉淀】由 synthesis 统一输出，本节不单独输出

## 产出

**单源（仅代码适配器运行）：** 输出【待沉淀】声明，格式遵循 `{practices_path}/common/handbook.md`「story 写入协议」：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-代码逻辑.md | {story_path}/{项目}/{模块}/ | {新建/patch} | 新增 {n} 条断言，现有 {m} 条 → 变更后 {m+n} 条 |
| index.json | {story_path}/{项目}/ | patch | {变更字段摘要} |

新增断言预览（供用户确认内容正确性）：
{逐行列出本次新增的全部断言，格式与代码逻辑.md 相同}
```

**index.json 声明字段（单源时）：**
`assert_seq`、`code_paths: [{ "path", "entry", "depth", "last_read" }]`、`status.code_logic: true`、`status.pending_count`、`last_updated`

patch 操作说明：「现有 {m} 条」须来自实际 Read 文件后统计，不得从 context 记忆推断。

**多源（文本 + 代码适配器同时运行）：** 本节不输出独立的【待沉淀】。断言 ID 分配照常进行，执行结果（含断言列表和 pending 项）传入 synthesis.md，由其统一输出最终的【待沉淀】。

# 文本适配器

从需求文档、口述或任意文本输入中独立建立可测试的业务行为理解。
产出写入 `{模块名}-功能-业务逻辑.md`，并向综合层提供断言集合。

---

## 支持的输入形式

```
· 用户直接粘贴的需求文档内容
· 用户口头描述的功能需求
· 本地文档文件路径
· 外部文档 URL（通过 WebFetch 读取）
```

需求内容不足时，暂停并向用户提问，不得自行假设补全。

---

## 前置状态检查

检查：`{story_path}/{确认项目名}/{模块名}/{模块名}-功能-业务逻辑.md`

```
不存在 → 全新模块，从零提炼

已存在 → 读取现有文件：
  · 与新需求对比，识别新增/变更/删除部分
  · 仅提炼变化部分，不重写未变内容
```

---

## 7问追问框架（对每个需求对象强制全部执行）

阅读输入后，先识别所有**需求对象**——每一个有独立可验证行为的功能单元（一个按钮、一条规则、一个状态、一个字段校验、一个权限限制等）。

对每个需求对象，逐一完成以下7个追问：

| # | 追问 | 追问目的 |
|---|------|---------|
| Q1 | **正常结果是什么？** 用户或系统能观察到什么可测量的变化？ | 功能类断言 |
| Q2 | **它有哪些状态？** 每个状态下用户能看到/感知到什么？ | 状态类断言 |
| Q3 | **谁可以触发它？** 需要什么前提条件？不满足时会怎样？ | 权限/前提类断言 |
| Q4 | **输入是什么？** 格式/范围/长度/必填规则？违反时有什么反馈？ | 输入约束类断言 |
| Q5 | **用户每个动作之后，系统给什么反馈？** | 交互反馈类断言 |
| Q6 | **它改变了什么？** 影响哪些其他模块或数据？依赖哪些？ | 上下游联动类断言 |
| Q7 | **什么情况下它会失败？** 失败后系统如何响应？ | 异常类断言 |

**规则：** 7问对每个需求对象全部执行，不得跳过。
- 输入中有描述 → 记录为对应维度的断言
- 输入中无描述 → 标注「PRD未描述」写入待确认项
- 一个追问可产生多条断言，按实际数量记录

遵循 `{practices_path}/tech-stacks/functional/story-formats.md`「业务逻辑文件」格式规范。

## 产出写入声明

执行完成后输出【待沉淀】声明，格式遵循 `{practices_path}/common/handbook.md`「story 写入协议」：

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-业务逻辑.md | {story_path}/{项目}/{模块}/ | {新建/patch} | {断言变更摘要，如：新增5条断言，修改2条} |
| index.json | {story_path}/{项目}/ | {新建条目/patch} | {变更字段摘要} |
```

**index.json 声明字段（按场景）：**

新模块首建：
```json
"{模块名}": {
  "description": "{模块一句话描述}",
  "prd_version": "{版本号（无版本时省略此字段）}",
  "status": {
    "business_logic": true,
    "code_logic": false,
    "tc_count": 0,
    "has_pending_items": {true/false}
  },
  "last_updated": "{YYYY-MM-DD}"
}
```

已有模块更新：仅声明变更的字段：
- `prd_version`（若本次有版本信号）
- `status.business_logic: true`
- `status.has_pending_items: {true/false}`（基于本次提炼结果中是否有待确认项）
- `last_updated`

> 若该模块 `status.tc_count = 0`，提醒用户：「{模块名} 尚无测试用例，如需生成执行『生成 {模块名} 用例』」

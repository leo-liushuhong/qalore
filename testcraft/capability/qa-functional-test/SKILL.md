# qa-functional-test：功能测试用例设计与产物输出

## 前置说明

以下变量由 testcraft 在调用前显式注入，此处直接使用：

| 变量 | 来源 | fallback |
|------|------|---------|
| `{practices_path}` | testcraft 注入 | 自动恢复：读 `~/.claude/testcraft-config.json` 重建 context |
| `{story_path}` | testcraft 注入 | 自动恢复：读 `~/.claude/testcraft-config.json` 重建 context |
| `{确认项目名}` | testcraft 注入 | 立即终止，输出「项目名 context 已丢失，请重新执行任务」 |

**不得用猜测值继续执行。**

## 执行模式判断（一次性完成，结果写入 context）

检查 `{story_path}/{确认项目名}/{模块名}/{模块名}-功能-测试用例.md` 是否存在，结合用户意图，确定执行模式和用例操作类型：

```
TC 文件不存在
→ 完整流程模式，用例操作：全新设计（TC 序号从 001 开始）

TC 文件已存在，且满足任一增量信号：
  · 用户提供了需求文档内容或文件路径
  · 对话中出现明确版本号（如 v.demo / v1.2）
  · 用户描述含「更新」「沉淀」「同时」「变更」等增量操作词
  · 用户明确要求「重新设计」或「补充用例」
→ 完整流程模式，用例操作：读取现有用例，逐条比对需求变化
    · 需求有新增        → 新增对应功能点的用例（3.1）
    · 需求有变更        → 覆盖更新受影响用例，追加 changelog（3.2）
    · 需求导致用例失效  → 删除失效用例，追加 changelog（3.3）
    · 仅补充覆盖        → 在现有基础上增加缺失维度

TC 文件已存在，无增量信号，用户描述仅含「输出/生成」
→ 直接输出模式：跳过设计流程和 story 写入，直接读取 story 用例生成产物

⚠️ 不确定时默认选完整流程模式。
```

将模式和用例操作类型写入 context，后续章节直接引用，不重复判断。

## 产物输出判断（直接输出模式立即执行；完整流程模式在写入声明确认后执行）

### 输出路径解析

检测用户消息中是否存在显式的文件系统路径（绝对路径或相对路径）：

```
存在外部路径（如 D:\test\functional\case）：
  · 该路径 = deliverable_path，供人阅读的独立交付文档输出位置
  · story_path 仍是唯一 story 写入目标，两者互不替代，不得混淆

  ⚠️ 强制规则：
  1. story 写入不因存在外部路径而跳过——story 写入是必须执行的步骤，
     与是否有外部路径无关
  2. 外部路径只接收最终 .mm 产物文件，不接收 story 文件
  3. 禁止将 story 文件写入 deliverable_path
  4. 禁止将测试用例输出为 .md 格式至 deliverable_path——deliverable 产物固定为 .mm 格式

  执行完成后：
    a. 【待沉淀】声明正常指向 story_path，等待用户确认后写入
    b. 同时将 .mm 产物文件输出至 {deliverable_path}/{文件名}.mm（文件名遵循输出规范）

不存在外部路径：
  · 仅写入 story_path，输出【待沉淀】声明
```

将解析结果写入 context：`deliverable_path = {值 / 无}`

### 是否输出 .mm

**直接输出模式**：始终输出 .mm，跳过以下判断。

**完整流程模式**：

```
明确要输出 .mm（任一信号词）：
  输出用例 / 生成用例 / 出用例 / 需要测试用例 / 测试文件 / 输出.mm / 全量
  → 输出 .mm

明确不要 .mm（任一信号词）：
  沉淀 / 存档 / 记录 / 只分析 / 只提炼 / 只更新 / 了解需求
  → 不输出 .mm（仍更新 TC 文件和 changelog）

无法判断（有 PRD 但无上述信号词）：
  → 询问用户：「本次是否需要输出 .mm 脑图产物？
     （TC 文件和 changelog 有需求变更时始终更新）」
  → 等待用户选择后继续
```

### .mm 输出范围（需输出 .mm 时执行）

```
┌──────────────────────────────────────────────────────────┐
│ 信号                          │ 默认输出范围              │
├──────────────────────────────────────────────────────────┤
│ 用户描述含"全量"               │ 全量（覆盖其他所有默认）  │
│ 完整版本 PRD + "输出用例"       │ 版本级                   │
│ 单 feature 文档/口述单个功能    │ Feature 级               │
│ 口述多个 feature（跨模块）      │ 版本级                   │
│ 无需求文档 + 指定模块           │ 模块级                   │
└──────────────────────────────────────────────────────────┘

以上均无法匹配 → 询问用户：
  「请确认输出范围：全量 / 版本级（仅本次变更）/ 模块级（某模块累积）/ Feature 级（仅该功能）」
```

| 确认输出范围 | .mm 文件包含内容 |
|------------|----------------|
| 全量 | 全项目所有模块、所有版本的累积用例 |
| 版本级 | 当前版本新增/更新的所有用例，跨模块 |
| 模块级 | 指定模块的全部累积用例，跨版本 |
| Feature 级 | 仅该 feature 的新增/更新用例 |

用户在功能层级上额外限定时，取更窄范围（正交叠加）。

格式、结构与命名遵循 `{practices_path}/tech-stacks/functional/output.md`「脑图结构规则」章节：
```
context 中存在 【practices:output.md:loaded】 → 直接复用，0 token
不存在 → Read({practices_path}/tech-stacks/functional/output.md)
         加载后写入：【practices:output.md:loaded】
```

**产出几个 .mm 文件，由确认的输出范围决定，与处理了几个模块无关。**

```
用户指定一个范围（如「demo 版本的测试用例」）→ 产出一个 .mm 文件
用户明确要求多个分开的产物（如「模块1和模块2分开出两个脑图」）→ 产出对应数量的 .mm 文件

⚠️ 禁止把逐模块处理过程中的中间结果当成独立产物输出。
  处理多模块 PRD 时：先逐模块完成设计和 story 写入，全部完成后再统一执行一次产物输出。
  中间不产出任何 .mm 文件。
```

## 读取上下文（完整流程模式）

**前置检查**：context 中是否存在 qa-understand 输出的交接块（含「【测试意图已提炼】」标记）：

```
存在交接块 → 直接从 context 读取统一断言集合，跳过磁盘读取
  · 仍需读取：{模块名}-功能-测试用例.changelog.md（了解历史变更，若存在）

不存在交接块 → 读取磁盘文件：
```

基础路径：`{story_path}/{确认项目名}/{模块名}/`

```
1. {模块名}-功能-业务逻辑.md（若存在则读）
2. {模块名}-功能-代码逻辑.md（若存在则读）
3. {模块名}-功能-测试用例.changelog.md（若存在则读，了解历史变更）

1 和 2 均不存在 → 无法继续：
  「{模块名} 无断言基准，请先提供需求文档或代码（触发 qa-understand）后重新执行」，终止
```

【上下游依赖提取（读取断言集合后立即执行）】
从断言集合中提取「上下游」维度的全部依赖方向，写入 context：
- 依赖方向为空：跳过
- 依赖方向非空：将每个方向列为上下游维度的强制设计输入，对照 practices 覆盖率规则逐一派生上下游用例

同时读取规范文件（按需加载，优先复用 context 中已有内容）：

```
设计用例时：
  context 中存在 【practices:cases.md:loaded】 → 直接复用，0 token
  不存在 → Read({practices_path}/tech-stacks/functional/cases.md)
           加载后写入：【practices:cases.md:loaded】

写入 changelog 时：
  context 中存在 【practices:changelog.md:loaded】 → 直接复用，0 token
  不存在 → Read({practices_path}/tech-stacks/functional/changelog.md)
           加载后写入：【practices:changelog.md:loaded】
```

output.md 在「产物输出判断」章节按需加载，此处不重复。

## TC ID 规划（完整流程模式，有新增用例时执行）

```
1. 从 index.json 获取当前模块的 tc_prefix
   不存在（新模块首次建立）→ 自动推断：
     · 有需求文档 → 取模块名称转为英文意译首字母缩写（2-5个大写字母）
     · 无需求文档 → 根据新增内容总结模块名转缩写
     · 检查与同项目已有前缀不冲突；冲突时尾部加数字区分（如 PIPE2）
     · 推断结果纳入本次 index.json 写入内容，写入确认时一并展示

2. 扫描测试用例.md，提取已有用例最大序号 N（文件不存在时 N=0）
3. 新用例 ID 从 N+1 开始连续递增，不得跳号、不得复用已有序号
4. 将完整 ID 列表纳入执行计划
```

## 执行前确认（完整流程模式，强制）

歧义检查（先于计划展示）：多个项目存在同名模块或功能点时，必须暂停确认，禁止自行假设。

展示执行计划：

```
【功能测试执行计划】
项目：{确认项目名}
模块：{模块名}
功能点范围：{列出将要覆盖的功能点}
测试维度：正向 / 边界 / 异常 / 上下游
预计用例数：P0 {n} 条 / P1 {n} 条 / P2 {n} 条 / P3 {n} 条
新增 TC ID 范围：TC-{PREFIX}-{起始序号} ～ TC-{PREFIX}-{结束序号}
产物输出：{.mm（{范围}）/ 仅更新 TC 文件}
规范版本：{practices version from memory}
```

等待人类响应：同意 → 执行；修正 → 调整后重新展示；拒绝 → 终止，不生成任何用例。

## 用例设计方法（完整流程模式）

**断言集合输入优先级：**

| 优先级 | 条件 | 设计输入 |
|--------|------|---------|
| 0（最高） | context 中存在 qa-understand 交接块 | 以统一断言集合为主输入（已整合所有信息源，冲突项已标注）|
| 1 | 无交接块，有磁盘文件 | 读取业务逻辑.md 和/或代码逻辑.md，合并使用 |

**覆盖策略**：遵循 `{practices_path}/tech-stacks/functional/cases.md`「覆盖率规则」章节。

**测试维度定义**：遵循 `{practices_path}/tech-stacks/functional/cases.md`「测试维度」章节。

**优先级判断**：遵循 `{practices_path}/tech-stacks/functional/cases.md`「优先级定义」章节。

**步骤与预期格式**：遵循 `{practices_path}/tech-stacks/functional/cases.md`「用例格式」章节。触发系统可观察响应的步骤（页面跳转、UI 状态变化、系统反馈、数据变更）必须内联「→ 预期：{内容}」；纯输入步骤可省略。

**断言置信度处理策略：**

| 断言状态 | 处理方式 |
|---------|---------|
| `（✓）` | 正常设计，优先级按业务重要程度判断，前置条件可精确到实现细节 |
| `（待代码确认）` | 正常设计，优先级不变，在用例备注中标注「实现待确认，执行前需与开发核实」 |
| `（仅实现视角）` | 必须设计用例，不得跳过，标注「PRD 未描述，基于代码实现设计」 |
| `⚠️` | 暂停，在执行计划中列出冲突，要求用户确认以哪个版本为准后再继续 |

## 产出写入声明

执行完成后输出【待沉淀】声明，格式遵循 `{practices_path}/common/handbook.md`「story 写入协议」。操作列取值遵循 handbook.md「用例变更规则」：`3.1新增 / 3.2更新 / 3.3删除`，changelog 文件操作固定为 `追加`。

```
【待沉淀】
| 文件 | 路径 | 操作 | 变更摘要 |
|------|------|------|---------|
| {模块名}-功能-测试用例.md | {路径}/ | {3.1新增/3.2更新/3.3删除} | TC-{PREFIX}-{起}-{止}（{n}条，{功能点摘要}）|
| {模块名}-功能-测试用例.changelog.md | {路径}/ | 追加 | {版本} {操作摘要} |
| index.json | {路径}/ | {新建条目/patch} | {变更字段摘要} |
```

**测试用例文件头部**（新建时写入，更新时同步刷新）：
```markdown
# {模块名} 功能测试用例
> TC 前缀：{PREFIX} | 用例总数：{n} 条 | 最新版本：{v1.2 / 未关联版本} | 最后更新：{YYYY-MM-DD}
```

**index.json 声明字段（按场景）：**
- 新模块首建：`tc_prefix`、`mm_short_id`、`status.tc_count: {n}`、`last_updated`
- 已有模块：更新 `status.tc_count: {新总数}`、`last_updated`

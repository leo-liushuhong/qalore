# qa-functional-test：功能测试用例设计与产物输出

## 前置说明

以下变量由 testcraft 在调用前显式注入，此处直接使用：

| 变量 | 来源 | fallback |
|------|------|---------|
| `{practices_path}` | testcraft 步骤 6 注入 | 自动恢复：按 testcraft 步骤 0-4 重建 context |
| `{story_path}` | testcraft 步骤 6 注入 | 自动恢复：按 testcraft 步骤 0-4 重建 context |
| `{确认项目名}` | testcraft 步骤 3 注入 | 暂停，向用户重新确认 |

**若任一变量不可用（context 丢失）：**
- `practices_path` / `story_path` → 自动恢复：读 `~/.claude/testcraft-config.json` 重新获取路径，按 testcraft 步骤 0-4 重建 context，然后继续执行本任务
- `确认项目名` → 立即终止，输出「项目名 context 已丢失，请重新执行任务」
- **不得用猜测值继续执行**

## 执行模式判断

每次执行前，先检查 story 文件是否已存在：

```
检查：{story_path}/{确认项目名}/{模块名}/{模块名}-功能-测试用例.md

不存在 → 完整流程模式
  · 走完整的设计 → 写 story → 产物输出 流程

已存在 → 检查当前上下文：
  · 用户提供了需求文档 / 对话中出现明确版本号 → 完整流程模式
  · 用户明确要求「重新设计」或「补充用例」     → 完整流程模式
  · 以上均无，仅要求「输出/生成用例」          → 直接输出模式
      跳过设计流程和 story 写入，直接读取 story 用例生成产物
```

## 读取上下文（完整流程模式）

**前置检查（优先于磁盘读取）**：检查 context 中是否存在 qa-requirements 输出的交接块（含「【业务逻辑已提炼】」标记）：

```
存在交接块（情况 0）→ 直接从 context 读取业务逻辑，跳过磁盘读取业务逻辑文件
  · 业务逻辑主输入：context 交接块中的功能点/业务规则/上下游依赖
  · 仍需读取：{模块名}-功能-测试用例.changelog.md（了解历史变更，若存在）
  · story 写入中的业务逻辑文件由 qa-requirements 并发写入，无需等待

不存在交接块 → 按以下顺序读取磁盘文件（情况 1.1 / 1.2 / 1.3）
```

执行前，按顺序读取以下 story 文件（情况 0 时仅读 changelog）：

```
基础路径：{story_path}/{确认项目名}/{模块名}/

1. {模块名}-功能-业务逻辑.md
   情况 1.1 — 不存在（新模块）：从当前需求输入内联提炼业务逻辑，
              提炼格式遵循 qa-requirements/SKILL.md「输出格式」章节（功能概述/功能点列表/业务规则/上下游依赖），
              提炼结果与测试用例一并纳入「写入 story」展示确认，确认后并行写入
   情况 1.2 — 已存在，无新需求输入：读取已有内容，直接进入用例设计
   情况 1.3 — 已存在，且有新需求输入：读取已有内容（了解存量状态，用于对比）；
              以需求文档为主输入识别新增/变更/删除功能点；
              提炼增量业务逻辑，与受影响的用例变更一并纳入「写入 story」展示确认

   【上下游依赖提取（强制子步骤，读取业务逻辑文件后立即执行）】
   读取业务逻辑文件后，提取「上下游依赖」节的全部依赖方向，写入 context：
   → 依赖方向为空：跳过
   → 依赖方向非空：将每个依赖方向列为上下游维度的强制设计输入，
                   对照 practices 覆盖率规则（第三层级）逐一派生上下游用例，
                   不得因测试方法衍生场景的「每类最多 1 条」限制而减少上下游用例数量

2. {模块名}-功能-代码逻辑.md           ← 理解技术实现细节（若存在则读）
3. {模块名}-功能-测试用例.changelog.md ← 了解历史变更，避免重复设计（若存在则读）
```

同时读取规范文件（优先复用 context 中已有内容）：

```
判断条件：context 中是否存在「# 功能测试规范」（standards.md 的一级标题）
  存在 → 直接复用，0 token
  不存在 → Read({practices_path}/tech-stacks/functional/standards.md)
           获取：用例字段定义、TC 命名规则、P0/P1/P2 判断标准
           直接用 context 中的 practices_path，不重走 config 读取链路
```

## 前置状态检查（完整流程模式）

检查路径：`{story_path}/{确认项目名}/{模块名}/{模块名}-功能-测试用例.md`

```
不存在 → 全新设计，覆盖全部功能点
        TC 序号从 001 开始

已存在 → 读取现有用例，逐条比对需求变化：
  · 需求有新增           → 3.1 新增对应功能点的用例
  · 需求有变更，用例仍有效 → 3.2 覆盖更新受影响用例，追加 changelog
  · 需求导致用例失效      → 3.3 删除失效用例，追加 changelog
  · 仅补充覆盖            → 在现有基础上增加缺失维度
```

## TC ID 规划（完整流程模式，有新增用例时执行）

```
1. 从 {story_path}/{项目名}/index.json 获取当前模块的 tc_prefix
   tc_prefix 不存在（新模块首次建立）→ 自动推断：
     · 有需求文档 → 严格取需求文档中该模块名称，转为英文意译首字母缩写（2-5个大写字母）
     · 无需求文档 → 根据新增内容总结模块名，转为英文意译首字母缩写（2-5个大写字母）
     · 检查与同项目已有前缀不冲突（读 index.json 的 modules 字段）
     · 冲突时尾部加数字区分（如 PIPE2）
     · 推断结果纳入本次 index.json 写入内容，在写入确认时一并展示
2. 扫描 {模块名}-功能-测试用例.md，提取已有用例中的最大序号 N
   （文件不存在时 N = 0）
3. 新用例 ID 从 N+1 开始连续递增，不得跳号、不得复用已有序号
4. 将规划好的完整 ID 列表纳入下一步的执行计划
```

## 执行前确认（完整流程模式，强制）

**歧义检查（先于计划展示执行）：**

**适用场景：testcraft 已注入项目名，但该项目内多个模块或功能点存在同名，需进一步定位**

触发条件：
- 多个项目存在同名模块或同名功能点

处理方式：
```
→ 必须暂停，向用户提问：
  「请确认：{模块/功能点名} 属于哪个项目？
   当前 story 中已有项目：{列出 story_path/ 下的目录名}」
→ 禁止自行假设并执行
→ 等用户确认后继续
```

歧义消除后，再展示执行计划：

```
【功能测试执行计划】
项目：{确认项目名}
模块：{模块名}
功能点范围：{列出将要覆盖的功能点}
测试维度：正向 / 边界 / 异常 / 上下游
预计用例数：P0 {n} 条 / P1 {n} 条 / P2 {n} 条 / P3 {n} 条
新增 TC ID 范围：TC-{PREFIX}-{起始序号} ～ TC-{PREFIX}-{结束序号}
规范版本：{practices version from memory}
```

等待人类响应：
- **同意** → 按计划执行
- **修正** → 调整计划后重新展示
- **拒绝** → 终止，不生成任何用例

## 用例设计方法（完整流程模式）

**用例设计输入优先级（按顺序取最高可用项）：**

| 优先级 | 条件 | 设计输入 |
|--------|------|---------|
| 0（最高） | context 中存在 qa-requirements 交接块 | 以交接块中的业务逻辑为第一输入（已由 qa-requirements 提炼，是业务理解的结构化产出）；PRD 仅作细节补充 |
| 1 | 有需求文档，无交接块 | 以需求文档为第一输入，从需求出发设计；业务逻辑.md 用于对比存量状态 |
| 2 | 无需求文档，有业务逻辑.md | 以业务逻辑.md + 新增业务逻辑为输入，从业务逻辑出发设计 |
| 3（最低） | 均无 | 以用户口述场景为输入 |

**覆盖优先级：PRD 业务场景优先，测试方法衍生场景次之。**

| 来源 | 覆盖策略 | 优先级由来 |
|------|---------|----------|
| PRD 明确描述的业务场景 | 全量覆盖，不得遗漏 | 强制对齐 PRD 标注的优先级 |
| 测试方法衍生场景（边界/异常/上下游） | 每类默认最多 1 条最典型；用户要求全覆盖时扩展 | PRD 未标注时按 practices 优先级定义推断 |

**四个测试维度说明：**

| 维度 | 设计思路 |
|------|---------|
| 正向路径 | 标准流程，输入合法，预期成功 |
| 边界条件 | 输入边界值、极值、空值、最大长度 |
| 异常场景 | 错误输入、权限不足、网络超时、并发冲突 |
| 上下游影响 | 此功能变化对依赖它的其他模块的影响 |

**优先级判断原则：详见 `{practices_path}/tech-stacks/functional/standards.md`「优先级定义」章节。**

## 写入 story（完整流程模式，强制展示确认）

写入基础路径：`{story_path}/{确认项目名}/{模块名}/`

执行前展示全部待写入内容，等待人类确认后写入：

| 文件 | 完整路径 | 操作 |
|------|---------|------|
| `{模块名}-功能-业务逻辑.md` | `{story_path}/{确认项目名}/{模块名}/` | 情况 1.1 新建；情况 1.3 增量 patch；情况 1.2 无变更时跳过 |
| `{模块名}-功能-测试用例.md` | `{story_path}/{确认项目名}/{模块名}/` | 新增条目 / 覆盖更新已有条目 / 删除失效条目 |
| `{模块名}-功能-测试用例.changelog.md` | 同上 | 追加变更记录（版本号+日期+条目+原因） |
| `index.json` | `{story_path}/{确认项目名}/` | 更新该模块的 tech_stacks 字段，追加 "functional" |

**并行写入：** 所有待写入文件互相无依赖，确认后并行发起 Write 调用，不得串行等待。

**变更类型与 story 操作对应：**

| 变更类型 | 用例文件操作 | changelog 记录格式 |
|---------|------------|-------------------|
| 3.1 新增 | 追加新用例条目 | `- 新增：{用例ID} {标题}（原因：{需求描述}）` |
| 3.2 更新 | 原地覆盖整条用例内容 | `- 更新：{用例ID} {变更字段}（原因：{变更原因}）` |
| 3.3 删除 | 从文件中完整删除该用例条目 | `- 删除：{用例ID} {标题}（原因：{失效原因}）` |

**index.json 更新规则：**
- 模块 entry 不存在 → 新建 entry：`{"description": "{模块一句话描述}", "tc_prefix": "{已推断值}", "mm_short_id": "{首字母小写}{序号}", "tech_stacks": ["functional"], "last_updated": "{YYYY-MM-DD}"}`
- 模块 entry 存在，不含 `"functional"` → 追加 `"functional"`，更新 `last_updated`
- 模块 entry 存在，已含 `"functional"` → 只更新 `last_updated`
- 禁止删除或覆盖已有其他字段

### 格式、结构与命名

- 默认格式、脑图层级结构、叶节点挂载原则、文件命名规则均以 practices 为准

### 输出范围规则

输出范围由 testcraft 步骤 5 注入 context，此处直接引用，不重复推断。

**功能层级 × 版本维度对照：**

| 确认输出范围 | .mm 文件包含内容 |
|------------|----------------|
| 全量 | 全项目所有模块、所有版本的累积用例 |
| 版本级 | 当前版本（来自 PRD 版本号）新增/更新的所有用例，跨模块 |
| 模块级 | 指定模块的全部累积用例，跨版本 |
| Feature 级 | 仅该 feature 的新增/更新用例 |

**用户在功能层级上额外限定时，取更窄范围（正交叠加）：**

| 用户描述示例 | 输出范围 |
|------------|---------|
| "v.0.2 的全量用例" | 全项目 × 版本级 |
| "图生图模块全量用例" | 图生图模块 × 模块级（含所有版本） |
| "图生图 v.0.2 的用例" | 图生图模块 × 版本级 |
| "全量用例" | 全项目 × 全量（所有版本） |

**匹配粒度：用户说哪个层级就输出哪个层级，不上扩也不下缩。**
